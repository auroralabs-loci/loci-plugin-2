**Callee: GAPBondMgr_FindAddr**
- Purpose: walk the bond table looking for a matching peer address.
- Control-flow shape: linear prologue â†’ loop over bonded entries â†’ return pointer
  or NULL. Number of loop iterations is bounded by the bond table size
  (NV_MAX_BONDS, typically 10 on CC23xx devices), so no unbounded recursion.
- Missing declaration risk: `GAPBondMgr_FindAddr` is declared in
  `gap_bond_mgr.h`. If `hci.c` does not `#include` that header, the compiler
  will silently treat the call as returning `int` in C89 mode, or error in C99+.
  The TI BLE5 stack compiles under C99, so a missing include will be a hard
  compile error â€” safe, but must be added as a plan step.
- Indirect calls: none expected in the lookup hot path.
- Call-order assumption: `GAPBondMgr_FindAddr` reads from NV-backed bond records.
  The GAP bond manager must already be initialized (`GAPBondMgr_Register` called
  at stack init time). `HCI_SendDataPkt` is only reached after the stack is up,
  so this ordering is satisfied.

**Callee: HCI_SendDataPkt (the function being modified)**
- The new guard runs before any existing data-path logic, so there is no

### Adjusted Plan (before any edits)

1. **Add `#include "gap_bond_mgr.h"`** (or the appropriate TI header that
   declares `GAPBondMgr_FindAddr`) at the top of `hci.c` if not already present.
2. **Audit callers of `HCI_SendDataPkt`**: confirm they inspect the return value
   and free the HCI data buffer when a non-success code is returned. If any
   caller ignores the return value, add a `hciPktFree(pkt)` call inside the
   rejection branch before returning, to avoid the leak.
3. **Measure latency**: run the MCP latency tool (once accessible) or profile
   on hardware to confirm the bond-table walk fits within the remaining headroom
   of the 480-cycle budget. If it does not, replace the inline walk with a
   cached bloom-filter or flat set checked in O(1).
4. **Write the guard** at the top of `HCI_SendDataPkt`, after the existing
   parameter validation (if any), once steps 1â€“3 are resolved:
   ```c

   uint8_t bondIdx;
   if (GAPBondMgr_FindAddr(pMsg->connHandle /* resolve to addr */, addrType, &bondIdx) == NULL) {
   ```
