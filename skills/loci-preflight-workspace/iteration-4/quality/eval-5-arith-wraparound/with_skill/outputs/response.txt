## Plan: Fix unsigned wraparound in `osal_mem_free`

### Context

The bug is in `osal_mem_free` in `osal_memory.c` (simplelink-lowpower-f3-sdk BLE5 stack). When
computing the usable size of a block being freed, the code subtracts the fixed header size from
the block's reported size field. Both operands are unsigned. If the stored size field in the
block header is smaller than `OSAL_MEM_HDRSZ` ‚Äî which can happen when memory is corrupted or
a bad pointer is passed ‚Äî the subtraction wraps to a very large unsigned value. That value is
then spliced back into the free list, corrupting the allocator's internal state and likely the
heap itself.

---

## Preflight: `osal_mem_free`

### 1. Call graph

(no binary ‚Äî static analysis only; `$LOCI_TEST_BLE_ROOT` is not present in this environment)

`osal_mem_free` is a self-contained OSAL heap-management routine. Its typical callees are:

- `HAL_ENTER_CRITICAL_SECTION` / `HAL_EXIT_CRITICAL_SECTION` ‚Äî macro wrappers around the
  architecture's interrupt-disable/restore primitives. Not recursive; no indirect dispatch.
- Internal free-list pointer manipulation ‚Äî direct struct-field writes, no function calls.

No indirect calls (`bl`-to-register), no recursion, and no call-order guards protecting the
free-list pointers beyond the critical section itself. The fix introduces no new callees.

Call graph: OK (no new callees; critical-section pairing is the main concern ‚Äî see Resources)

---

### 2. Arithmetic ranges

The defective expression has this shape:

```c
/* hdr->size and OSAL_MEM_HDRSZ are both unsigned (osalMemHdr_t / uint16_t) */
uint16_t freeSize = hdr->size - OSAL_MEM_HDRSZ;
```

Value-space analysis:

- `OSAL_MEM_HDRSZ` is a compile-time constant representing the byte size of the block header
  (typically 4 bytes for a 32-bit-aligned struct with a size field and a next pointer).
- `hdr->size` is a runtime value stored in memory. Any write that corrupts the heap (buffer
  overrun, wild pointer store, stack smash) can make `hdr->size` smaller than `OSAL_MEM_HDRSZ`.
- When `hdr->size < OSAL_MEM_HDRSZ`: e.g., `hdr->size == 2`, `OSAL_MEM_HDRSZ == 4`:
  `2 - 4` on a `uint16_t` wraps to `0xFFFE` (65534). The allocator then treats a 65534-byte
  region as a free block. On the next allocation or coalescing pass it will read and write
  memory far outside any valid heap boundary.
- There is no existing guard: the subtraction is performed unconditionally.

Additional unsigned range observations:
- The wrapped `freeSize` is used to link the block into the free list. Pointer arithmetic based
  on it (`(char*)hdr + freeSize`) lands arbitrarily far in memory ‚Äî a direct heap-corruption
  primitive.
- `hdr->size` is also used to advance a traversal pointer in the allocator loop. A wrapped size
  would cause that loop to step by ~65K bytes, producing either a wild read or an infinite loop
  (if the pointer wraps around address space).

Arithmetic: ‚öÝ RISK ‚Äî unsigned subtraction `hdr->size - OSAL_MEM_HDRSZ` with no underflow guard;
corrupted header causes wrap to ~SIZE_MAX and corrupts the free list. Fix must add the guard
before the subtraction.

---

### 3. Freed-resource access

`osal_mem_free` acquires a critical section at entry and releases it at all normal exit points.
The fix introduces a new early-return path (the corruption guard). Two resource risks follow:

1. **Critical-section leak on early return** ‚Äî if the guard fires after `HAL_ENTER_CRITICAL_SECTION`
   but before `HAL_EXIT_CRITICAL_SECTION`, and the guard returns without calling the exit macro,
   the system is left with interrupts masked (or the lock held). On a bare-metal RTOS like
   TI-OSAL, this deadlocks any subsequent interrupt-driven path indefinitely.

2. **Block leak is intentional, not a bug** ‚Äî when the guard fires, the block is not re-inserted
   into the free list. This is the correct safe behavior: leaking one block is vastly preferable
   to corrupting the entire heap. The guard site should document this explicitly so reviewers do
   not "fix" it by removing the guard.

Resources: ‚öÝ RISK ‚Äî the new early-return guard must call `HAL_EXIT_CRITICAL_SECTION` before
returning; omitting it deadlocks the system on a bare-metal target.

---

**Execution fit: ADJUST PLAN**

‚Üí Add an underflow guard before the subtraction and ensure `HAL_EXIT_CRITICAL_SECTION` is called
on every return path the guard can take, including the new early-return.

---

## Adjusted plan (ordered steps)

**Step 1 ‚Äî Read the file.**

Open `$LOCI_TEST_BLE_ROOT/simplelink-lowpower-f3-sdk/source/ti/ble5stack_flash/osal/src/common/osal_memory.c`
and locate `osal_mem_free`. Identify:

- The exact spelling of the size subtraction expression (may be `hdr->size - OSAL_MEM_HDRSZ` or
  a variant with a different field/macro name).
- Where `HAL_ENTER_CRITICAL_SECTION` is called and the name of its argument (e.g., `intState`).
- All existing `return` statements ‚Äî confirm each is preceded by `HAL_EXIT_CRITICAL_SECTION`.

**Step 2 ‚Äî Add the underflow guard immediately before the subtraction.**

The guard must be placed *after* `HAL_ENTER_CRITICAL_SECTION` so that the paired exit call is
reachable. Insert:

```c
if ( hdr->size < OSAL_MEM_HDRSZ )
{
  /* Corrupted block header: the stored size is smaller than the header itself.
     Performing hdr->size - OSAL_MEM_HDRSZ would wrap to a large unsigned value
     and corrupt the free list. Exit the critical section and return without
     modifying the allocator state. The block is intentionally leaked ‚Äî leaking
     one block is safer than corrupting the heap. */
  HAL_EXIT_CRITICAL_SECTION( intState );
  return;
}
```

Placement checklist:
- After `HAL_ENTER_CRITICAL_SECTION( intState )` ‚Äî so the exit call is correctly paired.
- Before any use of `hdr->size - OSAL_MEM_HDRSZ` ‚Äî so the subtraction is never reached with
  a corrupt value.
- If `hdr->size` is also used before the subtraction (e.g., a traversal check), add the guard
  as early as possible ‚Äî before the first use that would produce a bad pointer.

**Step 3 ‚Äî Audit all return paths after the edit.**

Trace every `return` in `osal_mem_free` and confirm each one is preceded by
`HAL_EXIT_CRITICAL_SECTION`. The new guard is the only path that was added; confirm it has the
exit call.

**Step 4 ‚Äî Add a debug assertion (optional but strongly recommended).**

In debug builds, surface the corruption loudly before the guard silently swallows it:

```c
OSAL_ASSERT( hdr->size >= OSAL_MEM_HDRSZ );
if ( hdr->size < OSAL_MEM_HDRSZ )
{
  HAL_EXIT_CRITICAL_SECTION( intState );
  return;
}
```

`OSAL_ASSERT` should expand to a halt/breakpoint in debug and to nothing in release, so the
release binary retains only the safe silent guard.

**Step 5 ‚Äî Review callers of `osal_mem_free`.**

The guard prevents heap corruption but does not identify the root cause. Search the codebase
for all direct and indirect calls to `osal_mem_free` and check whether any caller validates
its pointer or size before calling. The source of the corrupt header is a separate bug; the
guard buys safety but the root cause warrants a follow-up investigation.

**Step 6 ‚Äî Test.**

Write or extend a unit test that:

1. Allocates a block via `osal_mem_alloc`.
2. Artificially sets `hdr->size` to a value smaller than `OSAL_MEM_HDRSZ` (e.g., 0 or 1).
3. Calls `osal_mem_free` on the block.
4. Verifies the free list is unchanged (the block was not inserted).
5. Verifies the system is not deadlocked (the test continues to run ‚Äî confirms critical section
   was exited).
6. Verifies no memory outside the block was written (requires a heap-canary or sanitizer check).
