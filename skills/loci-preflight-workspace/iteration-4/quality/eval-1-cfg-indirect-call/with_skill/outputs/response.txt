## Plan: Add logging hook to `osal_set_event`

### Context

File: $LOCI_TEST_BLE_ROOT/simplelink-lowpower-f3-sdk/source/ti/ble5stack_flash/osal/src/common/osal.c
Function: `osal_set_event`

The standard OSAL implementation of `osal_set_event` looks approximately like:

```c
uint8 osal_set_event( uint8 task_id, uint16 event_flag )
{
  if ( task_id < tasksCnt )
  {
    halIntState_t   intState;
    HAL_ENTER_CRITICAL_SECTION(intState);
    tasksEvents[task_id] |= event_flag;  // Assign the task's event bits
    HAL_EXIT_CRITICAL_SECTION(intState);
    return ( SUCCESS );
  }
  else
  {
    return ( INVALID_TASK );
  }
}
```

The proposed change adds:
1. A global function pointer: `void (*logFn)(uint8 taskID, uint16 event)` initialized to `NULL`.
2. A call to `logFn(task_id, event_flag)` immediately before `tasksEvents[task_id] |= event_flag`.

No binary is available in this environment — $LOCI_TEST_BLE_ROOT is not mounted, and no `.o`,
`.elf`, or `.loci-build/` artifacts exist in the project. Proceeding with static analysis only.

## Preflight: `osal_set_event` (logging hook via function pointer)

### 1. Call Graph (no binary — static analysis only)

The existing callees of `osal_set_event` are:
- `HAL_ENTER_CRITICAL_SECTION` / `HAL_EXIT_CRITICAL_SECTION` — macro-expanded critical section
  wrappers; typically expand to inline assembly or a single instruction on Cortex-M targets
  (CPSID/CPSIE or equivalent).

The proposed change adds one new call site: `logFn(task_id, event_flag)`.

**Indirect call hazard — this is the primary CFG concern:**

`logFn` is a global function pointer. At the assembly level, the call compiles to a load of the
pointer from a data address followed by `BLX Rn` (or equivalent) on Cortex-M — an indirect call
through a register. This is flagged as a CFI (Control Flow Integrity) hazard:

- There is no compile-time guarantee that `logFn` points to a valid function at call time.
- If the global is left at its initial value (`NULL` / `0x00000000`), the indirect branch targets
  address 0. On most Cortex-M targets, address 0 is the vector table. Branching there with BLX
  causes a hard fault or, worse, silently executes whatever instruction is at that address.
- On targets without hardware CFI enforcement (no BTI, no shadow stack — typical for Cortex-M0+
  through M4), any value written to `logFn` by any code path (including a stack smash or a
  race-condition write) becomes a valid branch target.

**Call-order assumption — logFn must be set before `osal_set_event` is first called:**

The application is expected to set `logFn` at startup. However, OSAL tasks can post events during
their `osal_init_system` phase and before the application's main loop runs. If `osal_set_event` is
called before the application assigns `logFn`, the pointer is NULL and the call faults.

There is no guard or assertion in the proposed design enforcing initialization order. This is a
structural hazard.

**Reentrancy / interrupt context — `osal_set_event` is called from ISR context:**

`osal_set_event` is a documented ISR-callable function in TI's OSAL (the critical section wrap
exists precisely because it is called from interrupts). If `logFn` is set to a function that uses
RTOS primitives, blocking calls, or is not reentrant, calling it from interrupt context will cause
undefined behavior or deadlock.

The proposed design provides no annotation, typedef restriction, or runtime check to prevent a
non-interrupt-safe function from being installed.

Call graph: ⚠ RISK — indirect call through `logFn` (NULL-pointer fault if not set before first
            event post; unguarded indirect branch is a CFI hazard; no enforcement that the
            installed function is interrupt-safe since `osal_set_event` is ISR-callable).

### 2. Arithmetic Ranges

No new arithmetic expressions are introduced by the proposed change. The arguments passed to
`logFn` are:
- `task_id` (uint8): already validated by the `task_id < tasksCnt` guard above the call site.
- `event_flag` (uint16): passed through unchanged; no arithmetic performed on it.

No overflow, wraparound, shift hazard, or signed/unsigned mixing is introduced.

Arithmetic: OK

---

### 3. Freed-Resource Access

The proposed insertion point is inside the critical section, after `HAL_ENTER_CRITICAL_SECTION`
and before `HAL_EXIT_CRITICAL_SECTION`. This creates two resource-lifetime concerns:

**Critical section held across the logFn call:**

If `logFn` is placed inside the critical section (between ENTER and EXIT), then for the duration
of the logging callback, the system's critical section is held — interrupts are masked. Any
logging function that does I/O, calls `osal_msg_send`, or takes any non-trivial time will hold
off all interrupt-driven events, extending interrupt latency system-wide. This is a latency hazard
in a real-time BLE stack context.

If `logFn` is placed outside the critical section (before ENTER), then there is a TOCTOU window:
`event_flag` and `task_id` are consistent at call time, but `tasksEvents[task_id]` has not yet
been updated. The logger sees the pre-post state, which is the stated intent — but this ordering
must be explicit in the plan.

**No heap allocation, no freed pointer access:**

The logging hook itself does not allocate or free anything in the proposed design. The global
`logFn` pointer is a simple read (not freed, not moved). No use-after-free, double-free, or RAII
gap is introduced by the change to `osal_set_event` itself, assuming the installed callback is
well-behaved.

**Dangling reference risk (application side):**

If the application sets `logFn` to point to a function in a module that is later unloaded (unusual
for bare-metal but possible in some RTOS configurations), the pointer becomes dangling.
This is a design-level concern, not directly addressable within `osal_set_event` itself.

Resources: ⚠ RISK — if `logFn` is called inside the critical section, interrupt latency is
           unbounded for the duration of the callback; if called outside, the ordering must be
           documented explicitly.

## Adjusted Plan

### Step 1 — Define the function pointer type with an ISR-safety annotation

Before the global declaration, add a typedef that makes the calling-convention and ISR
expectation explicit:

```c
/* Logging callback type for osal_set_event.
 * MUST be interrupt-safe: no blocking calls, no RTOS primitives, no osal_msg_send.
 * Set once at startup before osal_init_system(); do not change at runtime. */
typedef void (*OsalEventLogFn_t)(uint8 taskID, uint16 event);

OsalEventLogFn_t logFn = NULL;
```

Using a named typedef (rather than a raw `void (*logFn)(...)`) makes the signature
self-documenting and allows static-analysis tools to track the pointer type.

### Step 2 — Call logFn BEFORE the critical section, with a NULL guard

The critical finding is that `logFn` must not be called while the critical section is held.
Place the call before `HAL_ENTER_CRITICAL_SECTION`:

```c
uint8 osal_set_event( uint8 task_id, uint16 event_flag )
{
  if ( task_id < tasksCnt )
  {
    halIntState_t intState;

    /* Logging hook: called before the event is posted and before the critical section
     * is entered, so interrupt latency is not extended.
     * logFn must be interrupt-safe if osal_set_event is called from ISR context. */
    if ( logFn != NULL )
    {
      logFn( task_id, event_flag );
    }

    HAL_ENTER_CRITICAL_SECTION(intState);
    tasksEvents[task_id] |= event_flag;
    HAL_EXIT_CRITICAL_SECTION(intState);
    return ( SUCCESS );
  }
  else
  {
    return ( INVALID_TASK );
  }
}
```

The NULL guard directly addresses the call-graph hazard: a NULL `logFn` no longer faults;
`osal_set_event` works safely before the application sets `logFn`.

### Step 3 — Document the race window in a comment

With `logFn` called before the critical section, there is a narrow window between the log call
and the actual `tasksEvents` update. Add a comment at the call site:

```c
/* NOTE: logFn is called before tasksEvents is updated. The logged event_flag reflects
 * the *about-to-be-posted* value, not the post-update bitmask. This is intentional:
 * calling inside the critical section would extend interrupt latency unboundedly. */
```

### Step 4 — Add a startup assertion (debug builds)

In the application's initialization code (before `osal_init_system()`):

```c
logFn = myEventLogger;          // Set before any events can be posted
osal_init_system();
```

In debug builds, add an assertion that catches any path where events are posted before logFn
is set — this is opt-in since the NULL guard makes the release build safe.

### Step 5 — Verify no existing callers post events before logFn is assignable

Search for calls to `osal_set_event` within `osal_init_system` or its callees. If any exist,
`logFn` must be set before `osal_init_system()` is called, not after.

---

## Summary of Preflight Findings

The two findings that change the plan:

1. **Indirect call / NULL fault (call graph):** The raw function pointer call without a NULL guard
   faults on the first event post if `logFn` has not been set — which can happen during
   `osal_init_system`. The NULL guard (`if (logFn != NULL)`) resolves this and eliminates the
   hard fault risk without requiring a mandatory initialization step from the application.

2. **Critical section scope (resources):** Placing the `logFn` call inside the critical section
   extends interrupt latency for the entire duration of the callback, which is unbounded and
   violates the real-time requirements of a BLE stack. Moving the call to before
   `HAL_ENTER_CRITICAL_SECTION` resolves this at the cost of a minor TOCTOU window (the log
   sees the pre-post state), which is acceptable and should be documented.

The CFI hazard (indirect call through a register) is inherent to the function-pointer design and
cannot be fully resolved without hardware CFI support or a dispatch table with range-checking.
Document the ISR-safety requirement in the typedef as the practical mitigation.
