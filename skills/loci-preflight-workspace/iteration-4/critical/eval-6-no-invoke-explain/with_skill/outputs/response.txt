GAPBondMgr_Init in the TI SimpleLink BLE5 stack (gapbondmgr.c) initializes the GAP Bond Manager, which is the component responsible for managing bonding, pairing, and long-term key (LTK) storage between BLE devices. The function must be called once during application startup, before any BLE connection or pairing activity begins.

The function signature is:

    bStatus_t GAPBondMgr_Init(gapBondParams_t *pParams);

or in some SDK versions:

    bStatus_t GAPBondMgr_Init(uint8_t taskID);

In the full-parameter form used in the SimpleLink Low Power F3 SDK, the initialization accepts a pointer to a parameter structure. The key responsibilities of GAPBondMgr_Init are:

1. Registering the Bond Manager as an internal OSAL task so it can receive HCI and GAP events related to pairing and encryption.
2. Restoring any previously saved bond records from non-volatile storage (NV) into the in-RAM bond table, so previously bonded peers are recognized immediately on the next connection.
3. Setting up the Security Manager (SM) with the local device's pairing capabilities, I/O capabilities, and key distribution preferences.
4. Registering callback hooks so the Bond Manager intercepts pairing-related GAP events (authentication, key distribution, encryption change) before they reach the application.

Parameters (gapBondParams_t fields or direct arguments depending on SDK version):

- taskID (uint8_t): The OSAL task ID of the application task. The Bond Manager uses this to forward bonding events (such as GAPBOND_PAIRING_STATE_EVT or GAPBOND_PASSCODE_NEEDED_EVT) to the application when user interaction is required or when a pairing completes.

- pairMode (uint8_t): Controls how the device responds to pairing requests from a remote peer.
  - GAPBOND_PAIRING_MODE_NO_PAIRING   — reject all pairing requests outright
  - GAPBOND_PAIRING_MODE_WAIT_FOR_REQ — accept pairing only when the remote initiates it
  - GAPBOND_PAIRING_MODE_INITIATE     — proactively send a pairing request on every new connection

- mitm (uint8_t): Boolean flag. When TRUE, the Bond Manager requires Man-In-The-Middle (MITM) protection during pairing. This forces the pairing procedure to use an authenticated method (Passkey Entry or Numeric Comparison) rather than Just Works, ensuring the resulting LTK is authenticated.

- ioCap (uint8_t): Declares the local device's I/O capabilities to the Security Manager. This value is exchanged with the remote peer during the pairing feature exchange and determines which pairing method is selected. Standard values are:
  - GAPBOND_IO_CAP_DISPLAY_ONLY       — can display a passkey, cannot accept input
  - GAPBOND_IO_CAP_DISPLAY_YES_NO     — can display and confirm yes/no
  - GAPBOND_IO_CAP_KEYBOARD_ONLY      — can accept numeric input, cannot display
  - GAPBOND_IO_CAP_NO_INPUT_NO_OUTPUT — no I/O; forces Just Works pairing
  - GAPBOND_IO_CAP_KEYBOARD_DISPLAY   — full keyboard and display; preferred for MITM

- bonding (uint8_t): Boolean flag. When TRUE, the device requests that long-term keys be saved after a successful pairing, enabling encrypted reconnections without re-pairing. When FALSE, the session keys are used for the current connection only and not stored.

- secureConnection (uint8_t): Controls LE Secure Connections (LESC) support.
  - GAPBOND_SECURE_CONNECTION_NONE    — use only Legacy Pairing (LE Legacy)
  - GAPBOND_SECURE_CONNECTION_ALLOW   — accept both Legacy and Secure Connections
  - GAPBOND_SECURE_CONNECTION_ONLY    — reject Legacy Pairing; require LESC

- authenPairingOnly (uint8_t): When TRUE, the Bond Manager rejects any pairing attempt that would result in an unauthenticated key (i.e., Just Works with no MITM). This is a stricter enforcement layer on top of the mitm flag.

- autoSyncWL (uint8_t): When TRUE, the Bond Manager automatically synchronizes the controller's White List (Filter Accept List) with the set of bonded device addresses stored in NV. This allows the application to enable White List filtering on advertising or scanning without manually managing the list.

- eccReGenPolicy (uint8_t): Controls how often the local ECC key pair used for LE Secure Connections is regenerated. A value of 0 means regenerate on every pairing; higher values allow the same key pair to be reused across multiple pairings before rotation, trading security margin for speed.

- maxBonds (uint8_t): The maximum number of bonded peers the Bond Manager will store. When the table is full and a new bond needs to be saved, the oldest or least-recently-used entry is evicted. This value must not exceed the compile-time NV storage allocation.

- maxCharCfg (uint8_t): The maximum number of GATT Client Characteristic Configuration (CCC) descriptor values the Bond Manager will persist per bond. Storing CCC values allows the device to restore notification/indication subscriptions for each bonded client across reconnections without requiring the client to re-subscribe.

- gapBondManagerTaskID (uint8_t, internal): Set by the function itself (not passed in); records the OSAL task ID assigned to the Bond Manager's own internal task for event routing.

Return value:
  SUCCESS (0) if initialization completed and NV bond records were loaded successfully.
  A non-zero error code if NV access failed or an invalid parameter was detected.

Initialization flow summary:
  GAPBondMgr_Init should be called after GAP_DeviceInit and before GAPRole_StartDevice (or equivalent). Once it returns SUCCESS, the Bond Manager is active and will intercept all subsequent pairing and encryption events. The application interacts with bonding behavior primarily through the callback registered via GAPBondMgr_Register, which the Bond Manager calls when it needs a passcode, when pairing completes, or when a bond is created or deleted.
