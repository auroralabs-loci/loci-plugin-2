## Preflight: GAPBondMgr_FindAddr (bond-index output variant)

*(no binary — static analysis only)*

**Understanding the existing function:**

`GAPBondMgr_FindAddr` searches the bond table for an entry matching a given peer address and address type. It returns a `bStatus_t` (SUCCESS or a failure code) indicating whether a match was found. The requested change adds a new `uint8_t *pBondIdx` output parameter so callers can learn which slot in the bond table matched, without a separate lookup.

### 1. Call Graph

`GAPBondMgr_FindAddr` iterates from `0` to `GAP_BONDINGS_MAX - 1`, calling an NV read helper (typically `osal_snv_read` or a `gapBondMgrReadBondRec`-style wrapper) to load each bond record, then comparing the stored address with the input address using `MAP_osal_memcmp` or equivalent.

Key callees for the new function:
- Same NV read wrappers as the existing function — no new callees are introduced.
- The bond index write (`*pBondIdx = i`) is a direct pointer store, not a function call.

Hazards (static reasoning):
- **Call-order assumption**: The bond manager must be initialized (NV subsystem initialized, `GAPBondMgr_Register` called) before any find operation. There is no visible guard enforced in source — the new function inherits this precondition silently. If called before initialization, NV reads may return stale or zero-initialized data, and `*pBondIdx` would be written with whatever index happened to compare equal, producing a false positive.
- **Indirect calls**: NV access wrappers on this stack route through `MAP_` prefix function pointer tables. These are indirect calls. This is pre-existing and not introduced by the change, but worth noting for call-graph tracing.
- No recursion or back-edges expected.

### 2. Arithmetic Ranges

- Bond index ranges `0` to `GAP_BONDINGS_MAX - 1` (commonly `0..7` or `0..15`). The loop counter fits in `uint8_t`; no overflow.
- **Risk — uninitialized output on not-found path**: If no match is found, `*pBondIdx` is never written. The caller receives an uninitialized (or previously garbage) value if it reads `*pBondIdx` after a non-SUCCESS return. Plan to write a sentinel (`GAP_BONDINGS_MAX` or `0xFF`) at function entry so the output is always defined.
- **Risk — NULL dereference**: If the caller passes `NULL` for `pBondIdx`, a blind `*pBondIdx = i` write will fault on a Cortex-M target (or produce undefined behavior). Plan to NULL-guard every write to `pBondIdx`.
- **Risk — GAP_BONDINGS_MAX == 0**: If the bond table is configured to zero (misconfigured build), the loop body never executes and `*pBondIdx` is never written, even though the sentinel was set at entry. The sentinel at entry handles this case correctly, but the loop guard should explicitly handle this so the compiler does not warn.
- No signed/unsigned mixing is expected in the loop; the return value (`bStatus_t`) and the index (`uint8_t`) remain separate.

### 3. Freed-Resource Access

- Bond records are read into stack-local structs inside the loop. No heap allocation is involved; no `malloc`/`free` paths exist in the expected implementation.
- The output parameter `pBondIdx` is caller-owned memory. The function writes through it but does not acquire or release it — no RAII obligation.
- No `std::move` or C++ resource transfer; this is a C codebase.
- No early-return resource leak: since no resources are acquired, all exit paths (not-found early return, loop exhaustion, match-found return) are clean.

## Adjusted Plan

### Step 1 — Read the existing function signature and loop structure

Before editing, confirm:
- The exact signature of `GAPBondMgr_FindAddr` (parameter list, return type).
- The name of the NV read helper used inside the loop.
- Whether `GAP_BONDINGS_MAX` is defined in a header or computed at runtime.
- The location of the corresponding declaration in `gapbondmgr.h` (or equivalent API header).

### Step 2 — Define the new function (or extend the existing one)

Two design options:

**Option A (new function, preferred for API stability):** Add a new function `GAPBondMgr_FindAddrEx` that takes all existing parameters plus `uint8_t *pBondIdx`. Implement it by delegating to or duplicating the loop logic. Existing callers of `GAPBondMgr_FindAddr` are unaffected.

**Option B (extend existing function):** Add `uint8_t *pBondIdx` to the existing `GAPBondMgr_FindAddr` signature. Update all existing call sites in the stack to pass `NULL` for `pBondIdx` if they don't need it.

Option A is lower risk (no call-site churn); Option B is simpler if call sites are few. Confirm with the team before proceeding.

### Step 3 — Write the sentinel initialization at function entry

```c
if (pBondIdx != NULL)
{
  *pBondIdx = GAP_BONDINGS_MAX; /* sentinel: not found */
}
```

This ensures `*pBondIdx` is always defined even if no match is found or `GAP_BONDINGS_MAX == 0`.

### Step 4 — Write the index on match, with NULL guard

Inside the loop, when a match is found:

```c
if (pBondIdx != NULL)
{
  *pBondIdx = (uint8_t)idx;
}
return SUCCESS;
```

The NULL guard allows callers that do not need the index to pass `NULL` safely, consistent with TI BLE5 API conventions for optional output parameters.

### Step 5 — Add the initialization precondition comment

At the top of the function body:

```c
/* Precondition: NV subsystem and bond manager must be initialized before
 * calling this function. Results are undefined if called before GAPBondMgr_Register
 * has been invoked and NV has been initialized. */
```

### Step 6 — Update the header declaration

Update `gapbondmgr.h` (or the relevant API header) to declare the new signature or new function. Export it only if it belongs to the public API; if it is an internal helper, keep it `static` or add it to the internal header only.

### Step 7 — Update call sites (if Option B)

Search for all call sites of `GAPBondMgr_FindAddr` in the stack and add `NULL` as the trailing argument. Build and verify no callers are broken.
