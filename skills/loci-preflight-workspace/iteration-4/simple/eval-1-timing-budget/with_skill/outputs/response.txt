## Preflight: HCI_SendDataPkt (adding GAPBondMgr_FindAddr call)

(no binary — static analysis only; LOCI MCP and local ELF access were unavailable at analysis time)

---

### 1. Call graph (CFG analysis)

Callee: GAPBondMgr_FindAddr

Based on function name, SDK context (TI BLE5 stack, CC2340R5), and typical BLE bond manager
implementations:

- GAPBondMgr_FindAddr performs a **linear scan** over all stored bond records, comparing a
  6-byte Bluetooth Device Address (BDA) per entry using an internal memcmp equivalent.
- The CC2340R5 supports up to 10 bond entries by default (configurable via GAP_BONDINGS_MAX).
- Each iteration: load bond record pointer, 6-byte compare, branch. On ARM Cortex-M0+ at 48 MHz:
  - osal_memcmp over 6 bytes ≈ 6–10 cycles/byte unrolled, or ~50–80 cycles if looped.
  - Loop overhead per bond entry: ~15–25 cycles.
  - Worst case (no match, 10 entries): ~10 × (50 + 20) = ~700 cycles.
  - Best case (first entry matches): ~70–100 cycles.
  - Average (5 entries, middle match): ~350–450 cycles.
- Function call overhead (PUSH/POP, BL): ~10–15 cycles round-trip.

Call graph flags:
- **No indirect calls or cycles** visible from function semantics.
- **Call-order assumption**: GAPBondMgr_FindAddr reads from the bond table (NV-backed RAM cache).
  This table must be initialized before HCI_SendDataPkt is called. The GAP bond manager is
  initialized at stack startup, so this ordering is met in normal operation — but the check does
  not enforce it. Flag for documentation, not a blocker.
- **Dead path**: if called before bonding is initialized, FindAddr may return 0xFF (not found)
  spuriously, causing HCI_SendDataPkt to drop valid data. This is a logic hazard, not a CFG hazard.

Call graph: ⚠ RISK — GAPBondMgr_FindAddr has O(N) worst-case latency (N = bond count).
             Call-order assumption (bond table initialized) is met at runtime but unenforced.

### 2. Arithmetic ranges

HCI_SendDataPkt itself:
- No new arithmetic is introduced by the bond check insertion — only a function call and a
  conditional branch on the return value.
- GAPBondMgr_FindAddr return value: uint8 bond index (0–9) on match, 0xFF on not-found. No
  overflow or wraparound risk in consuming this value as a boolean (CMP / BEQ pattern).

Arithmetic: OK

### Timing Budget Analysis

Budget: 10 μs at 48 MHz = **480 cycles total** for the entire HCI_SendDataPkt execution path.

Baseline HCI_SendDataPkt cost (without bond check):
- Function prologue/epilogue: ~15 cycles
- Buffer pointer validation, header reads: ~20 cycles
- osal_msg_allocate (OS heap alloc): ~80–150 cycles
- memcpy of payload: depends on payload length; minimum ~30 cycles for small packets
- osal_msg_send (OS queue post): ~60–100 cycles
- Estimated baseline total: ~205–315 cycles (without payload copy scaling)

Cost of adding GAPBondMgr_FindAddr at top:
- BL + PUSH/POP overhead: ~15 cycles
- GAPBondMgr_FindAddr body, worst case (10 bonds, no match): ~700 cycles
- GAPBondMgr_FindAddr body, best case (1 bond, immediate match): ~85 cycles
- GAPBondMgr_FindAddr body, typical case (5 bonds): ~350–400 cycles

Combined worst-case estimate:
  ~315 (baseline) + ~700 (FindAddr worst) = ~1,015 cycles

Combined typical-case estimate:
  ~260 (baseline) + ~375 (FindAddr typical) = ~635 cycles

Combined best-case estimate:
  ~205 (baseline) + ~85 (FindAddr best) = ~290 cycles

**The 480-cycle budget is violated in all cases except best-case (single bond, immediate match).**

Even the typical case (~635 cycles) exceeds the budget by ~32%.
The worst case (~1,015 cycles) exceeds it by ~111%.

## Preflight: HCI_SendDataPkt

Call graph:  ⚠ RISK — GAPBondMgr_FindAddr is O(N) over bond count; worst-case ~700 cycles
              alone. Unenforced call-order assumption on bond table initialization.
Arithmetic:  OK
Resources:   OK

Execution fit: ADJUST PLAN

→ Adding a GAPBondMgr_FindAddr call at the top of HCI_SendDataPkt will violate the 480-cycle /
  10 μs budget in typical and worst-case scenarios. The bond check must be moved out of the
  hot data path entirely.

---

### Recommended plan adjustments (before writing any code)

1. **Do not insert GAPBondMgr_FindAddr inside HCI_SendDataPkt.** The HCI data path is a
   hard-latency context; a table-scan lookup does not belong here.

2. **Cache the bond-verified state at connection time.** When a connection is established
   (HCI_LE_ConnectionCompleteEvent or GAP_LINK_ESTABLISHED_EVENT), call GAPBondMgr_FindAddr
   once and store a `isBonded` flag in the per-connection control block. Cost: zero cycles on
   the data path.

3. **Gate HCI_SendDataPkt on the cached flag.** At the top of HCI_SendDataPkt, load the
   per-connection `isBonded` field (1 load + 1 compare + 1 branch ≈ 5–8 cycles). This keeps
   the data path well within 480 cycles.

4. **Add an assertion** (debug builds only) that the bond manager is initialized before the
   first connection event fires, to enforce the call-order assumption explicitly.

These changes preserve the security intent (reject unverified peers) without adding
unbounded-latency work to the real-time data path.
