## How to Modify Connection_getConnIndex to Return -1 for Not-Found

### Current Implementation

In `/home/melisa/BLE/examples/rtos/LP_EM_CC2340R53/ble5stack/basic_ble_profiles/app/app_connection.c` (lines 443-456):

```c
uint16_t Connection_getConnIndex(uint16_t connHandle)
{
  uint8_t i;

  for (i = 0; i < MAX_NUM_BLE_CONNS; i++)
  {
    if (connectionConnList[i].connHandle == connHandle)
    {
      return i;
    }
  }

  return LL_INACTIVE_CONNECTIONS;
}
```

The function currently returns `LL_INACTIVE_CONNECTIONS` (defined in `ll.h` as `0xFF`) when no connection is found. The return type is `uint16_t`.

### Important Clarification

The question asks to return -1 "instead of 0 when no connection is found." The current not-found sentinel is actually `LL_INACTIVE_CONNECTIONS` (0xFF), not 0. Index 0 is already a valid found result (the first slot). The change needed is replacing the `LL_INACTIVE_CONNECTIONS` sentinel with -1.

### Why the Return Type Must Change

Returning -1 from a `uint16_t` function is legal C but semantically wrong: `(uint16_t)(-1)` wraps to `0xFFFF`. Any caller that stores the result in a signed variable and compares it to `-1` will get the wrong answer. The return type must be changed to `int16_t` so that -1 is an unambiguous signed negative value that cannot be confused with a valid array index.

### Step-by-Step Changes

**Step 1 — Change the return type in app_connection.c (line 443)**

```c
// Before:
uint16_t Connection_getConnIndex(uint16_t connHandle)

// After:
int16_t Connection_getConnIndex(uint16_t connHandle)
```

**Step 2 — Change the not-found return value in app_connection.c (line 455)**

```c
// Before:
  return LL_INACTIVE_CONNECTIONS;

// After:
  return -1;
```

**Step 3 — Update the declaration in app_main.h (line 318)**

```c
// Before:
uint16_t Connection_getConnIndex(uint16_t connHandle);

// After:
int16_t Connection_getConnIndex(uint16_t connHandle);
```

**Step 4 — Update the caller in app_pairing.c (lines 154-160)**

The only call site in this project is at line 154 of `app_pairing.c`:

```c
// Before:
uint8_t connIdx = Connection_getConnIndex(((BLEAppUtil_PairStateData_t *)pMsgData)->connHandle);
if (connIdx != LL_INACTIVE_CONNECTIONS)
{
  memcpy(connList[connIdx].peerAddress, linkInfo.addr, B_ADDR_LEN);
}

// After:
int16_t connIdx = Connection_getConnIndex(((BLEAppUtil_PairStateData_t *)pMsgData)->connHandle);
if (connIdx != -1)
{
  memcpy(connList[(uint8_t)connIdx].peerAddress, linkInfo.addr, B_ADDR_LEN);
}
```

Note the cast `(uint8_t)connIdx` when using `connIdx` as an array index. This suppresses the signed/unsigned mismatch warning and is safe because at that point the guard has already confirmed `connIdx >= 0`.

**Step 5 — Update the doxygen comments**

In `app_connection.c` (lines 440-441) and in `app_main.h` (lines 315-316), update the `@return` documentation:

```

 *          if there is no match, -1 will be returned.
```

### What NOT to Do

Do not return -1 while keeping `uint16_t` as the return type. The bit pattern `(uint16_t)(-1)` equals `0xFFFF`, which is not `LL_INACTIVE_CONNECTIONS` (0xFF) and is not -1 when compared in a signed context. This will silently break existing callers.

Do not remove or redefine `LL_INACTIVE_CONNECTIONS` — it is used by the BLE stack internally and by many other parts of the SDK unrelated to this function.
