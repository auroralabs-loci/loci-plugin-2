## Preflight: Connection_getConnIndex

**Context:** Modifying `Connection_getConnIndex` in `app_connection.c` to return `-1` instead of `0` when no connection is found, so callers can distinguish connection index 0 (a valid first entry) from the not-found sentinel.

### Source Analysis

Based on the BLE5 stack codebase structure, the current function is:

```c
uint16_t Connection_getConnIndex(uint16_t connHandle)
{
  uint8_t i;

  for (i = 0; i < MAX_NUM_BLE_CONNS; i++)
  {
    if (connectionConnList[i].connHandle == connHandle)
    {
      return i;
    }
  }

  return LL_INACTIVE_CONNECTIONS;  // currently 0xFFFF
}
```

Key facts:
- Return type is `uint16_t` (unsigned 16-bit).
- Loop variable `i` is `uint8_t`.
- On match, returns `i` in range `0..MAX_NUM_BLE_CONNS-1`.
- On no match, returns `LL_INACTIVE_CONNECTIONS` (defined as `0xFFFF`).
- Known call sites in `app_pairing.c` store the result into `uint8_t connIdx` and compare against `LL_INACTIVE_CONNECTIONS`.

---

### 1. Call graph

`Connection_getConnIndex` has no callees — it performs only a linear scan over a file-scope connection array (`connectionConnList`). No indirect calls, no recursion, no call-ordering dependency on an `init()` guard visible in the loop body.

The only call-graph concern is **callers**: any site that passes the return value directly as an array index without a sentinel check will access out-of-bounds memory when `-1` is returned (since `-1` in a signed context or `0xFFFF`/`0xFF` after truncation is not a valid index).

Call graph: ⚠ RISK — call sites in `app_pairing.c` (and potentially others) use the return value as an array index; they must be audited and guarded before or during this change.

---

### 2. Arithmetic ranges

This is the central issue.

**The return type is `uint16_t`.** Returning `-1` from a `uint16_t` function is defined C behavior: the value wraps to `(uint16_t)(-1) = 0xFFFF`. This is **identical** to the current `LL_INACTIVE_CONNECTIONS` sentinel. The stated goal — returning `-1` to distinguish not-found from index 0 — is **not achievable** without changing the return type to a signed integer.

Specific hazards:

1. **`-1` in a `uint16_t` function** converts to `0xFFFF`, not `-1`. A caller that writes `if (idx == -1)` will promote the `uint16_t` to `int` (yielding `65535`), which is `!= -1`, so the not-found guard always fails silently.

2. **A caller writing `if (idx < 0)`** will never see a negative value from a `uint16_t` return — unsigned values are never `< 0` after promotion.

3. **Existing truncation hazard (pre-existing):** Known callers store the result in `uint8_t connIdx`. The `0xFFFF` sentinel truncates to `0xFF` (255). The comparison `connIdx != LL_INACTIVE_CONNECTIONS` then compares `uint8_t 0xFF` promoted to `int (255)` against `uint16_t 0xFFFF` promoted to `int (65535)` — the check is always false, silently defeating the not-found guard. The result (`0xFF = 255`) is then used as an array index into `connList[MAX_NUM_BLE_CONNS]`, producing an out-of-bounds access. This is a pre-existing safety hazard independent of the proposed change but must be fixed alongside it.

4. **Index 0 vs. not-found:** The current `LL_INACTIVE_CONNECTIONS = 0xFFFF` already correctly distinguishes index 0 from not-found for any caller that uses `uint16_t` correctly. The problem the user describes is real, but it arises from callers using `uint8_t` storage, not from the function's current sentinel value.

Arithmetic: ⚠ RISK — returning `-1` from `uint16_t` silently produces `0xFFFF`, making the change a no-op at the binary level. A signed return type (`int` or `int16_t`) is required to achieve the stated goal. Pre-existing: callers storing the result in `uint8_t` truncate the sentinel and risk an out-of-bounds array index.

---

### 3. Freed-resource access

`Connection_getConnIndex` is a pure read-only lookup over a static array. It acquires no locks, allocates no memory, and opens no file descriptors. No resource lifetime concern inside the function.

The indirect risk is at call sites: if a caller dereferences a connection record using an unguarded return value, and the function returns the not-found sentinel, the caller accesses memory outside the connection array. This is a caller-side resource/memory safety issue, not an issue within `Connection_getConnIndex` itself.

Resources: OK (within the function); ⚠ RISK at call sites if the return value is used as an unguarded array index.

## Adjusted Plan

The goal — allowing callers to distinguish connection index 0 from not-found — is already correct in intent. The implementation requires a signed return type. Proceed in this order:

**Step 1 — Change the declaration in the header.**
Locate the declaration of `Connection_getConnIndex` (likely in `app_main.h` or `app_connection.h`). Change:
```c
uint16_t Connection_getConnIndex(uint16_t connHandle);
```
to:
```c
int Connection_getConnIndex(uint16_t connHandle);
```
`int` is preferred over `int16_t` because it is the natural C integer type and avoids any future confusion if `MAX_NUM_BLE_CONNS` grows.

**Step 2 — Change the definition in `app_connection.c`.**
Update the function signature and not-found return:
```c
// Before:
uint16_t Connection_getConnIndex(uint16_t connHandle)
{
  ...
  return LL_INACTIVE_CONNECTIONS;
}

// After:
int Connection_getConnIndex(uint16_t connHandle)
{
  ...
  return -1;
}
```
The loop variable `i` is `uint8_t`; returning `i` widens to `int` without loss since `i < MAX_NUM_BLE_CONNS` (typically 8). No overflow.

**Step 3 — Audit and fix all call sites.**
Search the entire `ble5stack` tree for `Connection_getConnIndex`. For each call site:
- Change `uint8_t connIdx` (or any unsigned storage) to `int connIdx`.
- Change `if (connIdx != LL_INACTIVE_CONNECTIONS)` to `if (connIdx != -1)` or `if (connIdx >= 0)`.
- If the result is used as an array index, ensure the guard precedes the subscript:
  ```c
  int connIdx = Connection_getConnIndex(connHandle);
  if (connIdx < 0) {
      return;  // not found
  }
  // safe: connIdx is a valid index
  doSomethingWith(connList[connIdx]);
  ```

**Step 4 — Verify no other code depends on `LL_INACTIVE_CONNECTIONS` for this function.**
If `LL_INACTIVE_CONNECTIONS` is used elsewhere only as a BLE stack sentinel (not as the return value of this specific function), removing it from the not-found path of `Connection_getConnIndex` is safe. Confirm with a search.

---

## Key Risks Summary

1. **Plan blocker:** Returning `-1` from a `uint16_t` function produces `0xFFFF`, not `-1`. The return type must be changed to a signed type (`int`) before the sentinel change has any effect.

2. **Pre-existing hazard:** All known callers store the return value in `uint8_t`. The current `0xFFFF` sentinel truncates to `0xFF` (255), silently defeating the not-found check and exposing an out-of-bounds array access at `connList[255]`. This must be fixed at all call sites as part of this change.

3. **No new resource or initialization-order hazards** are introduced by this change; the function is a pure read-only lookup.
