The task is to write a function that validates a BLE connection handle before it is passed to HCI_SendDataPkt in hci.c, returning an error code if the handle is out of range.

### Step 3 — Arithmetic ranges

The BLE connection handle is a 12-bit field per the Bluetooth Core Specification. Valid values are 0x0000–0x0EFF (0–2815 decimal); values 0x0F00–0xFFFF are reserved by the spec.

- The parameter type should be uint16_t to match HCI_SendDataPkt's own handle parameter type.
- The upper-bound check `connHandle > 0x0EFFu` is a straightforward unsigned comparison on a uint16_t — no overflow, no wraparound risk for the comparison itself.
- ⚠ RISK — Signed/unsigned promotion hazard: if any caller passes the handle as a signed int or int16_t, the implicit promotion to uint16_t silently wraps negative values to large positive values (e.g., -1 becomes 0xFFFF), which the guard will catch — but only by accident, not by design. The plan must explicitly declare the parameter as uint16_t and document the expected type to prevent callers from using signed types.
- No multiplication, shift, or addition in the validator's own arithmetic; no further overflow risk.

- No RAII concern. No use-after-free or double-free hazard.

## Adjusted Plan

**1. Confirm the handle parameter type**

Read the signature of HCI_SendDataPkt in hci.c and its declaration in the corresponding header (likely hci.h). Confirm the handle parameter is uint16_t. If any callers pass a signed type, note them for a follow-up cast fix. This step must happen before writing any code.

**2. Find all HCI_SendDataPkt call sites**

Search hci.c (and any callers in the ble5stack_flash tree) for every invocation of HCI_SendDataPkt. The validator must be inserted before each call site, not just the one mentioned in the prompt — otherwise the guard is incomplete and other paths remain unprotected.

**3. Identify or define the error return code**

Check hci.h, bcomdef.h, and the HCI error-code enum for an existing code that fits "invalid connection handle". TI's stack commonly defines HCI_ERROR_CODE_INVALID_HCI_CMD_PARAMS. If no suitable code exists, add one:

```c
/* In hci.h or the appropriate error-code header */
#define HCI_ERROR_INVALID_CONN_HANDLE   ((uint8_t)0x02U)  /* or next available slot */
```

Reuse an existing code if one is semantically appropriate; do not introduce a duplicate.

**4. Write the validator function**

Place it in hci.c, above HCI_SendDataPkt, with internal linkage (static) unless it needs to be called from outside the file. If external linkage is needed, add a forward declaration to hci.h.

```c
/*
 * validateBLEConnHandle
 *
 * Validates a BLE connection handle against the legal range defined by the
 * Bluetooth Core Specification (Vol 4, Part E, §7.1.5):
 *   valid range: 0x0000–0x0EFF
 *   reserved:    0x0F00–0xFFFF
 *
 * Parameters:
 *   connHandle — the connection handle to validate; must be uint16_t to
 *                match HCI_SendDataPkt's parameter type and avoid signed-to-
 *                unsigned promotion at the call site.
 *
 * Returns:
 *   HCI_SUCCESS (0)                  if the handle is within the valid range.
 *   HCI_ERROR_INVALID_CONN_HANDLE    if the handle is out of range.
 */
static uint8_t validateBLEConnHandle(uint16_t connHandle)
{
    /* Valid BLE connection-handle range: 0x0000–0x0EFF (Bluetooth Core Spec
     * Vol 4 Part E §7.1.5). Values 0x0F00–0xFFFF are reserved. */
    if (connHandle > 0x0EFFu)
    {
        return HCI_ERROR_INVALID_CONN_HANDLE;
    }
    return HCI_SUCCESS;
}
```

**5. Insert the validator at each HCI_SendDataPkt call site**

At every call site identified in step 2, insert a guard before the HCI_SendDataPkt call:

```c
uint8_t status = validateBLEConnHandle(connHandle);
if (status != HCI_SUCCESS)
{
    return status;
}
/* original HCI_SendDataPkt call follows */
HCI_SendDataPkt(connHandle, pbFlag, pktLen, pData);
```

**6. Verify no existing callers bypass the new validator**

After inserting guards, re-read the call-site list from step 2 and confirm every site is covered. If HCI_SendDataPkt is called from interrupt context (ISR), note that the validator itself is safe (no blocking, no allocation) but the error return path must be compatible with the ISR's error-handling convention.

---

## Key risks to watch during implementation

- Incomplete guard coverage: inserting the validator on only one of multiple HCI_SendDataPkt call paths leaves the others unprotected. Step 2 (auditing all call sites) is mandatory before writing any code.
- Signed parameter at a call site: if any caller passes a signed integer as the handle, the guard will still catch out-of-range values (due to wrap-to-large-positive behavior), but this is fragile. Add an explicit (uint16_t) cast at any such call site.
- Error-code collision: if an existing error code is redefined with a different numeric value, the change will break other error-handling logic. Reuse the existing code or pick a value that does not collide.
