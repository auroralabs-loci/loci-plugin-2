Here is a plan for writing a connection handle validation function and integrating it into `HCI_SendDataPkt` in the TI BLE5 stack HCI layer:

## Plan: Validate BLE Connection Handle Before HCI_SendDataPkt

### Overview

Add a helper function `HCI_ValidateConnHandle` that checks whether a given connection handle falls within the valid range before it is used in `HCI_SendDataPkt`. If the handle is out of range, the function returns an appropriate BLE error code and `HCI_SendDataPkt` returns early without sending the packet.

---

### Step 1: Identify the Valid Handle Range

In the BLE specification, connection handles are 12-bit values in the range `0x0000` to `0x0EFF`. Values `0x0F00` and above are reserved. The TI BLE5 stack typically defines this upper bound via a macro such as `LL_CONNHANDLE_INVALID` or a similar constant. Confirm the exact macro name and value in the HCI or LL header files before coding.

A typical guard:

```c
#define HCI_MAX_CONN_HANDLE   0x0EFF
```

---

### Step 2: Write the Validation Helper

Add the following function near the top of `hci.c` (or in a shared HCI utility section), before `HCI_SendDataPkt`:

```c
/*
 * @brief  Validate that a BLE ACL connection handle is within the legal range.
 *
 * @param  connHandle  The connection handle to validate.
 *
 * @return HCI_SUCCESS              if the handle is valid.
 *         HCI_ERROR_CODE_INVALID_HCI_CMD_PARAMS  if the handle is out of range.
 */
static hciStatus_t HCI_ValidateConnHandle(uint16 connHandle)
{
    if (connHandle > HCI_MAX_CONN_HANDLE)
    {
        return HCI_ERROR_CODE_INVALID_HCI_CMD_PARAMS;
    }
    return HCI_SUCCESS;
}
```

Adjust the return type and error code symbols to match what the existing file already uses (e.g., `HCI_StatusCodes_t`, `bleStat_t`, or a plain `uint8`).

---

### Step 3: Call the Validator at the Top of HCI_SendDataPkt

Locate `HCI_SendDataPkt` in `hci.c` and add a guard immediately after the function signature and any existing NULL-pointer checks:

```c
hciStatus_t HCI_SendDataPkt(uint16        connHandle,
                             uint8         pbFlag,
                             uint8         bcFlag,
                             uint16        pktLen,
                             uint8        *pData)
{
    hciStatus_t status;

    /* Validate the connection handle before any further processing. */
    status = HCI_ValidateConnHandle(connHandle);
    if (status != HCI_SUCCESS)
    {
        return status;
    }

    /* ... existing implementation ... */
}
```

---

### Step 4: Confirm Error Code Availability

Verify that `HCI_ERROR_CODE_INVALID_HCI_CMD_PARAMS` (or its equivalent) is already defined in an included header (commonly `hci.h` or `bcomdef.h`). If a more specific error code exists for invalid handles (e.g., `LL_STATUS_ERROR_UNKNOWN_CONN_HANDLE`), prefer that for better diagnostics.

---

### Step 5: Add or Update a Unit / Integration Test

If the project has HCI-layer unit tests, add a test case that:
1. Calls `HCI_SendDataPkt` with handle `0x0F00` and confirms it returns the error code.
2. Calls `HCI_SendDataPkt` with handle `0x0EFF` (boundary) and confirms it proceeds normally (assuming a valid connection exists).
