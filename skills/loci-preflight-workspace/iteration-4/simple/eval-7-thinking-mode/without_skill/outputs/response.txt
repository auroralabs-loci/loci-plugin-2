Adding a watchdog reset inside OAD_processQueue when the queue has been stalled for more than 30 seconds requires careful thought across several dimensions: detecting the stall reliably, choosing the right watchdog mechanism for the CC26xx platform, avoiding false positives, and integrating cleanly with the existing OAD state machine.

---

## 1. Understanding the existing OAD_processQueue structure

OAD_processQueue on CC26xx (oad.c in the BLE5 flash profile) is typically called from the OAD task's main loop whenever the RTOS event queue signals a pending message. The function dequeues OAD events (image block requests, image identify, image notify, etc.) and dispatches them to handler routines. The function itself is not a blocking wait — it pulls one item from the queue, processes it, and returns. The outer task loop re-calls it the next time an event fires.

A "stalled queue" therefore means one of two distinct conditions:

- **No new events arriving** — the peer has stopped sending block requests or the connection has silently dropped, leaving the OAD in mid-transfer indefinitely.
- **An event is sitting in the queue but OAD_processQueue is never called** — the task is stuck somewhere else (e.g., blocked in flash write, waiting on a semaphore, or caught in an infinite loop upstream).

The 30-second watchdog makes most sense for the first condition. The second condition is better caught by a system-level watchdog that is never petted by a stuck task, which the TI BLE stack already supports via the hardware Watchdog Timer (WDT) module.

For the remainder of this analysis I focus on condition one: the OAD transfer is in progress but no forward progress is being made for 30 seconds.

---

## 2. Choosing the right mechanism

### Option A: Hardware Watchdog (WDT) — not ideal here

The CC26xx hardware WDT is a system-wide reset mechanism. If you tie it directly to OAD queue progress you risk resetting the device even when the BLE stack is otherwise healthy (e.g., a legitimate pause in the peer's sending). The hardware WDT is better used as a backstop for complete task hangs, not for application-layer transfer timeouts.

### Option B: RTOS software timer (preferred)

TI-RTOS / FreeRTOS both provide software timers. A one-shot timer set to 30 seconds, reset on every successful OAD block receipt, is the cleanest approach:

- Zero hardware resource cost beyond the timer tick.
- Cancellable / restartable when transfer resumes.
- Fires a callback that can post a synthetic "abort + reset" event to the OAD task, keeping all reset logic inside the OAD task context and avoiding ISR-context resets.

### Option C: Timestamp comparison inside OAD_processQueue

If a software timer is not available or is undesirable, you can record the tick count at the last successfully processed block and compare it on every call to OAD_processQueue:

```c
static uint32_t oadLastProgressTick = 0;
```

Check elapsed time at the top of OAD_processQueue; if the transfer is active and more than 30 seconds have passed since the last block, trigger the watchdog action.

---

## 3. Tracking "last progress" correctly

The key is defining what counts as progress. In OAD terms, progress means a new image block was received and written to flash successfully. In oad.c this happens inside the handler for `OAD_WRITE_BLOCK_CHAR` (or equivalent event type). You need to update the progress timestamp **only** at that point, not on every call to OAD_processQueue (which may be called for unrelated events).

A minimal addition:

```c
// Near the top of oad.c, file-scope:
static uint32_t oadLastBlockTick   = 0;   // Clock_getTicks() at last block receipt
static bool     oadTransferActive  = false;

// In the block-received handler path:
oadLastBlockTick  = Clock_getTicks();     // or xTaskGetTickCount() for FreeRTOS
oadTransferActive = true;

// Clear on transfer complete or abort:
oadTransferActive = false;
```

---

## 4. Implementing the stall check inside OAD_processQueue

At the top of OAD_processQueue (or just before returning, so it runs on every dispatch cycle):

```c
static void OAD_checkStallWatchdog(void)
{
    if (!oadTransferActive)
    {
        return; // No transfer in progress, nothing to watch
    }

    uint32_t nowTicks     = Clock_getTicks();
    uint32_t elapsedTicks = nowTicks - oadLastBlockTick;

    // Clock_tickPeriod is in microseconds; 30 s = 30,000,000 us
    uint32_t timeoutTicks = 30000000UL / Clock_tickPeriod;

    if (elapsedTicks >= timeoutTicks)
    {
        // Queue has been stalled for >= 30 seconds
        OAD_handleStallTimeout();
    }
}
```

Call `OAD_checkStallWatchdog()` at the start of `OAD_processQueue`, before dispatching the dequeued event.

Note: `Clock_tickPeriod` is a TI-RTOS global giving microseconds per tick (typically 10 µs on CC26xx, so 30 s = 3,000,000 ticks). Verify against your kernel configuration.

---

## 5. The stall handler: what to actually do

A hard `SysCtrlSystemReset()` (the CC26xx system reset) from inside the OAD task is possible but abrupt. A better sequence:

```c
static void OAD_handleStallTimeout(void)
{
    // 1. Mark transfer as aborted to prevent re-entry
    oadTransferActive = false;

    // 2. Optionally send an OAD status notification to the peer
    //    (may not be possible if the connection is already dead)
    OAD_sendStatus(OAD_TIMEOUT);  // if such a function exists

    // 3. Abort the OAD image (clear any partial flash image)
    OAD_imgBlocksWritten = 0;
    // ... any other state cleanup ...

    // 4. Reset the device
    //    On CC26xx with TI-RTOS, use SysCtrlSystemReset() from driverlib:
    SysCtrlSystemReset();
    // This call does not return.
}
```

If you want a softer recovery (just abort the OAD without resetting), omit the `SysCtrlSystemReset()` call and instead post a restart-OAD event so the peer can retry. A full system reset is appropriate only when you have reason to believe the device is in an unrecoverable state, which a simple transfer timeout usually is not.

---

## 6. Using a software timer instead of polling

The more robust alternative avoids the polling approach entirely:

```c
// File-scope timer handle
static Clock_Struct oadStallClockStruct;
static Clock_Handle oadStallClockHandle = NULL;

#define OAD_STALL_TIMEOUT_MS   30000
#define OAD_STALL_TIMEOUT_TICKS \
    (OAD_STALL_TIMEOUT_MS * 1000UL / Clock_tickPeriod)

static void OAD_stallClockCb(UArg arg)
{
    // Called in Clock (Swi) context — post an event to the OAD task
    Event_post(oadEventHandle, OAD_EVT_STALL_TIMEOUT);
}

// When OAD transfer starts:
Clock_construct(&oadStallClockStruct, OAD_stallClockCb,
                OAD_STALL_TIMEOUT_TICKS, NULL);
oadStallClockHandle = Clock_handle(&oadStallClockStruct);
Clock_start(oadStallClockHandle);

// On each successful block receipt — restart the timer:
Clock_stop(oadStallClockHandle);
Clock_start(oadStallClockHandle);

// On transfer complete or abort:
Clock_stop(oadStallClockHandle);
```

Then in OAD_processQueue, handle `OAD_EVT_STALL_TIMEOUT` the same way as the stall handler above. This approach is more accurate (the timer fires exactly 30 s after the last block with no polling overhead) and is better suited to an event-driven RTOS architecture.

---

## 7. Edge cases and pitfalls

- **Tick counter wraparound**: `uint32_t` arithmetic handles wraparound correctly as long as you use subtraction (`nowTicks - oadLastBlockTick`) rather than comparison of absolute values.
- **Clock_tickPeriod availability**: Confirm the symbol is exported in your kernel build; if not, hard-code the tick rate (e.g., `#define TICKS_PER_MS 100` for a 10 µs tick).
- **Re-entrancy**: OAD_processQueue runs in a single task context on TI-RTOS BLE stacks, so static variables are safe without a mutex, but document this assumption.
- **False timeouts during legitimate pauses**: Some OAD peer implementations pause between blocks (e.g., during peer-side flash erase). Make sure 30 seconds is generous enough for the slowest legitimate peer. A configurable timeout (`#define OAD_STALL_TIMEOUT_S 30`) is preferable to a hard-coded literal.
- **Reset during connection**: A `SysCtrlSystemReset()` during an active BLE connection will result in an immediate link-layer disconnect from the central's perspective. The central should handle this gracefully (connection supervision timeout). No special BLE teardown is required before the hardware reset.
- **NV / flash integrity**: If the reset fires mid-write, the partial OAD image slot must be invalidated on the next boot. This is typically handled by the OAD boot manager checking the image CRC/magic before handing control to the new image. Verify your boot manager does this correctly.

---

## 8. Summary of recommended changes to oad.c

1. Add `oadLastBlockTick`, `oadTransferActive` static variables.
2. Update `oadLastBlockTick` and set `oadTransferActive = true` in the block-received handler.
3. Clear `oadTransferActive` on transfer complete/abort.
4. Either:
   - (polling) Add `OAD_checkStallWatchdog()` at the top of `OAD_processQueue`, or
   - (preferred) Construct a `Clock` object when transfer starts, restart it on each block, and handle `OAD_EVT_STALL_TIMEOUT` in `OAD_processQueue`.
5. In the stall handler, clean up OAD state and call `SysCtrlSystemReset()` if a hard reset is truly required; otherwise abort the transfer and allow the peer to retry.
