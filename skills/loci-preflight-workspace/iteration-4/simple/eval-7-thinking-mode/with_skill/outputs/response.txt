## Preflight: OAD_processQueue (watchdog reset on 30-second stall)

### Call graph (no binary — static analysis only)

The source lives at `$LOCI_TEST_BLE_ROOT/simplelink-lowpower-f3-sdk/source/ti/ble5stack_flash/profiles/oad/cc26xx/oad.c`. No compiled object is available, so analysis proceeds from source-level reasoning about the TI BLE5 OAD profile.

Callees the stall-detection code would invoke:

- **ClockP_getSystemTickPeriod() / Clock_getTicks()** — TI-RTOS tick reads. These are short leaf calls or inlines; no CFG concern. The tick unit (µs per tick, commonly 10 µs or 1 ms) must be confirmed at init — using the wrong unit makes the 30-second threshold wrong by orders of magnitude.
- **SysCtrlSystemReset()** — noreturn peripheral write that triggers a hardware reset. No return value. Must not be called while holding a mutex, inside a `Hwi_disable()` critical section, or from interrupt context — any held lock is permanently abandoned and may corrupt retained-RAM state on reboot.
- **Queue_get() / mq_receive()** — OAD queue dequeue. If the stall check is evaluated on every call to `OAD_processQueue` and the queue is empty (no active OAD session), "stall" is indistinguishable from "idle". Resetting in that case incorrectly reboots a healthy device.

CFG concerns:

- **Dead path (RISK)**: `OAD_processQueue` is called by the application task only after an RTOS event or semaphore post signals that there is work to do. If the queue stalls because the remote peer silently drops out, the task pends indefinitely and `OAD_processQueue` is never re-entered. A stall check placed inside the function body therefore never executes. The watchdog must be implemented at or above the pend site — either as a TI-RTOS `Clock` callback running in Swi context, a dedicated watchdog task, or the hardware `Watchdog` peripheral — not as an in-body timestamp comparison.
- **Call-order assumption (RISK)**: the stall timer must arm only after an OAD session begins (e.g., after `OAD_open` or the first image-identify event). Arming at module init causes it to fire 30 seconds after boot regardless of OAD activity.
- **Idle vs. stall discrimination**: the timer must be reset on each successful dequeue, and the 30-second window must only be running while `oadState != OAD_IDLE`. An idle device with an empty queue is not stalled.

### Arithmetic ranges

1. **Elapsed time: `currentTick - stallStartTick`**
   - Both values are `uint32_t`. Unsigned subtraction wraps correctly modulo 2^32 across a single rollover — no guard needed for the subtraction itself.
   - Rollover budget: at 10 µs/tick, UINT32_MAX ticks ≈ 42 949 s ≈ 11.9 hours. At 1 ms/tick, ≈ 49.7 days. Both are far beyond the 30-second window; a single rollover is safe.
   - **RISK**: the threshold constant `stallTicks = 30 * TICKS_PER_SECOND` is only correct if `TICKS_PER_SECOND` matches the actual kernel tick period. If hardcoded, it drifts silently when the kernel is reconfigured. Derive it at runtime: `stallTicks = 30000000UL / ClockP_getSystemTickPeriod()` (where `getSystemTickPeriod` returns µs per tick), or add a `static_assert` that the assumed period matches the build-time constant.

2. **Threshold value bounds**
   - At 10 µs/tick: `stallTicks = 30 000 000 / 10 = 3 000 000`. Within `uint32_t`. Safe.
   - At 1 µs/tick: `stallTicks = 30 000 000`. Within `uint32_t`. Safe.
   - No overflow risk for any realistic TI-RTOS tick configuration.

3. **Signed/unsigned mix**: if either operand is cast to `int32_t` for the comparison (a common copy-paste mistake), a rollover-wrapped unsigned elapsed value can appear negative and suppress the reset. Keep all tick variables and comparisons strictly `uint32_t`.

### Freed-resource access

- **stallStartTick scope (RISK)**: `stallStartTick` must be a persistent module-level or `static` local variable. If declared on the stack of `OAD_processQueue`, it is re-initialized to zero on every call, making elapsed time always zero — the reset never fires.
- **Queue entry ownership**: `OAD_processQueue` dequeues a message pointer, processes it, and frees it. If the stall timer fires asynchronously (via a Clock callback) while a message has been dequeued but not yet freed, the allocation is leaked. On a hardware reset this is harmless, but on retained-RAM designs where the heap survives reset, the orphaned block is unrecoverable. Document this as a known artifact.
- **Critical section gap**: if `OAD_processQueue` disables interrupts or holds a mutex around queue access, the stall check must happen before that section is entered. Invoking `SysCtrlSystemReset()` inside a `Hwi_disable()` block is functionally correct (the chip resets regardless) but is architecturally sloppy and misleads static analysis tools.
- **No double-free from the watchdog path** — `SysCtrlSystemReset()` is noreturn; no subsequent pointer read or free occurs on this path.

### Adjusted plan (before writing any code)

1. **Choose the watchdog site.** The correct location for a wall-clock stall check is outside `OAD_processQueue` itself, because the function only runs when the task is scheduled. Two viable options:
   - **TI-RTOS Clock object** (software): create a `Clock` that fires after 30 s; its callback (runs in Swi context) calls `SysCtrlSystemReset()`. `OAD_processQueue` calls `Clock_stop` + `Clock_start` (restart) on each successful dequeue to reset the countdown.
   - **Hardware Watchdog peripheral** (`Watchdog_open` / `Watchdog_clear`): arm with a 30 s timeout in `OAD_open`; clear on each dequeue in `OAD_processQueue`; disarm (or set reload to max) in `OAD_close`.
   The Clock approach is simpler and avoids watchdog peripheral ownership conflicts. The hardware watchdog approach is more robust against OS scheduler hangs.

2. **Arm only when an OAD session is active.** Start the Clock (or arm the watchdog) in `OAD_open` or on receipt of the first OAD_IMAGE_IDENTIFY event. Stop/disarm in `OAD_close` and on OAD completion/abort. Never arm at module init.

3. **Derive tick threshold at init.** Call `ClockP_getSystemTickPeriod()` once and compute:
   ```c
   static uint32_t stallTicks;
   // in OAD_open or module init:
   stallTicks = 30000000UL / ClockP_getSystemTickPeriod();
   ```
   Pass `stallTicks` as the Clock timeout. This survives kernel reconfiguration without source changes.

4. **Keep all tick types `uint32_t`.** `stallStartTick`, `currentTick`, `elapsed`, and `stallTicks` must all be `uint32_t`. No signed casts; no intermediate `int` temporaries.

5. **Place the dequeue-side reset outside any critical section.** In `OAD_processQueue`, call `Clock_stop` + `Clock_start` (or `Watchdog_clear`) before any `Hwi_disable()` / mutex acquire. Do not nest it inside the critical section protecting queue state.

6. **Mark the reset path clearly.** In the Clock callback or watchdog expiry handler, add a comment `/* noreturn — hardware reset */` and optionally clear OAD state flags in retained RAM before calling `SysCtrlSystemReset()` so the next boot can detect a stall-induced reset and log it.

7. **Add a session-active guard in the Clock callback.** Before resetting, confirm `oadState != OAD_IDLE` to prevent the callback firing during normal (non-OAD) operation if the clock was not stopped cleanly on `OAD_close`.
