GAP_DeviceInit takes four parameters. Here's what each one does:

```c
bStatus_t GAP_DeviceInit(uint8_t profileRole,
                         uint8_t taskID,
                         GAP_Addr_Modes_t addrMode,
                         uint8_t* pRandomAddr);
```

---

1. profileRole (uint8_t)

This is a bitmask specifying what GAP role(s) the device will operate in. The valid values are:

- GAP_PROFILE_BROADCASTER (0x01) — sends advertising events only, no connections
- GAP_PROFILE_OBSERVER (0x02) — receives advertising events only, no connections
- GAP_PROFILE_PERIPHERAL (0x04) — accepts incoming connections (the "server" side)
- GAP_PROFILE_CENTRAL (0x08) — initiates connections (the "client" side)

Most applications pass a single role (e.g., GAP_PROFILE_PERIPHERAL for a sensor). Multi-role devices can combine them, but not all combinations are valid — passing an invalid combination returns INVALIDPARAMETER.

---

2. taskID (uint8_t)

This is the OSAL/ICall task ID of your application task. The BLE stack uses this to route GAP events back to your app. When initialization completes, the stack sends a GAP_DEVICE_INIT_DONE_EVENT to the task identified by this ID. In practice, you pass selfEntity (or equivalent) — the handle your task registered with ICall.

Internally, this value is stored in gapAppTaskID and gapEndAppTaskID. If GAP_BOND_MGR is compiled in, the bond manager may use a separate task ID internally, but the application task ID is what receives the final events.

---

3. addrMode (GAP_Addr_Modes_t)

This controls what kind of Bluetooth address the device will use for advertising and connecting. The options are:

- ADDRMODE_PUBLIC (0x00) — always use the device's hardwired public address
- ADDRMODE_RANDOM (0x01) — always use a Random Static Address (you supply this via pRandomAddr)
- ADDRMODE_RP_WITH_PUBLIC_ID (0x02) — use a Resolvable Private Address (RPA) that rotates periodically, with the public address as the identity address
- ADDRMODE_RP_WITH_RANDOM_ID (0x03) — use a Resolvable Private Address with a random static address as the identity address

RPAs (modes 0x02 and 0x03) are the privacy-preserving option — the device's on-air address changes over time so passive observers can't track it, but bonded peers can still resolve it using the IRK.

If you change addrMode from what was previously stored in NV, all bonds and local NV data are erased automatically.

---

4. pRandomAddr (uint8_t*)

A pointer to a 6-byte array containing the Random Static Address you want the device to use. This parameter is only meaningful when addrMode is ADDRMODE_RANDOM or ADDRMODE_RP_WITH_RANDOM_ID — in those cases the validation in GAP_DeviceInit_validate_params will return INVALIDPARAMETER if this pointer is NULL. For ADDRMODE_PUBLIC and ADDRMODE_RP_WITH_PUBLIC_ID, this parameter is ignored.

The address is passed to the controller via MAP_LL_SetRandomAddress() and the stack copies it internally, so you don't need to keep the buffer alive after the call returns.

---

Initialization flow overview

GAP_DeviceInit can only be called once per reset — it checks GAP_INITSTATE_READY and returns bleIncorrectMode if initialization has already occurred.

The call is asynchronous: the return value just tells you whether the request was accepted (SUCCESS), not that initialization is complete. Internally, it:

1. Validates parameters (role, address mode, random address pointer)
2. If GAP_BOND_MGR is enabled, reads the last address mode from NV and erases NV if it has changed
3. Stores taskID into gapAppTaskID / gapEndAppTaskID
4. Calls MAP_gap_ParamsInit() to set up device configuration
5. Configures the IRK: set to all zeros for public/random static modes, or loaded from NV (or randomly generated) for RPA modes
6. Configures the SRK: loaded from NV or randomly generated
7. Calls GAP_DeviceInit_per_role() to initialize security, privacy, advertising, and scanning modules appropriate to the role

When the stack finishes, it delivers a GAP_DEVICE_INIT_DONE_EVENT to your task (taskID). That event carries the device's resolved address, the LE data packet length, and the number of available data packets.

Typical call in a peripheral application:

```c
GAP_DeviceInit(GAP_PROFILE_PERIPHERAL, selfEntity, addrMode, &pRandomAddress);
```

where addrMode and pRandomAddress are typically defined in a board-level config header (e.g., ti_ble_config.h).

If you need to set a custom IRK or SRK, do so via GapConfig_SetParameter(GAP_CONFIG_PARAM_IRK, ...) and GapConfig_SetParameter(GAP_CONFIG_PARAM_SRK, ...) before calling GAP_DeviceInit — the function checks whether these have been set and only falls back to NV or random generation if they haven't.
