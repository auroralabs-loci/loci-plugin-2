## Plan: Add Exponential Backoff to Connection Parameter Update Retry Logic

### Context

Target file: `$LOCI_TEST_BLE_ROOT/examples/rtos/LP_EM_CC2340R53/ble5stack/basic_ble_profiles/app/app_connection.c`

The file is not present in the local working tree (`$LOCI_TEST_BLE_ROOT` is unset in this environment). This plan proceeds with static analysis only, informed by the standard TI BLE5Stack `basic_ble_profiles` app template for CC2340R53.

The task is to add exponential backoff to the retry logic that re-issues connection parameter update requests after a failed `BLEAPPUTIL_LINK_PARAM_UPDATE_EVENT`. Each retry should double the wait interval, capped at 10 seconds (10,000 ms).

## Preflight: Connection_paramUpdateRetryWithBackoff

(no binary — static analysis only)

### 1. Call graph

The retry path in `app_connection.c` (BLEAppUtil template) calls:
- `BLEAppUtil_paramUpdateReq(gapUpdateLinkParamReq_t *pReq)` — issued when the retry timer fires.
- TI-RTOS clock primitives — `Util_constructClock`, `Util_startClock`, `Util_stopClock`, `Util_isActive` — to schedule the delayed retry.
- `linkDB_GetInfo(connHandle, &info)` — to validate the connection is still live before re-issuing the request.

Call-order flags:
- `BLEAppUtil_paramUpdateReq` must only be called while the BLE stack is initialized and the connection handle is still valid. The new retry callback fires from the RTOS Clock module (posted event context, not an ISR), which is the same context as the existing handlers — no ISR-vs-task ordering hazard.
- The retry timer object (`Clock_Struct`) must be allocated in persistent, connection-scoped storage, not on the stack of any handler. If it is a local variable, the clock callback will fire against a destroyed stack frame — a dangling reference.
- Call ordering: the backoff interval must be clamped before being passed to `Util_constructClock` / `ClockP_setTimeout`, not after. Computing an overflowed intermediate and then clamping is still UB on a signed type (though the interval will be stored as `uint32_t`, where it wraps deterministically — still incorrect behavior).
- Indirect call: timer callback dispatch goes through a function-pointer registered at `Util_constructClock` time. The callback must be registered (address stable, function defined) before the clock is first armed. Static functions satisfy this with no extra step.

Call graph: OK

### 2. Arithmetic ranges

Three issues to flag before writing:

**Issue 1 — Pre-multiply overflow (WARNING):**
The doubling `interval *= 2` on a `uint32_t` overflows when `interval > UINT32_MAX / 2` (~2.15 billion ms). The cap of 10,000 ms is far below that, so under normal operation no overflow occurs. However, if the initial retry interval is misconfigured or the field is read from a corrupted or uninitialized connection record, the multiply may overflow before the cap clamp executes, producing a wrap-around value that bypasses the cap. Mitigation: use the compare-before-multiply pattern:

    if (intervalMs > APP_CONN_MAX_RETRY_MS / 2u) {
        intervalMs = APP_CONN_MAX_RETRY_MS;
    } else {
        intervalMs *= 2u;
    }

This never computes the overflowed value.

**Issue 2 — Signed/unsigned mix at timer call site (WARNING):**
TI-RTOS clock APIs (`ClockP_setTimeout`, `Util_constructClock`) take a tick count as `uint32_t`. The backoff interval tracked in ms is also `uint32_t`. The BLE connection interval parameters (`intervalMin`, `intervalMax` in `gapUpdateLinkParamReq_t`) are `uint16_t` in units of 1.25 ms. These are distinct values: the RTOS retry delay and the BLE connection interval are not the same quantity and must not be mixed. Verify that no implicit narrowing or unit confusion occurs at the point where the retry timer is armed.

**Issue 3 — Unit mismatch: ms vs. RTOS ticks (WARNING):**
TI-RTOS clocks on CC2340R53 run at a board-configured tick period (commonly 10 µs per tick). Passing `10000` as a raw tick count would produce a 0.1-second timer, not 10 seconds. The cap and the current interval must be kept in ms and converted to ticks explicitly at the timer call site:

    uint32_t ticks = (intervalMs * 1000u) / ClockP_getSystemTickPeriod();
    /* or the board's MS_TO_TICKS(ms) macro if available */

Define `APP_CONN_MAX_RETRY_MS 10000u` and `APP_CONN_INITIAL_RETRY_MS 500u` (or the existing base delay) as ms-domain constants, and convert only when calling the clock API.

Arithmetic: RISK — three issues; adjust plan before writing (see steps 2–3 below).

### 3. Freed-resource access

- **Persistent timer storage:** The `Clock_Struct` for the retry timer must live in connection-scoped persistent storage (e.g., a new field in the per-connection record array, `connectionConnList[i]`, or a parallel `retryState[i]` array). If the clock struct is a local in the event handler, the timer callback fires against a destroyed frame.
- **Stale connection handle:** Between when the retry timer is armed and when it fires, the BLE connection may terminate. The retry callback must call `linkDB_GetInfo(connHandle, &info)` before calling `BLEAppUtil_paramUpdateReq`, and abort if the connection is gone. Cancel the timer from the disconnect handler as well.
- **Double-arm race:** If a second parameter update failure arrives while a retry timer is already armed for the same connection, calling `Util_constructClock` or `Util_startClock` on an already-running `Clock_Struct` may reset it (acceptable) or may be a no-op or error depending on the TI-RTOS version. Guard with `Util_stopClock` before re-arming:

        Util_stopClock(&retryState[idx].retryClock);
        /* then set new timeout and start */

- **Stale backoff state on reconnect:** When a connection terminates, the per-connection retry record is freed/reused. If the backoff interval field is not reset to the initial value at disconnect time, the next connection on the same slot inherits a maxed-out backoff delay.

Resources: RISK — four issues; adjust plan before writing (see steps 1, 4–5 below).

**Execution fit: ADJUST PLAN**
Three classes of adjustment before writing:
1. Use compare-before-multiply overflow guard for the doubling operation.
2. Convert ms to RTOS ticks explicitly at the timer call site; do not pass ms as tick count.
3. Add stale-connection guard in the retry callback, double-arm guard when scheduling, and backoff-state reset in the disconnect handler.

## Adjusted Plan: Step-by-Step Edit Sequence

### Step 0 — Read the file before editing

Before any edit, read `app_connection.c` in full to identify:
- The exact name of the per-connection record struct and array (commonly `App_connInfo`, `connectionConnList`, or `connRec`).
- The existing retry mechanism, if any (clock handle name, arm/disarm call sites, the event that triggers a retry).
- The tick-conversion macro or formula in use in this project.
- The existing disconnect handler name and its cleanup logic.

Adjust all names below to match what is actually present.

### Step 1 — Add persistent retry state to the per-connection record

Add a new struct or extend the existing per-connection struct to hold:

```c
typedef struct {
    Clock_Struct  retryClock;
    uint32_t      retryIntervalMs;  /* current backoff delay in ms */
    uint8_t       retryCount;
    gapUpdateLinkParamReq_t lastReq; /* params to retry */
} ConnParamRetryState_t;

static ConnParamRetryState_t retryState[MAX_NUM_BLE_CONNS];
```

Initialize all entries in `Connection_start()` (or wherever `connectionConnList` is initialized):

```c
for (int i = 0; i < MAX_NUM_BLE_CONNS; i++) {
    retryState[i].retryIntervalMs = APP_CONN_INITIAL_RETRY_MS;
    retryState[i].retryCount = 0;
}
```

### Step 2 — Define ms-domain constants

Near the top of `app_connection.c`:

```c
#define APP_CONN_INITIAL_RETRY_MS   500u    /* first retry delay: 500 ms */
#define APP_CONN_MAX_RETRY_MS       10000u  /* cap: 10 seconds           */
#define APP_CONN_MAX_RETRY_ATTEMPTS 8u      /* safety ceiling             */
```

Do not use these values directly as tick counts. Convert at call sites.

### Step 3 — Replace fixed retry scheduling with overflow-safe backoff

In the `BLEAPPUTIL_LINK_PARAM_UPDATE_EVENT` handler, in the failure branch (where `pPkt->status != SUCCESS`), replace or add the retry arm:

```c
ConnParamRetryState_t *pRetry = Connection_getRetryState(pPkt->connectionHandle);
if (pRetry != NULL && pRetry->retryCount < APP_CONN_MAX_RETRY_ATTEMPTS)
{
    /* Overflow-safe doubling: compare before multiplying */
    if (pRetry->retryCount == 0) {
        pRetry->retryIntervalMs = APP_CONN_INITIAL_RETRY_MS;
    } else if (pRetry->retryIntervalMs > APP_CONN_MAX_RETRY_MS / 2u) {
        pRetry->retryIntervalMs = APP_CONN_MAX_RETRY_MS;
    } else {
        pRetry->retryIntervalMs *= 2u;
    }

    /* Double-arm guard */
    Util_stopClock(&pRetry->retryClock);

    /* Convert ms to ticks at the call site */
    uint32_t ticks = (pRetry->retryIntervalMs * 1000u) / ClockP_getSystemTickPeriod();

    Util_constructClock(&pRetry->retryClock,
                        Connection_paramUpdateRetryCallback,
                        ticks,
                        0,        /* one-shot */
                        TRUE,
                        (uint32_t)pPkt->connectionHandle);
    pRetry->retryCount++;
    /* Save the parameters to retry — pPkt does not carry them,
       so they must be sourced from the original request or connection record */
    /* pRetry->lastReq = ...; */
}
```

Note: `gapLinkUpdateEvent_t` (pPkt) carries `status` and `connectionHandle` but not the original requested interval values. The `lastReq` field must be populated either at connection establishment (from `gapEstLinkReqEvent_t`) or at the point where `BLEAppUtil_paramUpdateReq` is first called. Add a step to save the request at issuance.

### Step 4 — Implement the retry callback with stale-connection guard

```c
static void Connection_paramUpdateRetryCallback(uint32_t connHandleArg)
{
    uint16_t connHandle = (uint16_t)connHandleArg;

    /* Stale-connection guard: abort if link is gone */
    linkDBInfo_t linkInfo;
    if (linkDB_GetInfo(connHandle, &linkInfo) != SUCCESS) {
        Connection_cancelRetry(connHandle);
        return;
    }

    ConnParamRetryState_t *pRetry = Connection_getRetryState(connHandle);
    if (pRetry == NULL) return;

    BLEAppUtil_paramUpdateReq(&pRetry->lastReq);
    /* The outcome will arrive as another BLEAPPUTIL_LINK_PARAM_UPDATE_EVENT;
       the handler above will either stop retrying (on success) or arm the next retry. */
}
```

### Step 5 — Cancel retry and reset backoff on disconnect

In the `BLEAPPUTIL_LINK_TERMINATED_EVENT` handler, after removing the connection from `connectionConnList`, add:

```c
Connection_cancelRetry(gapTermMsg->connectionHandle);
```

Where `Connection_cancelRetry` is:

```c
static void Connection_cancelRetry(uint16_t connHandle)
{
    ConnParamRetryState_t *pRetry = Connection_getRetryState(connHandle);
    if (pRetry == NULL) return;
    Util_stopClock(&pRetry->retryClock);
    pRetry->retryCount = 0;
    pRetry->retryIntervalMs = APP_CONN_INITIAL_RETRY_MS;
}
```

This prevents stale backoff state from leaking into future connections on the same slot.

### Step 6 — Add helper: Connection_getRetryState

```c
static ConnParamRetryState_t *Connection_getRetryState(uint16_t connHandle)
{
    for (int i = 0; i < MAX_NUM_BLE_CONNS; i++) {
        if (connectionConnList[i].connHandle == connHandle) {
            return &retryState[i];
        }
    }
    return NULL;
}
```

### Step 7 — Add forward declarations

At the top of `app_connection.c`, before the function definitions:

```c
static void Connection_paramUpdateRetryCallback(uint32_t connHandleArg);
static void Connection_cancelRetry(uint16_t connHandle);
static ConnParamRetryState_t *Connection_getRetryState(uint16_t connHandle);
```
