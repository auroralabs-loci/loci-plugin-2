## Plan: Add Exponential Backoff to Connection Parameter Update Retry Logic

### Overview
Modify the existing retry logic for connection parameter updates in `app_connection.c` so that each successive retry waits twice as long as the previous one, starting from the current base interval and capping at 10 seconds (10,000 ms).

---

### Step 1: Add State Variables for Backoff Tracking

Add per-connection (or global, depending on existing structure) state variables to track:
- The current retry wait interval (`uint32_t retryInterval`)
- Optionally, a retry count for logging/debugging

These should be initialized when a connection parameter update attempt begins and reset on success or final failure.

```c
#define CONN_PARAM_UPDATE_RETRY_BASE_DELAY_MS   500     // or whatever current base is
#define CONN_PARAM_UPDATE_RETRY_MAX_DELAY_MS    10000   // 10-second cap
```

---

### Step 2: Locate the Existing Retry Scheduling Logic

Find where the retry timer is started (likely a call to `Util_startClock` or similar RTOS timer API with a fixed delay). This is the site that needs to be updated to use the dynamic backoff interval instead of a constant.

---

### Step 3: Implement the Backoff Calculation

At the point where a retry is scheduled (after a failed connection parameter update response), apply the doubling logic before starting the timer:

```c
// Double the interval, capped at max
retryInterval = MIN(retryInterval * 2, CONN_PARAM_UPDATE_RETRY_MAX_DELAY_MS);

// Schedule next retry with the new interval
Util_restartClock(&connParamUpdateClock, retryInterval);
```

---

### Step 4: Reset Backoff State on Success or Disconnect

When the connection parameter update succeeds or the connection drops, reset `retryInterval` back to the base value so the next connection starts fresh:

```c
retryInterval = CONN_PARAM_UPDATE_RETRY_BASE_DELAY_MS;
```

---

### Step 5: Verify Integration Points

- Confirm the clock/timer used for retries supports dynamic period updates (most TI-RTOS `Util_Clock` wrappers do via `Util_restartClock`).
- Ensure the backoff state is scoped correctly — if multiple simultaneous connections are supported, the state must be per-connection (e.g., stored in the connection record array).
- Check that the initial retry (first attempt after failure) uses the base delay before any doubling occurs.

---

### Summary of Changes
| Location | Change |
|---|---|
| Top of file / header | Add `#define` constants for base and max delay |
| Connection record struct (if multi-connection) | Add `retryInterval` field |
| Retry scheduling site | Replace fixed delay with `retryInterval`, then double it |
| Success / disconnect handler | Reset `retryInterval` to base value |

---

**Note:** This plan is based on common patterns in TI BLE5 SDK app_connection.c files. Before implementing, the actual file should be read to confirm the existing retry mechanism, timer API used, and whether connection state is stored per-connection or globally.
