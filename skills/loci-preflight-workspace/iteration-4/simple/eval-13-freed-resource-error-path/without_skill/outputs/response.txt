## Preflight Report

**Target file:** `$LOCI_TEST_BLE_ROOT/simplelink-lowpower-f3-sdk/source/ti/ble5stack_flash/profiles/oad/cc26xx/oad.c`
**Function:** `OAD_open`
**Change class:** Resource-cleanup refactor — adding free/close calls on an early-return error path

---

### Resources Check — Freed-Resource / Resource-Lifetime Concern

The requested change adds cleanup of two resources on the `GATTServApp_RegisterService` failure path:

1. `oadCCCDTable` — heap-allocated (presumably via `ICall_malloc` or equivalent) before the `GATTServApp_RegisterService` call.
2. `hOadQ` — a POSIX message queue opened (via `mq_open`) before the `GATTServApp_RegisterService` call.

**Risk: Double-free on repeated OAD_open calls.**

If `OAD_open` can be called a second time after a previous call failed, and the global variables `oadCCCDTable` and `hOadQ` are not reset to NULL / `(mqd_t)-1` after being freed/closed, the new cleanup code will re-free an already-freed pointer and re-close an already-closed descriptor on any subsequent failure:

- `ICall_free(oadCCCDTable)` on a stale (non-NULL but already freed) pointer is undefined behavior — heap corruption.
- `mq_close(hOadQ)` on an already-closed or invalid descriptor returns `EBADF` and may corrupt the POSIX MQ table.

**Mitigation required:** After freeing/closing each resource, set the variable to its null sentinel:

```c
ICall_free(oadCCCDTable);
oadCCCDTable = NULL;
mq_close(hOadQ);
hOadQ = (mqd_t)-1;   /* or INVALID_MQ_HANDLE, depending on the platform macro */
return (status);
```

**Risk: Incomplete coverage of other error paths.**

`GATTServApp_RegisterService` is likely not the only call that can fail after `oadCCCDTable` and `hOadQ` are acquired. Any other early return added later — or already present — between resource acquisition and the `GATTServApp_RegisterService` check must also release both resources. Confirm that:

- There are no existing early returns between `mq_open` / `ICall_malloc` and the `GATTServApp_RegisterService` call that already skip cleanup (this fix would not cover those paths).
- Any future error paths added to `OAD_open` after this change follow the same cleanup discipline.

**Risk: Ownership on the success path.**

On the normal (success) return path, `oadCCCDTable` and `hOadQ` are presumably retained as long-lived module state and released by `OAD_close` (or equivalent). The new cleanup must only fire on the error path — confirm the `if (status != SUCCESS)` guard is correct and that the success path does not also call the new cleanup.

## Plan

### 1. Read the current OAD_open implementation

Read `oad.c` to confirm:
- The exact variable names and types for `oadCCCDTable` and `hOadQ`.
- Where each resource is first acquired (the `ICall_malloc` and `mq_open` call sites).
- Whether either variable has a null-sentinel check at the top of `OAD_open` (indicating re-entrancy is already considered).
- The exact structure of the `GATTServApp_RegisterService` failure branch.
- Any other early-return paths between resource acquisition and the register call.

### 2. Locate the GATTServApp_RegisterService failure branch

Find the block:

```c
status = GATTServApp_RegisterService(...);
if (status != SUCCESS)
{
    return (status);   /* ← this is the leaking return */
}
```

### 3. Add cleanup before the early return

Replace the early return with:

```c
status = GATTServApp_RegisterService(...);
if (status != SUCCESS)
{
    ICall_free(oadCCCDTable);
    oadCCCDTable = NULL;
    mq_close(hOadQ);
    hOadQ = (mqd_t)-1;
    return (status);
}
```

The null-sentinel assignments are required to prevent double-free if `OAD_open` is retried.

### 4. Audit all other error paths in OAD_open

Search the function body for any other early returns that occur after the `mq_open` and `ICall_malloc` calls. Apply the same cleanup pattern to each, or introduce a single cleanup label at the bottom of the function (goto-style cleanup) to avoid duplicating the free/close logic across multiple paths.

### 5. Verify OAD_close does not double-free

Confirm that `OAD_close` (or whatever tears down module state) guards against freeing a NULL `oadCCCDTable` or closing an invalid `hOadQ`, so that a failed `OAD_open` followed by `OAD_close` does not crash.

### 6. Test

- Call `OAD_open` with a condition that causes `GATTServApp_RegisterService` to fail; confirm no heap corruption and no leaked MQ descriptor.
- Call `OAD_open` twice in a row with a persistent failure; confirm the second call does not double-free.
- Call `OAD_open` successfully; confirm `oadCCCDTable` and `hOadQ` are still live and functional.
