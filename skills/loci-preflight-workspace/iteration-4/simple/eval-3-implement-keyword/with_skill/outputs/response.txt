# Plan: Implement Retry Loop in HCI_SendDataPkt for Invalid Connection Handle

## Context

File: $LOCI_TEST_BLE_ROOT/simplelink-lowpower-f3-sdk/source/ti/ble5stack_flash/hci/cc26xx/hci.c
Function: HCI_SendDataPkt (lines 1323–1354)
Guard macro: #if defined(CTRL_CONFIG) && (CTRL_CONFIG & (ADV_CONN_CFG | INIT_CFG))

Current function body (lines 1327–1354):

```c
hciStatus_t HCI_SendDataPkt( uint16  connHandle,
                             uint8   pbFlag,
                             uint16  pktLen,
                             uint8  *pData )
{
  hciStatus_t hciStatus;

  if ( hciPTMenabled == TRUE )
  {
    hciStatus = HCI_ERROR_CODE_CONTROLLER_BUSY;
  }
  else if ( (pktLen == 0) || (pData == NULL) || ((connHandle & 0xFF00) != 0) )
  {
    hciStatus = HCI_ERROR_CODE_INVALID_HCI_CMD_PARAMS;
  }
  else // packet is okay
  {
    hciStatus = MAP_LL_TxData( connHandle, pData, pktLen, pbFlag );
  }

  return( hciStatus );
}
```

Key observations from reading source and headers:

1. MAP_LL_TxData maps to LL_TxData via the ROM jump table. Documented return values
   (ll.h:1392-1395): LL_STATUS_SUCCESS, LL_STATUS_ERROR_BAD_PARAMETER,
   LL_STATUS_ERROR_INACTIVE_CONNECTION, LL_STATUS_ERROR_OUT_OF_TX_MEM,
   LL_STATUS_ERROR_UNEXPECTED_PARAMETER.

2. The "invalid connection handle" error at the LL layer is
   LL_STATUS_ERROR_INACTIVE_CONNECTION (0x02), which aliases
   LL_STATUS_ERROR_UNKNOWN_CONN_HANDLE (0x02). The LL header comment states:
   "Unknown Connection Identifier for now; may be needed for multiple connections."
   (ll.h:151-152)

3. The existing guard `(connHandle & 0xFF00) != 0` rejects structurally malformed
   handles. Handles that are structurally valid but reference a connection that is
   not yet active or was torn down fall through to MAP_LL_TxData, which returns
   LL_STATUS_ERROR_INACTIVE_CONNECTION.

4. ll.h line 1368-1369 contains an explicit API contract warning:
   "The HCI should not call this routine if a buffer is still pending from a
    previous call. This is fatal!"
   And ll.h line 1363-1366 states:
   "If the return status is LL_STATUS_ERROR_OUT_OF_TX_MEM, then the HCI must not
    release the buffer until it receives the LL_TxDataCompleteCback callback."
   This means retrying after OUT_OF_TX_MEM while the LL still holds the buffer
   violates the API contract.

5. ll.h line 1371-1374: "If the connection should be terminated within the LL before
   the Host knows, attempts by the HCI to send more data (after receiving a
   LL_TxDataCompleteCback) will fail (LL_STATUS_ERROR_INACTIVE_CONNECTION)."
   On INACTIVE_CONNECTION, the LL does NOT retain pData — it is safe to retry.

6. hci.c includes no RTOS sleep/delay primitives (no Task_sleep, ICall_sleep,
   OSAL_sleep). A busy retry loop is the only option without rearchitecting the
   call site.

## Preflight: HCI_SendDataPkt (retry loop)

### 1. Call Graph (no binary — static analysis only)

MAP_LL_TxData is already declared and called in the existing body; no new forward
declarations are required. The retry loop adds additional calls to the same callee
under a guarded condition.

RISK: ll.h:1368-1369 states explicitly that calling LL_TxData while a buffer is
still pending from a previous call is "fatal." This constraint applies when
OUT_OF_TX_MEM is returned — the LL retains pData in that case. Retrying on any
error code other than INACTIVE_CONNECTION could trigger this fatal violation.

The retry must be gated strictly to (hciStatus == LL_STATUS_ERROR_INACTIVE_CONNECTION).
Retrying on OUT_OF_TX_MEM, BAD_PARAMETER, or UNEXPECTED_PARAMETER is unsafe and/or
pointless (deterministic failures that will not resolve on retry).

Call graph: ⚠ RISK — retrying MAP_LL_TxData on any error other than
            LL_STATUS_ERROR_INACTIVE_CONNECTION violates the LL "fatal" contract
            (ll.h:1368). Must gate retry strictly to that error code.

### 2. Arithmetic Ranges

The retry counter will be a local uint8 (or int) running 0 to 2, loop bound is 3.
No overflow is possible. connHandle is already validated to have zero high byte before
reaching the retry path. No new arithmetic expressions beyond the loop counter.

Arithmetic: OK

### 3. Freed-Resource Access

pData ownership per LL_TxData contract:
- LL_STATUS_SUCCESS: LL may retain pData; caller must not free until
  LL_TxDataCompleteCback fires.
- LL_STATUS_ERROR_OUT_OF_TX_MEM: LL retains pData; caller must not free until
  LL_TxDataCompleteCback fires.
- LL_STATUS_ERROR_INACTIVE_CONNECTION: LL does NOT retain pData; caller may reuse
  or free it freely.

Since the retry is gated to INACTIVE_CONNECTION only, pData is guaranteed not held
by the LL on each retry — passing it again is safe. No local resources are allocated
by the proposed change. No double-free or RAII gap.

Resources: OK (conditional on gating retries to INACTIVE_CONNECTION only)

## Adjusted Plan

### Step 1 — Map "invalid connection handle" to the correct LL error code

The task asks to retry "if the connection handle is invalid." At the LL boundary,
this maps to LL_STATUS_ERROR_INACTIVE_CONNECTION (0x02). The existing parameter
guard already rejects structurally malformed handles (high-byte non-zero) before
reaching MAP_LL_TxData. The retry is meaningful only for handles that are
structurally valid but reference a connection not yet active or torn down — exactly
the INACTIVE_CONNECTION return case.

### Step 2 — No delay between retries (design constraint noted)

No RTOS sleep primitive is available in this translation unit. The retry loop is a
tight spin — acceptable in a task context where the connection table update is
fast, but this is a best-effort mitigation, not a guaranteed recovery.

### Step 3 — The edit

Replace the `else` branch in HCI_SendDataPkt:

```c
  else // packet is okay
  {
    uint8 retries = 0;
    do
    {
      hciStatus = MAP_LL_TxData( connHandle, pData, pktLen, pbFlag );
      retries++;
    } while ( (hciStatus == LL_STATUS_ERROR_INACTIVE_CONNECTION) && (retries < 3) );
  }
```

This loop:
- Attempts MAP_LL_TxData up to 3 times total (retries starts at 0; increments after
  each call; loop exits when retries == 3 or when hciStatus is not
  LL_STATUS_ERROR_INACTIVE_CONNECTION, whichever comes first).
- Retries ONLY on LL_STATUS_ERROR_INACTIVE_CONNECTION — the sole return code where
  the LL does not retain pData, making re-calling safe per the API contract.
- Does NOT retry on LL_STATUS_ERROR_OUT_OF_TX_MEM (fatal if buffer still pending).
- Does NOT retry on LL_STATUS_ERROR_BAD_PARAMETER or UNEXPECTED_PARAMETER
  (deterministic; retry would not resolve them).
- Uses a uint8 counter; no overflow with bound 3.
- Allocates no resources; no new free/RAII obligations.

### Step 4 — Retry count semantics

"Retries up to 3 times" interpreted as 3 total call attempts:
- Attempt 1: retries becomes 1; if INACTIVE_CONNECTION, continue.
- Attempt 2: retries becomes 2; if INACTIVE_CONNECTION, continue.
- Attempt 3: retries becomes 3; loop exits regardless of result.
Final hciStatus reflects the last call's return value.

If the intent is 3 additional retries after the first attempt (4 total), change
the bound to `retries < 4`.

## Summary of Preflight Findings

The critical finding is the LL buffer-pending contract violation risk: ll.h:1368-1369
explicitly states that calling LL_TxData again while a buffer is pending from a
previous call is "fatal." The retry must be narrowly gated to
LL_STATUS_ERROR_INACTIVE_CONNECTION — the only return code where the LL does not
hold pData and a re-call is safe per the documented contract. No arithmetic hazards
or resource lifetime issues exist given that constraint.
