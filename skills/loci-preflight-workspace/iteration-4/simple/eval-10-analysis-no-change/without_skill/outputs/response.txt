Here is what I found reviewing OAD_open in oad.c (lines 424–512). I have not changed anything.

---

1. mq_open return value is never checked

   hOadQ = mq_open("/OADQueue", O_CREAT | O_NONBLOCK, 0, &attr);

   mq_open returns (mqd_t)-1 on failure. There is no check afterward. If the
   queue creation fails, hOadQ holds an invalid descriptor and every subsequent
   mq_send / mq_receive call (lines 637, 795, 1084, 1096, 2059) will silently
   fail or fault. The function continues past this point and returns OAD_SUCCESS
   as though nothing went wrong.

2. oadStateTimeout is set AFTER the clock is constructed with it

   Util_constructClock(&oadActivityClk, oadInactivityTimeout,
                       oadStateTimeout, 0, false, OAD_QUEUE_EVT);  // line 474-475

   if(oadTimeout > OAD_MIN_INACTIVITY_TIME)
   {
       oadStateTimeout = oadTimeout;   // line 509
   }

   The clock is built using the old default value of oadStateTimeout
   (OAD_DEFAULT_INACTIVITY_TIME = 10000 ms) regardless of what the caller
   passed in. Only after the clock is already constructed does the code update
   oadStateTimeout. The clock's duration is baked in at construction time, so
   the caller-supplied timeout has no effect on the first clock period. It does
   take effect later when Util_rescheduleClock is called (line 1126), but the
   initial duration is always the default. This is likely not what was intended.

3. No cleanup / rollback on GATTServApp_RegisterService failure

   If GATTServApp_RegisterService fails (line 484-487), status is set to
   OAD_NO_RESOURCES, but:
   - oadCCCDTable remains allocated (heap leak).
   - The message queue (hOadQ) has already been opened.
   - The clock (oadActivityClk) has already been constructed.

   The function then falls through the rest of the body (flash_init,
   hasExternalFlash, the timeout assignment) before returning the error.
   Callers who get OAD_NO_RESOURCES back have no way to know these resources
   were partially acquired, and calling OAD_open again would leak the previous
   allocation and attempt to open the same named queue a second time.

4. Queue name collision on repeated calls

   mq_open is called with the fixed name "/OADQueue". If OAD_open is somehow
   called more than once (or after a partial failure and retry), O_CREAT will
   succeed and return the already-existing queue — but the previous hOadQ handle
   is simply overwritten, leaking the old descriptor. The comment in the header
   says "Only call this function once," but there is no enforcement of that
   constraint.

5. Minor: mq_maxmsg is 32 with a comment "Maybe need to decrease"

   attr.mq_maxmsg = 32;  // line 469 — "Maybe need to decrease"

   This is a left-over TODO comment. A 32-message queue of sizeof(oadTargetWrite_t)
   entries may consume more heap than intended; it should be validated against
   the actual maximum number of in-flight OAD writes.

---

Summary of severity:

- Issue 1 (mq_open not checked): medium-high — silent failure path.
- Issue 2 (timeout ordering): low-medium — behavioral bug, wrong initial duration.
- Issue 3 (no cleanup on error): medium — resource leak on the failure path.
- Issue 4 (queue name reuse): low — only matters if the "call once" contract is violated.
- Issue 5 (TODO comment): low — code hygiene / sizing question.
