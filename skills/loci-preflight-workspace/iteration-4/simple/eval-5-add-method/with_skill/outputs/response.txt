                                GAP_Peer_Addr_Types_t addrType,

                                GAP_Peer_Addr_Types_t *pIdentityAddrType,
                                uint8_t *pIdentityAddr)

The function iterates over bonds[0..gapBond_maxBonds-1]. For each slot it:
  1. Compares bonds[idx].addr with pDevAddr via MAP_osal_memcmp (an indirect
     call through the ROM jump table).
  2. If no direct match and the address is a random-private-resolvable (RPR),
     reads the IRK from NV with osal_snv_read, reverses it via
     MAP_LL_ENC_ReverseBytes (indirect), and resolves via
     MAP_LL_PRIV_ResolveRPA (indirect).
  3. On a match, writes the slot index to *pIdx if pIdx != NULL, optionally
     writes identity address/type, and returns SUCCESS.
  4. If the loop exhausts without a match, returns bleGAPNotFound.

pIdx is already present in the existing signature as a nullable output
parameter. The bond index is therefore already surfaced by the existing API.

INTERPRETATION OF THE TASK
----------------------------------------------------------------------

Because the existing signature already carries pIdx, the meaningful
interpretation is to add a NEW narrower wrapper function —
GAPBondMgr_FindAddrWithIndex — that:
  - Makes the bond-index output non-nullable (mandatory),
  - Removes the optional identity-address outputs that callers wanting only
    the index do not need,
  - Delegates entirely to the proven existing function.

### 1. Call graph (CFG analysis)

The new wrapper will call GAPBondMgr_FindAddr, which is defined earlier
in the same translation unit (line 992). GAPBondMgr_FindAddr itself makes
three categories of calls:

  a. MAP_osal_memcmp — indirect call through TI ROM jump table (MAP_ prefix).
     Present in every iteration of the bond-search loop.
  b. osal_snv_read — NV read for IRK, executed only on the RPR resolution
     path.
  c. MAP_LL_ENC_ReverseBytes, MAP_LL_PRIV_ResolveRPA — indirect calls on the
     RPR path only.

Hazards:

  - Indirect calls via MAP_ table: all three indirect calls are normal
    ROM-dispatch pattern on TI BLE5. Not a CFI hazard introduced by this
    change; inherited from GAPBondMgr_FindAddr.
  - Call-order assumption: the bond manager must be initialized (NV init,
    GAPBondMgr_Register) before any find operation. gapBond_maxBonds is a
    module-level static initialized to GAP_BONDINGS_MAX; bonds[] is
    populated by NV reads at init. If the wrapper is called before
    initialization, osal_snv_read returns stale/uninitialized data and
    the written index would be garbage. The existing function makes this
    same assumption with no visible guard; the new wrapper inherits it.
    Flag: document the precondition in the function header comment.
  - No recursion. No cycles in the call graph. The new wrapper itself is
    a leaf with respect to memory ownership.
  - No dead paths: GAPBondMgr_FindAddr is defined before the new function
    in the same TU and declared in gapbondmgr.h which is already included.
    The SUCCESS and bleGAPNotFound paths are both reachable.

Call graph: ⚠ RISK — call-order assumption (bond manager NV init must
            precede call); indirect calls via MAP_ table (inherited, not new).

### 2. Arithmetic ranges

  - gapBond_maxBonds is declared static uint8_t, initialized to
    GAP_BONDINGS_MAX (typically 8, max 10 by TI documentation). The loop
    index idx is uint8_t — no overflow possible.
  - The written bond index fits in uint8_t: valid values are 0..7 (or
    0..9), well within [0, 255].
  - No new arithmetic introduced in the wrapper; all range checking is
    delegated to GAPBondMgr_FindAddr (NULL check on pDevAddr, range check
    on addrType vs. PEER_ADDRTYPE_RANDOM_OR_RANDOM_ID).
  - The one new arithmetic concern: pBondIndex is a required output
    parameter in the wrapper. If the caller passes NULL, a blind write
    *pBondIndex = idx inside the inner function (routed through pIdx) will
    be guarded by the existing "if (pIdx != NULL)" check — meaning NULL
    would be silently accepted by the inner function and no write would
    occur. The wrapper's contract says pBondIndex is mandatory, so the
    wrapper must add its own NULL guard and return INVALIDPARAMETER rather
    than silently accepting NULL and returning SUCCESS with no index written.
  - Output value on not-found: GAPBondMgr_FindAddr does not write *pIdx on
    the bleGAPNotFound path. The value at *pBondIndex would be whatever the
    caller placed there. Plan to write a sentinel (0xFF or GAP_BONDINGS_MAX)
    at entry to define the output on all paths.

Arithmetic: ⚠ RISK — NULL guard on pBondIndex required in wrapper (inner
            function silently ignores NULL pIdx); output undefined on
            not-found path without a sentinel written at entry.

### 3. Freed-resource access

  - GAPBondMgr_FindAddr reads bond records from module-level statics
    (bonds[] array) and from NV into a stack-local buffer (tempIRK[KEYLEN]).
    No heap allocation in the function or its callees on the search path.
  - No pointer is freed, moved, or stored past the frame lifetime.
  - The new wrapper acquires no resources. It passes through the pDevAddr
    and addrType arguments it receives; it owns no memory.
  - The one lifetime note: pBondIndex is a caller-owned output pointer. The
    wrapper must not write through it after returning an error — the existing
    inner function's guard ("if pIdx != NULL write *pIdx then return SUCCESS")
    ensures the write only happens on SUCCESS. The sentinel approach
    (write GAP_BONDINGS_MAX at entry, overwrite with real index on SUCCESS)
    keeps the output always defined without creating a use-after-error write.

Resources: OK — no heap, no free paths, no dangling references.

  Search gapbondmgr.c and the rest of the stack for all callers of
  GAPBondMgr_FindAddr. The new function is a wrapper and does not change

Step 1 — Header declaration (gapbondmgr.h)
  Add immediately after the existing GAPBondMgr_FindAddr declaration:

    /**
     * @brief Find a bond by address and return the bond index.
     *
     * Wrapper around GAPBondMgr_FindAddr that makes the bond index
     * output mandatory. The bond manager must be initialized before
     * calling this function; calling it before NV initialization
     * produces undefined behavior (precondition, not enforced here).
     *
     * @param pDevAddr     Pointer to the device address to search for.
     *                     Must not be NULL.
     * @param addrType     Address type of pDevAddr.
     * @param pBondIndex   Output: bond slot index on SUCCESS. Set to
     *                     GAP_BONDINGS_MAX (sentinel) if address is not
     *                     found. Must not be NULL.
     *
     * @return SUCCESS          Address found; *pBondIndex is valid.
     * @return INVALIDPARAMETER pDevAddr or pBondIndex is NULL.
     * @return bleInvalidRange  addrType out of range.
     * @return bleGAPNotFound   Address not found; *pBondIndex == GAP_BONDINGS_MAX.
     */
    extern bStatus_t GAPBondMgr_FindAddrWithIndex(uint8_t *pDevAddr,
                                                  GAP_Peer_Addr_Types_t addrType,
                                                  uint8_t *pBondIndex);

Step 2 — Implementation (gapbondmgr.c)
  Place immediately after GAPBondMgr_FindAddr (after line 1063):

    /*********************************************************************
     * @brief   Find a bond by address, returning the bond index as a
     *          required output parameter.
     *
     * Precondition: GAPBondMgr must be initialized (NV init complete)
     * before calling. No runtime guard is enforced.
     *
     * Public function defined in gapbondmgr.h.
     */
    bStatus_t GAPBondMgr_FindAddrWithIndex(uint8_t *pDevAddr,
                                           GAP_Peer_Addr_Types_t addrType,
                                           uint8_t *pBondIndex)
    {
      // Preflight finding: pBondIndex is mandatory in this wrapper.
      // GAPBondMgr_FindAddr silently accepts NULL pIdx — guard here.
      if (pBondIndex == NULL)
      {
        return (INVALIDPARAMETER);
      }

      // Preflight finding: define output on all paths, including not-found.
      // Sentinel value = GAP_BONDINGS_MAX (one past last valid index).
      *pBondIndex = GAP_BONDINGS_MAX;

      // Delegate to existing validated implementation.
      // pBondIndex serves as pIdx; identity outputs not needed by this API.
      return GAPBondMgr_FindAddr(pDevAddr, addrType,
                                 pBondIndex, NULL, NULL);
    }

Key design decisions driven by preflight:

  NULL guard on pBondIndex (Arithmetic finding):
    GAPBondMgr_FindAddr checks "if (pIdx != NULL)" before each write,
    so it would silently accept NULL and return SUCCESS with nothing written.
    The wrapper's contract is that the index is always provided; a NULL
    caller gets INVALIDPARAMETER instead of a silent no-op.

  Sentinel at entry (Arithmetic finding):
    *pBondIndex = GAP_BONDINGS_MAX before the inner call ensures the output
    is defined even when bleGAPNotFound is returned. Callers can check
    *pBondIndex == GAP_BONDINGS_MAX as a secondary signal without relying
    solely on the return value.

  Initialization precondition comment (Call graph finding):
    No runtime guard is added — the existing GAPBondMgr_FindAddr does not
    enforce one either, and adding one here would require access to internal
    init-state tracking that is not part of the public API. The comment
    documents the precondition explicitly so callers understand the contract.

  No redundant validation of pDevAddr or addrType:
    GAPBondMgr_FindAddr already returns INVALIDPARAMETER for NULL pDevAddr
    and bleInvalidRange for out-of-range addrType. Re-checking here would
    duplicate logic that is already tested and maintained in the inner
    function. The wrapper propagates those return values unchanged.

  pIdentityAddrType and pIdentityAddr passed as NULL:
    The contract of this wrapper is specifically "find by address and return
    index." Callers who also need the resolved identity address must use
    the full GAPBondMgr_FindAddr API directly.

No forward declarations needed: GAPBondMgr_FindAddr is defined before the
new function in the same file, and gapbondmgr.h (which declares both) is
already #included.

No new arithmetic. No resource ownership changes. No call-order assumptions
beyond what the inner function already carries.
