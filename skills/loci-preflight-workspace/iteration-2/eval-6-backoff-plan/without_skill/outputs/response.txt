## Plan: Add Exponential Backoff to Connection Parameter Update Retry Logic in app_connection.c

**Note:** `app_connection.c` does not exist in this repository. The plan below describes exactly what to do once the file is located (or created).

### Step 1 — Read the existing retry logic

Read `app_connection.c` in full to identify:
- The function that retries connection parameter updates (likely `conn_param_update_retry`, `update_conn_params`, or similar).
- The current wait mechanism between retries (fixed constant, `os_delay_ms`, RTOS timer, etc.).
- The retry counter and max-retry limit.

### Step 2 — Identify variables to add or change

- Add a local `uint32_t retry_interval_ms` initialized to the existing fixed delay value.
- Add constants `CONN_PARAM_RETRY_INITIAL_MS` and `CONN_PARAM_RETRY_MAX_MS` (10000).

### Step 3 — Edit the retry loop

Replace the fixed-delay call with the doubling variable, clamped at 10000 ms:

```c
uint32_t retry_interval_ms = CONN_PARAM_RETRY_INITIAL_MS;
for (int attempt = 0; attempt < MAX_CONN_PARAM_RETRIES; attempt++) {
    if (conn_param_update(handle, &params) == 0) return 0;
    os_delay_ms(retry_interval_ms);
    retry_interval_ms *= 2;
    if (retry_interval_ms > CONN_PARAM_RETRY_MAX_MS)
        retry_interval_ms = CONN_PARAM_RETRY_MAX_MS;
}
```

### Step 4 — Verify no other callers are affected

Search for other sites that depend on the old fixed-delay behavior and confirm they are unaffected.

### Step 5 — Update header constants

Move the new constants into the shared header and remove the old fixed-delay constant to keep the codebase consistent.
