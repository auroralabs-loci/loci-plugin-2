**How GAP_DeviceInit validates its addrMode parameter**

All addrMode validation is performed inside `GAP_DeviceInit_validate_params()` (gap.c, lines ~107–221), called at the top of both `GAP_DeviceInit()` and `GAP_DeviceInit_noGAPBondMgr()`.

**Step 1 — Already-initialized guard**

If `MAP_gapGetState() == GAP_INITSTATE_READY`, the function returns `bleIncorrectMode` immediately — addrMode is irrelevant.

**Step 2 — Range check**

The `GAP_Addr_Modes_t` enum has four legal values (0x00–0x03). Any value `> ADDRMODE_RP_WITH_RANDOM_ID` (0x03) returns `INVALIDPARAMETER`. Also: if addrMode is `ADDRMODE_RANDOM` (0x01) or `ADDRMODE_RP_WITH_RANDOM_ID` (0x03) and `pRandomAddr == NULL`, that also returns `INVALIDPARAMETER`.

**Step 3 — Random Static Address format check**

Only applies when addrMode is `ADDRMODE_RANDOM` or `ADDRMODE_RP_WITH_RANDOM_ID`. The supplied address is rejected if: (a) lower 46 bits are all zero, (b) all 48 bits are 1 (0xFF:FF:FF:FF:FF:FF), or (c) the top 2 bits of byte[5] are not `0b11` — checked via `GAP_IS_ADDR_RS()` using `RANDOM_ADDR_HDR_MASK = 0xC0` and `STATIC_ADDR_HDR = 0xC0`. `ADDRMODE_PUBLIC` and `ADDRMODE_RP_WITH_PUBLIC_ID` skip this check entirely.

- `gap.c` lines 107–221: validate_params
- `gap.h`: GAP_Addr_Modes_t enum (~line 758), GAP_IS_ADDR_RS macro (~line 133), RANDOM_ADDR_HDR_MASK/STATIC_ADDR_HDR constants (~line 804)
