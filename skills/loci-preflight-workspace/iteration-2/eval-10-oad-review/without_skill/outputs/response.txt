Review of OAD_open in oad.c (lines 424–512) — no changes made:

**1. `oadCCCDTableSize` declared as `uint8_t` — potential silent overflow**

```c
uint8_t oadCCCDTableSize = (sizeof(gattCharCfg_t) * linkDBNumConns) * OAD_CCCD_CNT;
```

`sizeof(gattCharCfg_t)` is typically 4 bytes, `linkDBNumConns` can be up to 8, `OAD_CCCD_CNT` is 3. That's 96 bytes today — fine — but any increase silently overflows and causes `ICall_malloc` to allocate a far-too-small buffer. Should be `size_t` or at least `uint16_t`.

**2. `mq_open` return value is never checked**

`hOadQ = mq_open("/OADQueue", O_CREAT | O_NONBLOCK, 0, &attr)` is not validated against `(mqd_t)-1`. If the queue creation fails, `hOadQ` holds an invalid descriptor and all subsequent enqueue operations silently fail.

**3. Clock constructed with stale `oadStateTimeout`**

The clock is built before `oadStateTimeout` is updated with the caller's value. The initial clock period always uses the default `OAD_DEFAULT_INACTIVITY_TIME` regardless of the `oadTimeout` the caller passes.

**4. No cleanup on GATT registration failure — heap leak and partial init**

When `GATTServApp_RegisterService` fails, the code sets `status = OAD_NO_RESOURCES` but continues executing `flash_init()` and `oadCreateFactoryImageBackup()` before returning. `oadCCCDTable` is allocated but never freed. Every failed open attempt leaks memory.

**5. No re-entry guard**

Nothing prevents `OAD_open` from being called twice. A second call overwrites `oadCCCDTable` without freeing the old allocation, re-constructs the clock on top of a running one, and re-opens the queue.

**6. `OAD_close` never calls `mq_close` or `mq_unlink`**

`OAD_close` calls `oadResetState()` and `flash_close()` but never closes or unlinks `/OADQueue`. Every open/close cycle leaks a queue descriptor.
