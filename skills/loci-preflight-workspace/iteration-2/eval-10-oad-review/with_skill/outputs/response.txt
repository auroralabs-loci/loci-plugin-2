Review of OAD_open (read-only analysis — no changes made):

6 issues identified:

1. **No double-open guard (Medium)** — The comment says "Only call this function once" but nothing enforces it. A second call leaks the previous `oadCCCDTable` allocation (no free before reassignment), re-constructs the RTOS clock in-place (can corrupt the timer list), and re-registers the GATT service.

2. **`uint8_t` size variable — latent buffer overflow (High)** — `oadCCCDTableSize` is declared `uint8_t`. If `sizeof(gattCharCfg_t) * linkDBNumConns * OAD_CCCD_CNT` exceeds 255 (possible as struct size or connection count grows), the value silently wraps, `ICall_malloc` allocates too little, and the pointer arithmetic into `oadCCCDTable` writes past the end of the allocation.

3. **`mq_open` return value never checked (Medium)** — `hOadQ = mq_open(...)` is not validated against `(mqd_t)-1`. If queue creation fails, all subsequent enqueue operations use an invalid descriptor.

4. **Timeout guard uses `>` not `>=` (Low)** — A caller passing exactly `OAD_MIN_INACTIVITY_TIME` is silently ignored; the default timeout is used instead. Should be `>=`.

5. **No resource cleanup on GATT registration failure (Medium)** — When `GATTServApp_RegisterService` fails, the function sets `status = OAD_NO_RESOURCES` but does not free `oadCCCDTable`, close `hOadQ`, or destruct the clock. It also continues executing `flash_init` and the factory image path.

6. **`oadStateTimeout` written after clock construction (Low)** — The clock is constructed using the current (default) value of `oadStateTimeout`. The caller-supplied `oadTimeout` is only written afterward, so the first timeout interval always uses the default regardless of what the caller passed.

No changes made. Let me know which issues you want to address first.
