**Plan: Validate BLE Connection Handle Before HCI_SendDataPkt**

**Background:** BLE connection handles are 12-bit values. The Bluetooth Core Specification defines valid handles as `0x0000`–`0x0EFF`; values `0x0F00`–`0x0FFF` are reserved. Passing an out-of-range handle to `HCI_SendDataPkt` risks undefined controller behavior.

**Constants:**
```c
#define BLE_CONN_HANDLE_MIN    0x0000
#define BLE_CONN_HANDLE_MAX    0x0EFF
#define BLE_ERR_INVALID_HANDLE (-1)
```

**Validation wrapper:**
```c
int ValidateAndSendDataPkt(uint16_t connHandle, uint8_t *pData, uint16_t dataLen)
{
    if (connHandle > BLE_CONN_HANDLE_MAX)
        return BLE_ERR_INVALID_HANDLE;

    return HCI_SendDataPkt(connHandle, pData, dataLen);
}
```

**Key decisions:** Use `uint16_t` to avoid signed comparisons; only upper-bound check is needed. Return a named constant so callers distinguish this from HCI errors. Never invoke `HCI_SendDataPkt` when the handle is invalid.

**Integration:** Replace all direct `HCI_SendDataPkt` call sites with `ValidateAndSendDataPkt` and handle `BLE_ERR_INVALID_HANDLE` in callers.

**Tests:** Boundary values `0x0000`, `0x0EFF` (valid), `0x0F00`, `0xFFFF` (invalid).
