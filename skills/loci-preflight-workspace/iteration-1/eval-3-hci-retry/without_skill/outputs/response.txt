## Analysis of HCI_SendDataPkt

### Current Implementation (lines 1323-1354 in hci.c)

The function `HCI_SendDataPkt` in
`/home/melisa/BLE/simplelink-lowpower-f3-sdk/source/ti/ble5stack_flash/hci/cc26xx/hci.c`
currently has three branches:

1. If PTM mode is enabled -> return `HCI_ERROR_CODE_CONTROLLER_BUSY`
2. If packet length is zero, data pointer is NULL, or the upper byte of
   `connHandle` is non-zero -> return `HCI_ERROR_CODE_INVALID_HCI_CMD_PARAMS`
3. Otherwise -> call `MAP_LL_TxData(connHandle, pData, pktLen, pbFlag)` and
   return whatever status it returns

`MAP_LL_TxData` maps to `LL_TxData` (via `map_direct.h`). The Link Layer
returns `LL_STATUS_ERROR_UNKNOWN_CONN_HANDLE` (value `0x02`, defined in
`controller/cc26xx/inc/ll.h`) when the given connection handle does not
correspond to a known, active connection. This value is identical to
`HCI_ERROR_CODE_UNKNOWN_CONN_ID` (value `0x02`, defined in `inc/hci.h`).

### Plan: Add a retry loop for invalid connection handle

#### Where to make the change

Inside the `else` branch (line 1344-1351) that currently makes a single call
to `MAP_LL_TxData`. The surrounding early-exit checks (PTM mode, bad
parameters) should remain unchanged because those are definitive error
conditions that do not benefit from retrying.

#### What "invalid connection handle" means at runtime

A connection handle can temporarily appear invalid at the HCI layer if the
handle has been allocated by the upper stack but the Link Layer's connection
table entry has not yet been finalized (a transient race). Retrying up to 3
times gives the LL time to settle without changing the API contract.

#### Step-by-step change

Replace the current single-call `else` block:

```c
  else // packet is okay
  {
    hciStatus = MAP_LL_TxData( connHandle, pData, pktLen, pbFlag );
  }
```

With a retry loop:

```c
  else // packet is okay
  {
    uint8 retryCount;

    hciStatus = HCI_ERROR_CODE_UNKNOWN_CONN_ID; // initialise to trigger loop

    for ( retryCount = 0; retryCount < 3; retryCount++ )
    {
      // attempt to send the packet
      // Note: A return of HCI_SUCCESS from this routine indicates that either
      //       the data was transmitted and freed, or it is still in use
      //       (i.e. queued).
      hciStatus = MAP_LL_TxData( connHandle, pData, pktLen, pbFlag );

      // If the connection handle was not recognised by the Link Layer, retry.
      // Any other status (success or a different error) exits immediately.
      if ( hciStatus != HCI_ERROR_CODE_UNKNOWN_CONN_ID )
      {
        break;
      }
    }
    // hciStatus now reflects the outcome of the last attempt.
  }
```

#### Key design decisions

- The retry condition checks for `HCI_ERROR_CODE_UNKNOWN_CONN_ID` (0x02),
  which is the value returned by `MAP_LL_TxData` /`LL_TxData` when the
  connection handle is not found in the LL connection table
  (`LL_STATUS_ERROR_UNKNOWN_CONN_HANDLE` has the same numeric value).
- All other error codes (e.g., `HCI_ERROR_CODE_MEM_CAP_EXCEEDED`,
  `HCI_ERROR_CODE_CONTROLLER_BUSY`) cause an immediate exit from the loop
  because those are not transient unknown-handle conditions.
- `HCI_SUCCESS` also breaks immediately; there is no need to retry a
  successful send.
- The loop variable `retryCount` is declared as `uint8` to match the type
  conventions already used in the file for small counters.
- No sleep/delay is inserted between retries; the assumption is that the
  condition is resolved within the same scheduling slice. If a delay were
  needed it would require a RTOS-aware mechanism (e.g., `Task_sleep`) that
  is outside the scope of this function.
- After all 3 attempts, if the handle is still unknown, `hciStatus` will
  hold `HCI_ERROR_CODE_UNKNOWN_CONN_ID` and the function returns that error
  to the caller (L2CAP), which is a meaningful, standard BT error code.

#### No other changes required

- The function signature, preprocessor guards (`#if defined(CTRL_CONFIG)...`),
  and all other branches remain untouched.
- No new headers need to be included; `HCI_ERROR_CODE_UNKNOWN_CONN_ID` is
  already reachable via the existing include chain that provides `hciStatus_t`
  and the other `HCI_ERROR_CODE_*` constants used in this file.
