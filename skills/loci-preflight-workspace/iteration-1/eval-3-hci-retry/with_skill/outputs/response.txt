# Plan: Add Retry Loop to HCI_SendDataPkt for Invalid Connection Handle

## Context

File: /home/melisa/BLE/simplelink-lowpower-f3-sdk/source/ti/ble5stack_flash/hci/cc26xx/hci.c
Function: HCI_SendDataPkt (lines 1323–1354)
Guard macro: #if defined(CTRL_CONFIG) && (CTRL_CONFIG & (ADV_CONN_CFG | INIT_CFG))

The current function body:

```c
hciStatus_t HCI_SendDataPkt( uint16  connHandle,
                             uint8   pbFlag,
                             uint16  pktLen,
                             uint8  *pData )
{
  hciStatus_t hciStatus;

  // various checks
  if ( hciPTMenabled == TRUE )
  {
    hciStatus = HCI_ERROR_CODE_CONTROLLER_BUSY;
  }
  else if ( (pktLen == 0) || (pData == NULL) || ((connHandle & 0xFF00) != 0) )
  {
    hciStatus = HCI_ERROR_CODE_INVALID_HCI_CMD_PARAMS;
  }
  else // packet is okay
  {
    hciStatus = MAP_LL_TxData( connHandle, pData, pktLen, pbFlag );
  }

  return( hciStatus );
}
```

Key observations from reading the source and headers:

1. MAP_LL_TxData maps to LL_TxData (via rom/map_direct.h or the ROM jump table). Its documented
   return values are: LL_STATUS_SUCCESS, LL_STATUS_ERROR_BAD_PARAMETER,
   LL_STATUS_ERROR_INACTIVE_CONNECTION, LL_STATUS_ERROR_OUT_OF_TX_MEM,
   LL_STATUS_ERROR_UNEXPECTED_PARAMETER.

2. The "invalid connection handle" error returned by LL_TxData is
   LL_STATUS_ERROR_INACTIVE_CONNECTION (0x02), which is defined as the same value as
   LL_STATUS_ERROR_UNKNOWN_CONN_HANDLE (0x02). The LL header comment says:
   "Unknown Connection Identifier for now; may be needed for multiple connections."

3. The existing upper-layer guard `(connHandle & 0xFF00) != 0` only catches handles whose
   high byte is non-zero. It does NOT catch a valid-format handle that references a
   connection that is temporarily not yet active or is mid-setup — that case falls through
   to MAP_LL_TxData and returns LL_STATUS_ERROR_INACTIVE_CONNECTION.

4. There is no sleep/delay primitive in this TU (no Task_sleep, ICall_sleep, or OSAL_sleep
   in hci.c). The file includes icall.h, bcomdef.h, and the ROM jump table but no RTOS
   primitives at this layer.

5. hciStatus_t is an 8-bit status type (uint8); it is used identically to llStatus_t at
   this call site.

## Preflight: HCI_SendDataPkt (retry loop)

### 1. Call Graph Ordering (CFI)

The retry loop will call MAP_LL_TxData up to 3 additional times (4 total including the
first attempt). MAP_LL_TxData is already declared and called in the existing body — no
new forward declarations are needed.

RISK: The LL spec note in ll.h says explicitly:
  "The HCI should not call this routine if a buffer is still pending from a previous
   call. This is fatal!"
  (ll.h line 1368-1369)

This is a direct call-ordering violation. If LL_TxData returns LL_STATUS_ERROR_OUT_OF_TX_MEM
(not the invalid-connection case), the buffer is retained by the LL. Calling LL_TxData again
with the same buffer on the next retry iteration while the LL still holds it is explicitly
flagged as "fatal" by the API contract. The retry loop must be strictly scoped to only the
inactive-connection error code and must NOT retry on LL_STATUS_ERROR_OUT_OF_TX_MEM.

Secondary CFI note: MAP_LL_TxData resolves through a ROM jump table at runtime. There is no
recursion risk. No init-order issue — hciPTMenabled is checked first and this is an
interrupt/task context function, not a static initializer.

Call graph: ⚠ RISK — retrying on wrong error codes could violate the LL "fatal" contract.
The plan must gate retries strictly to LL_STATUS_ERROR_INACTIVE_CONNECTION only.

### 2. Arithmetic Ranges

The retry counter will be a local `uint8_t` (or plain `int`) running from 0 to 2 (3
iterations). No overflow is possible with a bound of 3.

The connHandle parameter is already validated to have `(connHandle & 0xFF00) == 0` before
reaching the retry path, so it fits in 8 bits. No signed/unsigned hazard here — connHandle
is uint16 throughout.

No new arithmetic expressions are introduced by a simple counter loop.

Arithmetic: OK

### 3. Freed-Resource Access

pData is passed in by the caller. Ownership rules from the LL_TxData API:
- On LL_STATUS_SUCCESS: the LL may retain pData; the HCI must NOT free it until the
  LL_TxDataCompleteCback fires.
- On LL_STATUS_ERROR_OUT_OF_TX_MEM: the LL retains pData (caller must wait for callback).
- On LL_STATUS_ERROR_INACTIVE_CONNECTION: the LL does NOT retain pData; the caller may
  reuse or free it.

Retry scenario: if LL_TxData returns LL_STATUS_ERROR_INACTIVE_CONNECTION, pData has NOT
been retained by the LL. It is safe to pass pData again on the next retry call.

No local resources are allocated by the proposed change. No RAII gap. No double-free risk
as long as the retry is gated to INACTIVE_CONNECTION returns only.

Resources: OK (conditional on gating retries to INACTIVE_CONNECTION only)

## Adjusted Plan

### Step 1 — Understand what "invalid connection handle" means at the LL boundary

The task asks to retry "if the connection handle is invalid." At the LL layer this
maps to LL_STATUS_ERROR_INACTIVE_CONNECTION (0x02). The existing parameter-level guard
already rejects structurally malformed handles (high-byte non-zero) before reaching
MAP_LL_TxData. The retry is therefore meaningful only for handles that are structurally
valid but reference a connection that is not yet active or has been torn down between
layers — exactly the INACTIVE_CONNECTION return case.

### Step 2 — No delay between retries (design constraint)

There is no RTOS sleep primitive available in this TU. A busy retry loop (up to 3 times)
is the only option without rearchitecting the call site. This is appropriate only if the
condition can resolve rapidly (e.g., a brief race during connection setup). Three retries
with no sleep is a tight spin — acceptable in an ISR/task context where the connection
table update is lock-free and fast, but the design should note this is a best-effort
mitigation, not a guaranteed recovery.

### Step 3 — The edit (do NOT apply — planning only)

Replace the `else` branch of HCI_SendDataPkt as follows:

```c
  else // packet is okay
  {
    uint8 retries = 0;
    do
    {
      hciStatus = MAP_LL_TxData( connHandle, pData, pktLen, pbFlag );
      retries++;
    } while ( (hciStatus == LL_STATUS_ERROR_INACTIVE_CONNECTION) && (retries < 3) );
  }
```

This:
- Retries up to 3 times total (first attempt + 2 retries, loop exits after 3 iterations
  or on any non-INACTIVE_CONNECTION result, whichever comes first).
- Does NOT retry on LL_STATUS_ERROR_OUT_OF_TX_MEM (which would be "fatal" per the LL
  contract).
- Does NOT retry on LL_STATUS_ERROR_BAD_PARAMETER or UNEXPECTED_PARAMETER (which are
  deterministic and would not resolve on retry).
- Uses a `uint8` counter: no overflow risk with bound 3.
- Does not allocate any resources; no new free/RAII obligations.

### Step 4 — Verify the retry count semantics with the task description

The task says "retries up to 3 times." Disambiguate:
- If "retries" means 3 additional attempts after the first: use `retries < 3` as the
  loop guard with a post-increment counter starting at 0, and accept the loop runs
  at most 4 times total (1 initial + 3 retries). Adjust the condition to `retries < 3`
  with the counter incremented after the call, starting from 0.
- If "retries up to 3 times" means 3 total attempts: use `retries < 3` with increment
  before the do-while check, or restructure as a for loop with bound 3.

The plan above (do-while, retries starts at 0, increments after each call, loop continues
while retries < 3) produces: attempt 1 (retries becomes 1), attempt 2 (retries becomes 2),
attempt 3 (retries becomes 3, loop exits). That is 3 total calls, matching "retries up to
3 times" if interpreted as 3 total attempts. Confirm interpretation before writing.

## Summary of Preflight Findings

The critical finding is the LL contract violation risk: the existing LL_TxData API
documentation explicitly states that calling it again while a buffer is pending is "fatal."
The retry must be narrowly scoped to LL_STATUS_ERROR_INACTIVE_CONNECTION, which is the
only return code where the LL does not retain pData and a re-call is safe. The plan is
otherwise clean: no arithmetic hazards, no resource lifetime gaps, no call-ordering issues
beyond the one corrected by the gating condition.

LOCI MCP call graph data was not available for this analysis (static analysis only).
