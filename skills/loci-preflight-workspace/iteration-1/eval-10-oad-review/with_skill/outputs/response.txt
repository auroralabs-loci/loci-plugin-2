### Context

File: /home/melisa/BLE/simplelink-lowpower-f3-sdk/source/ti/ble5stack_flash/profiles/oad/cc26xx/oad.c
Function: OAD_open (line 424)
Signature: uint8_t OAD_open(uint32_t oadTimeout)

OAD_open is the one-time initialiser for the OAD (Over-the-Air Download) GATT profile.
It allocates the CCCD table, sets up pointer offsets into that table, initialises GATT
configuration, opens a POSIX message queue, starts a clock, registers the GATT service,
initialises the flash interface, and optionally creates a factory image backup.

### Check 1 — Call graph ordering (CFI)

Callee chain from OAD_open:
  ICall_malloc
  GATTServApp_InitCharCfg  (calls into linkDBNumConns loop — safe, linkDBNumConns is
                             initialised to MAX_NUM_BLE_CONNS at link time)
  mq_open
  Util_constructClock      (uses oadStateTimeout, which is set to OAD_DEFAULT_INACTIVITY_TIME
                             at file scope — value is valid before OAD_open is called)
  GATTServApp_RegisterService
  flash_init
  hasExternalFlash
  oadCheckFactoryImage     (conditional on !EXCLUDE_OAD_OFFCHIP && useExternalFlash)
  oadCreateFactoryImageBackup (conditional on oadCheckFactoryImage returning false)

ISSUE — Call ordering of Util_constructClock vs oadStateTimeout update:

  Util_constructClock is called at line 474 with oadStateTimeout as the period argument.
  The user-supplied oadTimeout is not applied to oadStateTimeout until lines 506-510,
  AFTER the clock is already constructed. The clock is therefore always built with the
  default value (OAD_DEFAULT_INACTIVITY_TIME = 10000 ms) regardless of what the caller
  passes to OAD_open. The intent of the parameter is defeated.

  Severity: RISK — the user's timeout is silently ignored for the clock period.
  The fix is to apply oadStateTimeout before calling Util_constructClock.

ISSUE — mq_open return value is never checked:

  hOadQ = mq_open("/OADQueue", O_CREAT | O_NONBLOCK, 0, &attr);   // line 471

  mq_open returns (mqd_t)-1 on failure and sets errno. The return value is stored in
  hOadQ but is never tested. All subsequent mq_receive / mq_send calls in the module
  use hOadQ directly. If mq_open fails, every queue operation will immediately fail with
  EBADF, and the module will silently drop all OAD messages with no diagnostic output.

  Severity: RISK — silent failure; queue operations proceed on an invalid descriptor.

### Check 2 — Arithmetic ranges

Issue 1 — uint8_t overflow in oadCCCDTableSize:

  uint8_t oadCCCDTableSize = (sizeof(gattCharCfg_t) * linkDBNumConns) * OAD_CCCD_CNT;
                                                                          (line 427-428)

  sizeof(gattCharCfg_t) is typically 4 bytes (uint16 connHandle + uint16 value, packed)
  or up to 8 bytes depending on alignment.
  linkDBNumConns defaults to MAX_NUM_BLE_CONNS, which is 3 (peripheral) or up to however
  the application defines it; the macro can legally be larger.
  OAD_CCCD_CNT = OAD_CHAR_CNT = 3.

  Worst-case in a typical build: 4 * 3 * 3 = 36 — fits in uint8_t.
  But if sizeof(gattCharCfg_t) = 8 (with padding) and MAX_NUM_BLE_CONNS = 12 (a legal
  value documented in the SDK), the product is 8 * 12 * 3 = 288, which wraps to 32 in
  uint8_t. ICall_malloc(32) succeeds, but the three pointer offsets at lines 447-449
  then point past the end of the allocation, corrupting adjacent heap or causing a fault.

  Severity: RISK — uint8_t is too narrow for the computed size; should be uint16_t or
  size_t.

Issue 2 — Pointer arithmetic relies on oadCCCDTableSize being correct:

  oadImgBlockConfig = oadCCCDTable + linkDBNumConns;      // line 447
  oadExtCtrlConfig  = oadCCCDTable + 2*linkDBNumConns;   // line 449

  These offsets are computed in units of gattCharCfg_t elements, which is correct as
  pointer arithmetic. However, if oadCCCDTableSize has overflowed (see Issue 1), the
  allocation is undersized and these pointers are out-of-bounds. This is a consequence
  of Issue 1 rather than an independent bug.

Issue 3 — Signed/unsigned comparison at line 506:

  if(oadTimeout > OAD_MIN_INACTIVITY_TIME)

  oadTimeout is uint32_t; OAD_MIN_INACTIVITY_TIME is defined as 4000 (an int literal).
  The comparison promotes 4000 to uint32_t, which is correct here. No risk.

### Check 3 — Freed-resource access / resource lifetimes

Issue 1 — oadCCCDTable heap allocation is not freed on the GATTServApp_RegisterService
          failure path:

  The function allocates oadCCCDTable via ICall_malloc at line 436.
  If GATTServApp_RegisterService fails (line 484), status is set to OAD_NO_RESOURCES
  and the function falls through and returns that status at line 511 — but oadCCCDTable
  is never freed. The three pointers oadImgIdentifyConfig, oadImgBlockConfig, and
  oadExtCtrlConfig remain pointing into a live but logically orphaned allocation.

  If OAD_open is retried (e.g., the application attempts a second call after failure),
  oadCCCDTable is overwritten with a new allocation, losing the reference to the first
  one entirely — a memory leak.

  Severity: RISK — heap leak on any failure path after the ICall_malloc succeeds.

Issue 2 — mq_open resource not freed on failure path:

  As noted in Check 1, mq_open's return is unchecked. Beyond that: if OAD_open fails
  after mq_open succeeds (e.g., GATTServApp_RegisterService returns failure), hOadQ
  holds an open message queue descriptor that is never closed with mq_close. On a retry
  of OAD_open, mq_open with O_CREAT on the same name "/OADQueue" will succeed and return
  a new descriptor — but there is now a duplicate open on the same queue, and the old
  descriptor is leaked.

  Severity: RISK — file-descriptor / queue-descriptor leak on retry.

Issue 3 — oadCreateFactoryImageBackup return value is conditionally ignored:

  In the #ifndef EXCLUDE_OAD_OFFCHIP block (lines 496-503):
    status = oadCreateFactoryImageBackup();

  This correctly propagates a failure. No issue here, the return value is captured.

### Summary

Call graph:  RISK — (a) Util_constructClock is called before oadStateTimeout is updated,
             so the user-supplied timeout is never applied to the clock period;
             (b) mq_open return value is unchecked, silent queue failure is possible.

Arithmetic:  RISK — oadCCCDTableSize is declared uint8_t; with larger connection counts
             or struct sizes the multiplication overflows, making the allocation
             undersized and the subsequent pointer offsets out-of-bounds.

Resources:   RISK — oadCCCDTable (heap) and hOadQ (queue descriptor) are not released
             on the GATTServApp_RegisterService failure path, causing leaks on retry.

Execution fit: ADJUST PLAN
-> Before touching any other OAD logic, address (1) the uint8_t size type for
   oadCCCDTableSize, (2) the missing mq_open failure check and corresponding cleanup,
   (3) the missing ICall_free on failure paths, and (4) move the oadTimeout ->
   oadStateTimeout assignment to before the Util_constructClock call.
