# Plan: Add Exponential Backoff to Connection Parameter Update Retry Logic

## Source Analysis

### File: app_connection.c (static analysis only)

The file handles BLE connection events on a TI CC2340R53 target, using the BLEAppUtil framework. The relevant logic lives in `Connection_ConnEventHandler`, specifically the `BLEAPPUTIL_LINK_PARAM_UPDATE_EVENT` case (lines 176-201).

**Current retry behavior:**
There is no retry logic in the current code. When a parameter update fails (i.e., `pPkt->status != SUCCESS`), the handler logs a message and exits. It does not attempt to retry the update. The task asks us to *add* retry logic with exponential backoff â€” this is a greenfield addition, not a modification of existing retry state.

**Key API available:**
- `BLEAppUtil_paramUpdateReq(gapUpdateLinkParamReq_t *pReq)` â€” sends a new parameter update request.
- `gapUpdateLinkParamReq_t` has fields: `connectionHandle`, `intervalMin`, `intervalMax`, `connLatency`, `connTimeout`, `signalIdentifier`.
- `App_connInfo` (in `connectionConnList[]`) holds `connHandle` and `peerAddress` but no retry state.
- RTOS timer mechanism: `Clock_Struct` + `Util_constructClock(...)` (used in app_cgm.c). This is the pattern for deferred/timed retries.
- `linkDB_GetInfo(connHandle, &linkInfo)` â€” used to confirm connection is still valid.

**Observation on existing PARAM_UPDATE_EVENT handler:**
The `pPkt->status` check is inside an `if (linkDB_GetInfo(...) == SUCCESS)` guard. Any retry must also operate within a valid connection context. If the connection has terminated between the failed update and the retry attempt, the retry must be suppressed.

## Preflight

## Preflight: Connection_paramUpdateRetryCallback (new function)

**Call graph:**
- The retry callback will call `BLEAppUtil_paramUpdateReq(gapUpdateLinkParamReq_t *pReq)`.
- `BLEAppUtil_paramUpdateReq` is declared in `bleapputil_api.h` (already included via the `bleapputil_api.h` include chain from `app_main.h`). No forward declaration needed.
- The callback will be invoked from the RTOS Clock module (like `CGMP_clockCB` in app_cgm.c). This means it fires in the application task context via a posted event â€” NOT from an ISR â€” which is consistent with how TI BLE app-layer callbacks work. Calling `BLEAppUtil_paramUpdateReq` from this context is safe.
- No recursion: the callback fires once per timer expiry; the timer is restarted only if the retry attempt fails again and the retry count has not been exhausted.
- Call order: the clock is constructed only after a failed update event, which means the BLE stack is already initialized. No init-ordering problem.

Call graph: OK

**Arithmetic ranges:**
- The backoff interval doubles on each retry: `next_interval = current_interval * 2`.
- If stored as `uint32_t` (milliseconds), doubling from an initial value of (for example) 1000 ms:
  - Retry 1: 1000 ms
  - Retry 2: 2000 ms
  - Retry 3: 4000 ms
  - Retry 4: 8000 ms
  - Retry 5: 16000 ms â†’ exceeds cap of 10000 ms, so clamp to 10000 ms.
- With `uint32_t`, the maximum pre-cap value is 16000 ms, which is well within `uint32_t` range (max ~4.29 billion). No overflow risk at the scale of seconds.
- RISK: If the initial value is ever set larger than ~2 billion ms (pathological), doubling would overflow `uint32_t`. Mitigate with a compile-time constant (`#define CONN_PARAM_RETRY_INITIAL_MS 1000U`) and a `static_assert` or documented precondition.
- The cap comparison (`if (next_interval > CONN_PARAM_RETRY_MAX_MS)`) is unsigned vs. unsigned â€” no sign/unsigned mismatch.
- Retry count stored as `uint8_t`: maximum sensible retry count before cap is reached is about 4 doublings from 1 s to 10 s. A `uint8_t` max (255) is more than sufficient; no wraparound concern.

Arithmetic: OK (with the precondition that the initial interval constant is kept small)

**Freed-resource access:**
- The retry state (retry count, current interval, connHandle, param struct) must outlive the callback. It cannot be a local in `Connection_ConnEventHandler`.
- Two storage options:
  1. A `static` or file-scope struct keyed by `connHandle` (simplest, matches existing `connectionConnList` pattern).
  2. Heap-allocated state passed as the `arg` to `Util_constructClock` (matches TI's clock callback idiom where the arg is a uint32 event ID).
- RISK (use-after-connection-termination): if the connection terminates before the retry timer fires, the `connHandle` stored in the retry state becomes stale. The callback must call `linkDB_GetInfo(connHandle, &linkInfo)` before calling `BLEAppUtil_paramUpdateReq`, and abort the retry if the connection is gone.
- RISK (double-arm of clock): if a second parameter update failure arrives while a retry timer is already running for the same `connHandle`, `Util_constructClock` called again would leak or corrupt the prior `Clock_Struct`. The plan must stop any in-flight clock before starting a new one (`Util_stopClock` then re-arm, or check-and-skip).
- No heap allocation is needed if using a static per-connection retry table (same pattern as `connectionConnList`).

Resources: WARN â€” two issues to address before writing:
  1. Stale-handle guard in the callback (call linkDB_GetInfo before retrying).
  2. Clock double-arm guard (Util_stopClock before Util_constructClock on repeated failures).

Execution fit: ADJUST PLAN
â†’ Add stale-connection check in the retry callback, and add a clock stop-before-restart guard when arming the retry timer on repeated failures.

## Adjusted Plan

### New state to add to the TU

Add a static retry-state table, one entry per connection, alongside `connectionConnList`:

```c
#define CONN_PARAM_RETRY_INITIAL_MS   1000U   /* 1 second */
#define CONN_PARAM_RETRY_MAX_MS       10000U  /* 10 seconds cap */
#define CONN_PARAM_RETRY_MAX_ATTEMPTS 8U      /* safety ceiling */

typedef struct {
    uint16_t       connHandle;
    uint32_t       nextIntervalMs;
    uint8_t        retryCount;
    Clock_Struct   retryClock;
    gapUpdateLinkParamReq_t lastReq;
} ConnParamRetryState_t;

static ConnParamRetryState_t retryStateList[MAX_NUM_BLE_CONNS];
```

### New header includes

`Clock_Struct` comes from `<ti/drivers/utils/List.h>` or the RTOS ClockP layer. The CGM profile includes it via the BLE util headers. Verify which header provides `Clock_Struct` in this TU's include chain; if not already transitive, add it explicitly.

### New forward declaration

```c
static void Connection_paramUpdateRetryCallback(uint32_t connHandleArg);
static void Connection_armRetryTimer(uint16_t connHandle, gapUpdateLinkParamReq_t *pReq, uint32_t delayMs);
static void Connection_cancelRetryTimer(uint16_t connHandle);
static ConnParamRetryState_t *Connection_getRetryState(uint16_t connHandle);
```

### Step 1 â€” Initialize retry state in Connection_start()

In `Connection_start()`, after the loop that initializes `connectionConnList`, add a loop that sets `retryStateList[i].connHandle = LINKDB_CONNHANDLE_INVALID` and zeros the retry count and interval for all entries.

### Step 2 â€” Cancel retry on termination

In `Connection_ConnEventHandler`, `BLEAPPUTIL_LINK_TERMINATED_EVENT` case, after calling `Connection_removeConnInfo`, call `Connection_cancelRetryTimer(gapTermMsg->connectionHandle)`.

This fixes the use-after-termination resource risk.

### Step 3 â€” Arm retry on failure

In `BLEAPPUTIL_LINK_PARAM_UPDATE_EVENT`, in the `else` branch (where `pPkt->status != SUCCESS`):

```c
ConnParamRetryState_t *pRetry = Connection_getRetryState(pPkt->connectionHandle);
if (pRetry != NULL && pRetry->retryCount < CONN_PARAM_RETRY_MAX_ATTEMPTS)
{
    // Compute next interval with cap
    uint32_t delay = (pRetry->retryCount == 0)
                     ? CONN_PARAM_RETRY_INITIAL_MS
                     : pRetry->nextIntervalMs;
    if (delay > CONN_PARAM_RETRY_MAX_MS) delay = CONN_PARAM_RETRY_MAX_MS;

    // Reconstruct the request from the last known good parameters
    // (store them at ESTABLISHED time or from the failed event's pPkt fields)
    Connection_armRetryTimer(pPkt->connectionHandle, &pRetry->lastReq, delay);

    // Advance state
    pRetry->nextIntervalMs = (delay * 2 > CONN_PARAM_RETRY_MAX_MS)
                              ? CONN_PARAM_RETRY_MAX_MS
                              : delay * 2;
    pRetry->retryCount++;
}
```

Note: `pPkt` (gapLinkUpdateEvent_t) does not directly carry the requested parameters â€” it carries the opcode and status. The connection parameters to retry with must come from a stored copy. Plan must include saving the desired parameters when the update is first requested (or at connection establishment from `gapEstLinkReqEvent_t` fields).

### Step 4 â€” Connection_armRetryTimer (with double-arm guard)

```c
static void Connection_armRetryTimer(uint16_t connHandle,
                                     gapUpdateLinkParamReq_t *pReq,
                                     uint32_t delayMs)
{
    ConnParamRetryState_t *pRetry = Connection_getRetryState(connHandle);
    if (pRetry == NULL) return;

    // Double-arm guard: stop any existing timer before re-arming
    Util_stopClock(&pRetry->retryClock);

    // Save the request to retry
    pRetry->lastReq = *pReq;

    // Construct one-shot clock; TI convention: period=0 means one-shot
    Util_constructClock(&pRetry->retryClock,
                        Connection_paramUpdateRetryCallback,
                        delayMs,    /* initial timeout */
                        0,          /* period = 0 â†’ one-shot */
                        TRUE,       /* start immediately */
                        (uint32_t)connHandle);
}
```

### Step 5 â€” Connection_paramUpdateRetryCallback

```c
static void Connection_paramUpdateRetryCallback(uint32_t connHandleArg)
{
    uint16_t connHandle = (uint16_t)connHandleArg;

    // Stale-connection guard
    linkDBInfo_t linkInfo;
    if (linkDB_GetInfo(connHandle, &linkInfo) != SUCCESS)
    {
        // Connection gone; abandon retry
        Connection_cancelRetryTimer(connHandle);
        return;
    }

    ConnParamRetryState_t *pRetry = Connection_getRetryState(connHandle);
    if (pRetry == NULL) return;

    // Issue the retry
    BLEAppUtil_paramUpdateReq(&pRetry->lastReq);
}
```

### Step 6 â€” Connection_cancelRetryTimer

```c
static void Connection_cancelRetryTimer(uint16_t connHandle)
{
    ConnParamRetryState_t *pRetry = Connection_getRetryState(connHandle);
    if (pRetry == NULL) return;
    Util_stopClock(&pRetry->retryClock);
    pRetry->retryCount = 0;
    pRetry->nextIntervalMs = 0;
    pRetry->connHandle = LINKDB_CONNHANDLE_INVALID;
}
```

### Step 7 â€” Connection_getRetryState (helper)

Finds a retry state entry by connHandle. Returns NULL if not found. Also returns a free slot (LINKDB_CONNHANDLE_INVALID) for new connections.

## Risk Summary (from Preflight)

| Risk | Severity | Mitigation in Plan |
|---|---|---|
| Stale connHandle in retry callback | WARN | `linkDB_GetInfo` guard in callback (Step 5) |
| Clock double-arm on repeated failure | WARN | `Util_stopClock` before re-arm (Step 4) |
| Missing copy of desired param values | WARN | Store `lastReq` in `retryStateList`; populate at connection or update-request time (Step 3 note) |
| uint32_t overflow in interval doubling | OK (low) | Checked: max pre-cap value 16000 ms, well within uint32_t |
| `gapLinkUpdateEvent_t` lacks the original requested params | Design gap | Store params at request issuance; plan must trace where the original request is initiated to add storage there |
