CURRENT STATE ANALYSIS

The file currently handles BLEAPPUTIL_LINK_PARAM_UPDATE_EVENT in Connection_ConnEventHandler. When a parameter update fails (pPkt->status != SUCCESS), it simply logs a message — there is no retry logic at all. The BLEAPPUTIL_LINK_PARAM_UPDATE_REQ_EVENT handler accepts or rejects requests based on connLatency but does not drive retries.

There is no existing retry counter, no timer mechanism, and no backoff state stored anywhere in this module.

PLAN

Step 1: Add per-connection backoff state to App_connInfo

The App_connInfo struct (defined in app_main.h or a related header) needs two new fields per connection:
- uint8_t paramUpdateRetryCount  — tracks how many retries have been attempted for this connection handle
- uint32_t paramUpdateRetryDelayMs — the current backoff delay in milliseconds

These fields should be initialized to 0 in Connection_start() when iterating over connectionConnList, and reset to 0 in Connection_addConnInfo() when a new connection is added, and again reset to 0 on a successful parameter update.

Step 2: Define backoff constants near the top of app_connection.c

Add the following macro definitions after the existing includes:

  #define PARAM_UPDATE_INITIAL_DELAY_MS   500      // 500 ms initial wait
  #define PARAM_UPDATE_MAX_DELAY_MS       10000    // 10 second cap
  #define PARAM_UPDATE_MAX_RETRIES        10       // safety limit to prevent infinite retries

Step 3: Introduce a timer or clock mechanism

Since this is an RTOS environment using BLEAppUtil, use the existing Clock/timer APIs available in the TI BLE stack (e.g., BLEAppUtil_startTimer or the underlying OSAL/RTOS clock primitives). Create a single one-shot clock instance per connection (or a shared one if only one connection at a time is expected) that fires after the backoff delay and triggers a retry.

Declare a static Clock_Struct or equivalent per connection slot:

  static Clock_Struct paramUpdateClocks[MAX_NUM_BLE_CONNS];
  static bool paramUpdateClockActive[MAX_NUM_BLE_CONNS];

Initialize these in Connection_start().

Step 4: Implement a retry callback function

Add a new static function:

  static void Connection_paramUpdateRetryClockCb(UArg connIndex)

This callback, invoked when the clock fires, should:
1. Retrieve the connHandle from connectionConnList[connIndex].
2. Re-issue the connection parameter update request using BLEAppUtil_setConnParams() or the equivalent API.
3. The callback itself does not increment the retry count — that happens when the result is received.

Step 5: Modify the BLEAPPUTIL_LINK_PARAM_UPDATE_EVENT handler

In the existing failure branch (pPkt->status != SUCCESS):

a. Look up the connection index using Connection_getConnIndex(pPkt->connectionHandle).
b. Check whether the retry count is below PARAM_UPDATE_MAX_RETRIES.
c. If retries remain:
   - Increment connectionConnList[idx].paramUpdateRetryCount.
   - Compute the delay: if retryCount == 1, use PARAM_UPDATE_INITIAL_DELAY_MS; otherwise double the previous delay, capping at PARAM_UPDATE_MAX_DELAY_MS.
   - Store the new delay back into connectionConnList[idx].paramUpdateRetryDelayMs.
   - Start the one-shot clock for this connection index with the computed delay.
   - Log a message indicating the retry attempt and scheduled delay.
d. If the retry limit is reached, log a final failure message and stop retrying.

In the success branch (pPkt->status == SUCCESS):
- Reset connectionConnList[idx].paramUpdateRetryCount to 0.
- Reset connectionConnList[idx].paramUpdateRetryDelayMs to 0.
- Stop the clock if it is somehow still active.

Step 6: Reset state on connection termination

In the BLEAPPUTIL_LINK_TERMINATED_EVENT handler, after calling Connection_removeConnInfo(), also stop any active retry clock for that connection index and reset the retry/delay fields.
