## Task
Modify Connection_getConnIndex in app_connection.c to return -1 instead of 0 when no
connection is found, so callers can distinguish between index 0 and "not-found".

## Source Analysis

Lines 443-456

Current implementation:

    uint16_t Connection_getConnIndex(uint16_t connHandle)
    {
      uint8_t i;

      for (i = 0; i < MAX_NUM_BLE_CONNS; i++)
      {
        if (connectionConnList[i].connHandle == connHandle)
        {
          return i;
        }
      }

      return LL_INACTIVE_CONNECTIONS;
    }

Key observations:
- Return type is uint16_t (unsigned 16-bit integer).
- The loop variable i is uint8_t.
- On a match, the function returns i (a value 0..MAX_NUM_BLE_CONNS-1).
- On no-match, the function returns LL_INACTIVE_CONNECTIONS which is defined as 0xFFFF
  (confirmed in .loci-build/aarch64/conn_stub.c: #define LL_INACTIVE_CONNECTIONS 0xFFFF).
- The declaration in app_main.h also uses uint16_t Connection_getConnIndex(uint16_t connHandle).

Callers (within this example tree) in app_pairing.c:

    uint8_t connIdx = Connection_getConnIndex(...);
    if (connIdx != LL_INACTIVE_CONNECTIONS)
    {
        memcpy(connList[connIdx].peerAddress, linkInfo.addr, B_ADDR_LEN);
    }

The caller stores the return value into uint8_t connIdx and checks against LL_INACTIVE_CONNECTIONS
(0xFFFF). This is already a type mismatch: assigning a uint16_t return value into uint8_t truncates
0xFFFF to 0xFF (255 decimal), not to 0. The comparison connIdx != LL_INACTIVE_CONNECTIONS then
compares uint8_t 0xFF against uint16_t 0xFFFF — the uint8_t is implicitly promoted to uint16_t
giving 0x00FF, which is != 0xFFFF, so the not-found guard silently fails. This is a pre-existing
bug independent of the proposed change.

## Preflight: Connection_getConnIndex (proposed change: return -1 on not-found)

### 1. Call graph (CFI)

The function has no callees — it only iterates connectionConnList, a file-scope array initialized
in Connection_start(). No ordering hazard: the function is always called after
Connection_start() registers the handlers and initializes the list.

No recursion, no static-initialization ordering concern, no forward-declaration gap.

Call graph: OK

### 2. Arithmetic ranges

The proposed change is to replace `return LL_INACTIVE_CONNECTIONS;` with `return -1;`.

The return type is uint16_t. Returning -1 from a uint16_t function is defined C behavior: -1 is
implicitly converted to (uint16_t)(-1) = 0xFFFF. This is identical in bit pattern to the current
LL_INACTIVE_CONNECTIONS sentinel (0xFFFF). So on the surface, returning -1 instead of
LL_INACTIVE_CONNECTIONS produces no change in behavior for a uint16_t return type.

This is the central problem: the stated goal (return -1 to distinguish not-found from index 0) is
unachievable without a return-type change.

- With return type uint16_t: -1 converts to 0xFFFF, not -1. A caller comparing against -1 using
  == will promote the uint16_t to int, giving 0xFFFF (65535) != -1. The comparison will always
  be false for callers that write `if (idx == -1)`.

- A caller writing `if (idx < 0)` will never see a negative value from a uint16_t function,
  regardless of what the function returns internally.

- Index 0 is a valid connection index. The value 0 is representable in uint16_t and distinctly
  different from 0xFFFF, so the existing sentinel (LL_INACTIVE_CONNECTIONS = 0xFFFF) already
  distinguishes index 0 from not-found correctly for uint16_t callers.

- If the goal is to let callers write `if (idx == -1)` or `if (idx < 0)`, the return type must
  change to a signed type (int8_t, int16_t, or int).

Existing caller arithmetic hazard (pre-existing, independent of this change):
  uint8_t connIdx = Connection_getConnIndex(...);
  The 0xFFFF not-found sentinel is truncated to 0xFF on assignment. 0xFF != LL_INACTIVE_CONNECTIONS
  (0xFFFF) after integer promotion in the guard, so the not-found path is silently bypassed.
  If connIdx holds 0xFF as a "not-found" result it is then used as an array index
  (connList[connIdx]) — an out-of-bounds access (index 255 into a MAX_NUM_BLE_CONNS=8 array).
  This is a security/safety hazard independent of the proposed change.

Arithmetic: ⚠ RISK — returning -1 from uint16_t is a no-op (converts to 0xFFFF); the stated
goal of distinguishing -1 from 0 requires a signed return type. Additionally, the pre-existing
truncation of the return value into uint8_t at every known call site creates an array-index
overflow risk for the existing sentinel.

### 3. Freed-resource access

The function reads connectionConnList[], a static array. It holds no heap allocations, acquires no
locks, and opens no resources. The function has no resource lifetime concern.

Resources: OK

## Adjusted Plan

The goal is to make callers able to distinguish index 0 (a valid first connection) from
"not found". This goal is already met by the existing code for any caller that correctly uses
uint16_t and compares against LL_INACTIVE_CONNECTIONS — 0xFFFF is not a valid index. However,
if the explicit requirement is to return -1 (signed sentinel), the following plan is correct:

Step 1 — Change the return type in the header.
  In app_main.h, change:
    uint16_t Connection_getConnIndex(uint16_t connHandle);
  to:
    int Connection_getConnIndex(uint16_t connHandle);

  Rationale: int can hold both 0..MAX_NUM_BLE_CONNS-1 (valid indices) and -1 (sentinel).
  A signed 16-bit type (int16_t) would also work since MAX_NUM_BLE_CONNS <= 127.

Step 2 — Change the return type and sentinel in app_connection.c.
  Change:
    uint16_t Connection_getConnIndex(uint16_t connHandle)
  to:
    int Connection_getConnIndex(uint16_t connHandle)
  Change the not-found return from:
    return LL_INACTIVE_CONNECTIONS;
  to:
    return -1;

  The loop variable i is uint8_t; returning i widens to int without loss since i < MAX_NUM_BLE_CONNS
  (<= 8). No overflow.

Step 3 — Fix all call sites.
  In app_pairing.c (and any other caller), change:
    uint8_t connIdx = Connection_getConnIndex(...);
    if (connIdx != LL_INACTIVE_CONNECTIONS)
  to:
    int connIdx = Connection_getConnIndex(...);
    if (connIdx != -1)

  This eliminates the pre-existing truncation hazard and makes the sentinel comparison correct.
  It also prevents connIdx from being used as an array index with an out-of-bounds value.

Step 4 — Verify no other callers store into uint8_t or compare against LL_INACTIVE_CONNECTIONS.
  Search for all uses of Connection_getConnIndex across the project tree before writing, to
  ensure no caller is missed. The search above found callers in app_pairing.c in multiple
  example directories; each must be updated in the same way.

---

## Key Risks Summary

1. CRITICAL (plan blocker): Returning -1 from a uint16_t function does NOT produce -1; it
   produces 0xFFFF. The return type must be changed to a signed type (int or int16_t) to
   achieve the stated goal.

2. PRE-EXISTING HAZARD: All known callers store the return value into uint8_t. With the current
   uint16_t return type, the 0xFFFF sentinel truncates to 0xFF, silently defeating the not-found
   guard and potentially causing an out-of-bounds array access at connList[0xFF]. This must be
   fixed at call sites as part of this change.

3. NO new resource or call-order hazards are introduced by this change.
