## Analysis of Connection_getConnIndex

### Current Implementation

The function is in:
  /home/melisa/BLE/examples/rtos/LP_EM_CC2340R53/ble5stack/basic_ble_profiles/app/app_connection.c
  lines 443-456

```c
uint16_t Connection_getConnIndex(uint16_t connHandle)
{
  uint8_t i;

  for (i = 0; i < MAX_NUM_BLE_CONNS; i++)
  {
    if (connectionConnList[i].connHandle == connHandle)
    {
      return i;
    }
  }

  return LL_INACTIVE_CONNECTIONS;
}
```

The function:
- Returns a uint16_t (declared in app_main.h line 318)
- Returns LL_INACTIVE_CONNECTIONS (defined in ll.h as 0xFF) when no match is found
- Returns the array index (0, 1, 2, ...) when a match is found
- The loop variable i is uint8_t, so valid indices are 0..MAX_NUM_BLE_CONNS-1

### The Core Problem

The current "not found" sentinel is LL_INACTIVE_CONNECTIONS (0xFF), not 0. So the current code already distinguishes "index 0" from "not found" — 0 means the first slot was matched, 0xFF means nothing matched.

The user is asking to return -1 instead of 0 when no connection is found. This is a sentinel-value change: replacing LL_INACTIVE_CONNECTIONS (0xFF) with -1 as the not-found indicator.

### Why This Is Non-Trivial

1. **Return type mismatch**: The function returns uint16_t. Returning -1 from a uint16_t function is legal C (it wraps to 0xFFFF), but it is semantically confusing and fragile. The right approach is to change the return type to int16_t (or int) so that -1 is an unambiguous signed negative value.

2. **All call sites check against LL_INACTIVE_CONNECTIONS**: The known caller in this file's project is in app_pairing.c line 154-157:
   ```c
   uint8_t connIdx = Connection_getConnIndex(...);
   if (connIdx != LL_INACTIVE_CONNECTIONS)
   {
       memcpy(connList[connIdx].peerAddress, ...);
   }
   ```
   If the function now returns -1 instead of LL_INACTIVE_CONNECTIONS, this guard will silently break: (uint8_t)(-1) == 0xFF == LL_INACTIVE_CONNECTIONS, so by accident it might still work in that one spot — but that is coincidence, not correctness. Any new caller using int16_t or int comparisons would break differently.

3. **LL_INACTIVE_CONNECTIONS is 0xFF (255)**: It was chosen because it is out of range for a valid array index (MAX_NUM_BLE_CONNS is typically 8). The value -1 (or 0xFFFF as uint16_t) serves the same logical purpose but requires signed arithmetic to be unambiguous.

### Recommended Plan

Step 1 — Change the return type in the .c file and the header.

In app_connection.c, change:
```c
uint16_t Connection_getConnIndex(uint16_t connHandle)
```
to:
```c
int16_t Connection_getConnIndex(uint16_t connHandle)
```

In app_main.h line 318, change the declaration to match:
```c
int16_t Connection_getConnIndex(uint16_t connHandle);
```

Step 2 — Change the not-found return value.

In app_connection.c, change the final return statement from:
```c
  return LL_INACTIVE_CONNECTIONS;
```
to:
```c
  return -1;
```

Step 3 — Update the caller in app_pairing.c.

The variable receiving the result must be widened to hold -1:
```c
// Before:
uint8_t connIdx = Connection_getConnIndex(...);
if (connIdx != LL_INACTIVE_CONNECTIONS)

// After:
int16_t connIdx = Connection_getConnIndex(...);
if (connIdx != -1)
```
Also update the array subscript usage on line 160 — since connIdx is now int16_t, the compiler will issue a signed/unsigned mismatch warning for array indexing. Add a cast or a local uint8_t after the guard:
```c
if (connIdx != -1)
{
    memcpy(connList[(uint8_t)connIdx].peerAddress, linkInfo.addr, B_ADDR_LEN);
}
```

Step 4 — Search for any other callers in this project tree.

Run a project-wide search for Connection_getConnIndex and audit every call site. In this repository there are many parallel project directories (basic_ble, data_stream, basic_persistent, etc.). The change above applies only to the basic_ble_profiles project for LP_EM_CC2340R53. Each other project that needs the same fix must be updated independently.

Step 5 — Update doxygen comment.

In app_connection.c lines 440-442 and in app_main.h lines 314-317, update the @return doc:
```
 * @return  the index of the entry that has the given connection handle.
 *          if there is no match, -1 will be returned.
```

### What NOT to Do

Do not simply return -1 while keeping uint16_t as the return type. (uint16_t)(-1) is 0xFFFF, which is a valid bit pattern but will silently compare unequal to the signed -1 in any caller that stores the result in a signed variable. The type change in step 1 is required to make the sentinel semantically correct.

Do not remove the LL_INACTIVE_CONNECTIONS macro from ll.h — it is used by the BLE stack itself and by many other parts of the SDK unrelated to this function.
