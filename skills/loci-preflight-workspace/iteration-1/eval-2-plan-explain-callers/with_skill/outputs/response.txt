### What connection_find does

Source: evals/preflight/fixtures/conn_manager.c, lines 24–31

    ble_conn_t *connection_find(uint16_t handle)
    {
        for (int i = 0; i < MAX_CONNECTIONS; i++) {
            if (s_conn_table[i].handle == handle && s_conn_table[i].active)
                return &s_conn_table[i];
        }
        return NULL;
    }

connection_find is a lookup function for the BLE connection table.

- It accepts a 16-bit connection handle (uint16_t handle).
- It performs a linear scan over s_conn_table, a static array of up to
  MAX_CONNECTIONS (8) ble_conn_t entries.
- For each entry it checks two conditions: the handle field matches AND the
  entry is marked active.
- If a matching active entry is found it returns a pointer to that entry
  (into the static array).
- If no match is found it returns NULL.

The function is intentionally simple: it is a pure lookup with no side effects
and no resource acquisition. The returned pointer is interior to a static
(process-lifetime) array, so it is safe to dereference as long as the entry
is not concurrently zeroed or the table is not reallocated (it is fixed-size).

Notable caveat in the fixture: the comment at the top of conn_manager.c states
that connection_find is also defined in legacy/conn_compat.c with a different
signature. That file is not present in the fixture directory, but the note
flags a real hazard — two definitions of the same symbol in the same link unit
is undefined behavior in C and will produce linker errors or silent
wrong-function dispatch depending on toolchain flags.

### Callers of connection_find

       int connection_close(uint16_t handle)
       {
           ble_conn_t *conn = connection_find(handle);
           if (!conn)
               return -1;
           memset(conn, 0, sizeof(*conn));
           return 0;
       }
