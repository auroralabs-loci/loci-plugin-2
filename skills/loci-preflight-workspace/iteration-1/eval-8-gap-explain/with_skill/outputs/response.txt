GAP_DeviceInit — Parameters and Initialization Flow
====================================================

Signature (from gap.h):

    bStatus_t GAP_DeviceInit(uint8_t       profileRole,
                             uint8_t       taskID,
                             GAP_Addr_Modes_t addrMode,
                             uint8_t      *pRandomAddr);

This call must be made only once per reset. It sets up the GAP layer and triggers
the asynchronous GAP_DEVICE_INIT_DONE_EVENT (gapDeviceInitDoneEvent_t) when
initialization completes.

---

Parameter 1: profileRole  (uint8_t)
-------------------------------------
A bitmask that tells the stack which BLE role(s) this device will play.

Valid single-role values:
  GAP_PROFILE_BROADCASTER  0x01  -- sends advertising events only (no connections)
  GAP_PROFILE_OBSERVER     0x02  -- receives advertising events only (no connections)
  GAP_PROFILE_PERIPHERAL   0x04  -- accepts incoming connection requests
  GAP_PROFILE_CENTRAL      0x08  -- initiates outgoing connection requests

Valid multi-role combinations (OR of the above):
  BROADCASTER | OBSERVER
  PERIPHERAL  | OBSERVER
  CENTRAL     | BROADCASTER
  CENTRAL     | PERIPHERAL

Any other combination returns INVALIDPARAMETER. Additionally, each role must
match the HOST_CONFIG compile-time flags or the call fails (e.g., you cannot
enable CENTRAL on a build compiled without CENTRAL_CFG).

---

Parameter 2: taskID  (uint8_t)
--------------------------------
The OSAL task ID of the application task that should receive unprocessed GAP
events. GAP events that are not consumed by internal stack layers are forwarded
as OSAL messages to this task. In the BLEAppUtil layer the value passed here is
BLEAppUtilSelfEntity, which is the entity registered for the app's event loop.
There is no explicit range check on this field in the source — it is stored
directly and used to route messages.

---

Parameter 3: addrMode  (GAP_Addr_Modes_t)
------------------------------------------
Specifies how the local device's own Bluetooth address is formed for all
subsequent operations (advertising, scanning, initiating, etc.).

  ADDRMODE_PUBLIC            (0x00)  Always use the factory-programmed public
                                     address. pRandomAddr is ignored.

  ADDRMODE_RANDOM            (0x01)  Always use a Random Static Address.
                                     pRandomAddr must be non-NULL and valid.

  ADDRMODE_RP_WITH_PUBLIC_ID (0x02)  Use a Resolvable Private Address (RPA)
                                     that encodes the public address as the
                                     identity. Peers that have the IRK can
                                     resolve the RPA back to the public address.
                                     pRandomAddr is ignored.

  ADDRMODE_RP_WITH_RANDOM_ID (0x03)  Use a Resolvable Private Address (RPA)
                                     that encodes a random static address as
                                     the identity. pRandomAddr must be non-NULL
                                     and valid.

Important: if the address mode changes between resets (compared to what was
stored in NV), the stack erases all bonds and locally stored information from
NV before reinitializing.

IRK behavior is tied to addrMode:
  - PUBLIC or RANDOM:       IRK is forced to all-zeros (identity address mode,
                            no privacy).
  - RP_WITH_PUBLIC_ID or
    RP_WITH_RANDOM_ID:      IRK is loaded in priority order:
                              1. Value set by GapConfig_SetParameter() before
                                 this call.
                              2. Value previously stored in NV by gapbondmgr.
                              3. Randomly generated.

---

Parameter 4: pRandomAddr  (uint8_t *)
---------------------------------------
Pointer to a 6-byte buffer containing the Random Static Address to program into
the controller.

Required (must be non-NULL) when addrMode is:
  ADDRMODE_RANDOM or ADDRMODE_RP_WITH_RANDOM_ID

Ignored (may be NULL) when addrMode is:
  ADDRMODE_PUBLIC or ADDRMODE_RP_WITH_PUBLIC_ID

Validity rules enforced by GAP_DeviceInit_validate_params():
  - The two most-significant bits of byte[5] must be 11b (static random address
    header, checked by GAP_IS_ADDR_RS).
  - The address must not be all-zeros in the lower bits (excluding the 2 MSBs
    of byte[5]).
  - The address must not be all-0xFF.

Passing an address that fails these checks returns INVALIDPARAMETER.

---

Return values
-------------
  SUCCESS            Initialization has started; wait for GAP_DEVICE_INIT_DONE_EVENT.
  INVALIDPARAMETER   profileRole is unsupported, addrMode is out of range, or
                     pRandomAddr is NULL / invalid when required.
  bleIncorrectMode   GAP_DeviceInit has already been called since the last reset.
  bleInternalError   NV erase (triggered by address-mode change) failed.

---

Initialization flow summary
----------------------------
1. (Optional) Call GapConfig_SetParameter() to pre-set IRK or SRK before init.
2. Call GAP_DeviceInit() with the desired role, task ID, address mode, and
   (if needed) random address.
3. Internally the stack:
   a. Validates all parameters.
   b. Detects whether addrMode changed vs. NV; if so, erases NV bonds.
   c. Programs the controller address.
   d. Calls gap_ParamsInit() to set up role-specific state.
   e. Resolves IRK/SRK from user config, NV, or random generation.
   f. Calls GAP_DeviceInit_per_role() to initialize security, privacy,
      advertising, and scanning subsystems appropriate for the role.
4. The application receives GAP_DEVICE_INIT_DONE_EVENT carrying the resolved
   device address, HCI data packet length, and total packet count.
