GAP_DeviceInit takes four parameters. Here's what each one does:

```c
extern bStatus_t GAP_DeviceInit(uint8_t profileRole,
                                uint8_t taskID,
                                GAP_Addr_Modes_t addrMode,
                                uint8_t* pRandomAddr);
```

---

1. profileRole (uint8_t)

This is a bitmask specifying what GAP role(s) the device will operate in. The valid values are:

- GAP_PROFILE_BROADCASTER (0x01) — sends advertising events only, no connections
- GAP_PROFILE_OBSERVER (0x02) — receives advertising events only, no connections
- GAP_PROFILE_PERIPHERAL (0x04) — accepts incoming connections (the "server" side)
- GAP_PROFILE_CENTRAL (0x08) — initiates connections (the "client" side)

Most applications pass a single role (e.g., GAP_PROFILE_PERIPHERAL for a sensor). Multi-role devices can combine them, but not all combinations are valid — passing an invalid combination returns INVALIDPARAMETER.

---

2. taskID (uint8_t)

This is the OSAL/ICall task ID of your application task. The BLE stack uses this to route unprocessed GAP events back to your app. When initialization completes, the stack sends a GAP_DEVICE_INIT_DONE_EVENT (as a gapDeviceInitDoneEvent_t) to the task identified by this ID. In practice, you pass selfEntity (or equivalent) — the handle your task registered with ICall.

---

3. addrMode (GAP_Addr_Modes_t)

This controls what kind of Bluetooth address the device will use for advertising and connecting. The options are:

- ADDRMODE_PUBLIC (0x00) — always use the device's hardwired public address
- ADDRMODE_RANDOM (0x01) — always use a Random Static Address (you supply this via pRandomAddr)
- ADDRMODE_RP_WITH_PUBLIC_ID (0x02) — use a Resolvable Private Address (RPA) that rotates periodically, with the public address as the identity address
- ADDRMODE_RP_WITH_RANDOM_ID (0x03) — use a Resolvable Private Address with a random static address as the identity address

RPAs (modes 0x02 and 0x03) are the privacy-preserving option — the device's on-air address changes over time so passive observers can't track it, but bonded peers can still resolve it using the IRK.

---

4. pRandomAddr (uint8_t*)

A pointer to a 6-byte array containing the Random Static Address you want the device to use. This parameter is only meaningful when addrMode is ADDRMODE_RANDOM or ADDRMODE_RP_WITH_RANDOM_ID — in those cases it must not be NULL. For ADDRMODE_PUBLIC and ADDRMODE_RP_WITH_PUBLIC_ID, this parameter is ignored (pass NULL or anything).

The stack copies the address internally, so you don't need to keep the buffer alive after the call returns.

---

Initialization flow overview

GAP_DeviceInit can only be called once per reset. It kicks off asynchronous initialization — the return value just tells you whether the request was accepted (SUCCESS), not that initialization is complete. When the stack finishes, it delivers a GAP_DEVICE_INIT_DONE_EVENT to your task (taskID). That event (gapDeviceInitDoneEvent_t) carries the device's resolved address, the LE data packet length, and the number of available data packets.

If you need to change the address mode or random address, you have to reset the device and call GAP_DeviceInit again. If the new configuration differs from what was stored in NV last time, all bonds and local NV data are erased automatically.

A couple of things must happen before GAP_DeviceInit is called if you want to override defaults:

- IRK and SRK can be set via GapConfig_SetParameter(GAP_CONFIG_PARAM_IRK, ...) and GapConfig_SetParameter(GAP_CONFIG_PARAM_SRK, ...). If you don't set them, the stack will use values from NV (if available from a previous pairing session) or generate new random ones.

Typical call in a peripheral application:

```c
GAP_DeviceInit(GAP_PROFILE_PERIPHERAL, selfEntity, addrMode, &pRandomAddress);
```

where addrMode and pRandomAddress are typically defined in a board-level config header (e.g., ti_ble_config.h).
