The function already has a "pIdx" output parameter (uint8_t *). When a
matching bond is found — either by direct address comparison or by RPA
resolution against a stored IRK — it writes the zero-based bond slot
index into *pIdx if pIdx != NULL, then returns SUCCESS. If no match is
found it returns bleGAPNotFound. pIdx may be NULL (callers that don't
need the index pass NULL).

In other words: the bond index is already an output parameter of
GAPBondMgr_FindAddr. It exists today. The task as stated describes
adding something that is already present.

INTERPRETATION OF THE TASK
----------------------------------------------------------------------

Given the existing signature already carries pIdx, the meaningful
interpretation of "add a method that also returns the bond index as an
output parameter" is one of:

  A. Add a NEW wrapper function — e.g., GAPBondMgr_FindAddrWithIndex —
     that makes the bond-index output mandatory (non-nullable) and
     documents the contract explicitly, delegating to the existing
     function internally.

  B. Add an overloaded or convenience variant that collapses the five
     parameters into a simpler three-parameter form where the bond index
     is always returned.

The plan below targets interpretation A: a new public function
GAPBondMgr_FindAddrWithIndex that wraps the existing logic and enforces
that the caller always receives a bond index on success.

## Preflight: GAPBondMgr_FindAddrWithIndex

### 1. Call graph ordering (CFI)

The new function will call GAPBondMgr_FindAddr, which is defined earlier
in the same translation unit (line 992) and declared in gapbondmgr.h.
No ordering problem: GAPBondMgr_FindAddr is fully defined before any
call site and its prototype is visible at the top of the TU via
the #include chain. No recursion. The new function itself requires no
init sequence — it is stateless beyond what GAPBondMgr_FindAddr already
manages.

Call graph: OK

### 2. Arithmetic ranges

The new function passes through pDevAddr and addrType unchanged to
GAPBondMgr_FindAddr, which already validates both (NULL check on
pDevAddr; range check on addrType against
PEER_ADDRTYPE_RANDOM_OR_RANDOM_ID). The only local variable is the
uint8_t that receives *pIdx from the inner call. uint8_t is wide enough
for the bond index: gapBond_maxBonds is at most 10 (GAP_BONDINGS_MAX),
and the loop index inside GAPBondMgr_FindAddr uses uint8_t throughout.
No new arithmetic is introduced.

Arithmetic: OK

### 3. Freed-resource access

The new function owns no heap resources. It passes pointers it receives
as arguments directly to GAPBondMgr_FindAddr. The internal `bonds` array
and `gapBond_maxBonds` are module-level statics owned by gapbondmgr.c;
GAPBondMgr_FindAddr already reads them under its existing access
discipline. The new wrapper does not acquire, allocate, or free anything.

The one lifetime concern worth noting: pBondIndex is a caller-owned
output pointer. The function must validate it is non-NULL before
writing through it (since the whole point of the new variant is to
guarantee the index is returned — passing NULL defeats that). A NULL
guard with INVALIDPARAMETER return should be the first check.

Resources: OK

   proceed with the wrapper plan.

Step 1 — Header declaration (gapbondmgr.h)
  Add a new extern declaration immediately after the existing

        uint8_t *pBondIndex);

  pBondIndex is required (not nullable). The function returns SUCCESS
  with *pBondIndex set, or INVALIDPARAMETER if either pDevAddr or
  pBondIndex is NULL, or bleGAPNotFound if no match exists.

Step 2 — Implementation (gapbondmgr.c)
  Place the new function immediately after GAPBondMgr_FindAddr
  (after line 1063). The implementation:

    bStatus_t GAPBondMgr_FindAddrWithIndex(uint8_t *pDevAddr,
                                           GAP_Peer_Addr_Types_t addrType,
                                           uint8_t *pBondIndex)
    {
      // Validate required output parameter
      if (pBondIndex == NULL)
      {
        return (INVALIDPARAMETER);
      }

      // Delegate to existing function; pBondIndex serves as pIdx
      return GAPBondMgr_FindAddr(pDevAddr, addrType,
                                 pBondIndex, NULL, NULL);
    }

  Key design decisions justified by preflight:
  - NULL guard on pBondIndex placed first — this is the one new
    resource/validity check that GAPBondMgr_FindAddr does not perform
    (it allows a NULL pIdx). Without this guard a caller who passes
    NULL would get SUCCESS with no index written, violating the
    wrapper's contract.
  - No duplicate validation of pDevAddr or addrType — GAPBondMgr_FindAddr
    already checks both and returns INVALIDPARAMETER / bleInvalidRange
    respectively; re-checking here would be redundant.
  - pIdentityAddrType and pIdentityAddr are passed as NULL because the
    narrow contract of this wrapper is specifically "find and return
    index." Callers who need identity information should use the full
    GAPBondMgr_FindAddr API.

No forward declarations needed: GAPBondMgr_FindAddr is defined before
the new function in the same file, and both are declared in gapbondmgr.h
which is already included.

No arithmetic changes. No resource ownership changes. No call-order
assumptions introduced.
