Plan: Adding a Watchdog Reset Inside OAD_processQueue for a 30-Second Stall
============================================================================

File: /home/melisa/BLE/simplelink-lowpower-f3-sdk/source/ti/ble5stack_flash/profiles/oad/cc26xx/oad.c
Function: OAD_processQueue (lines 629-818)

---

Understanding the Existing Code
--------------------------------

OAD_processQueue is called by the application task whenever the OAD profile signals
that the queue needs servicing (via OAD_QUEUE_EVT). It drains all messages currently
in the POSIX message queue hOadQ using a mq_receive loop, dispatches each message
through a state machine (OAD_IDLE -> OAD_CONFIG -> OAD_DOWNLOAD -> OAD_COMPLETE),
and returns a status code.

Key observations:
- The queue is opened with O_NONBLOCK, so mq_receive returns immediately with -1
  when the queue is empty. There is no blocking inside this function.
- The function is driven externally: the application task calls it only when it
  receives an event. If the BLE peer stops sending blocks mid-download, the
  application task may never call OAD_processQueue again — the queue is simply
  empty, and the OAD state machine sits frozen in OAD_DOWNLOAD.
- There is already an inactivity timer (oadActivityClk, a Clock_Struct) that fires
  oadInactivityTimeout after oadStateTimeout milliseconds (default 10 000 ms). That
  callback enqueues an OAD_TIMEOUT event and signals the application, causing
  OAD_processQueue to reset the state machine gracefully. So for normal inactivity,
  the existing mechanism is sufficient.
- The 30-second stall scenario described is therefore a case where
  OAD_processQueue is being called but the queue contains no messages and nothing
  is making progress — for example, the activity clock was never started, or an
  edge-case bug prevents the timer from firing — yet the OAD state is stuck in a
  non-IDLE state (e.g., OAD_DOWNLOAD) for more than 30 seconds.

The goal is: if OAD_processQueue is entered and the state machine is in a
non-IDLE state but finds nothing to process, and this condition has persisted for
more than 30 continuous seconds, perform a hardware reset via SystemReset() so the
device recovers cleanly.

Step-by-Step Plan
-----------------

### Step 1: Add a static timestamp variable to track the "queue empty while
           non-idle" condition.

Add a new static variable just below the other static state variables (around
line 328, near oadStateTimeout):

    static uint32_t oadStallTimestamp = 0;  // tick count when stall was first detected

This will record when we first detected the queue-empty-while-busy condition.
A value of 0 means "not currently tracking a stall."

### Step 2: Include a clock-tick or time source.

The code already includes "util.h" which wraps TI-RTOS/FreeRTOS Clock primitives.
On TI-RTOS the appropriate call is Clock_getTicks() (returns system ticks, each
tick is typically 10 us on CC23xx). On FreeRTOS the equivalent is
xTaskGetTickCount() / pdMS_TO_TICKS(). To stay consistent with the existing
abstraction layer in util.h, the safest portable choice is:

    #include <ti/sysbios/knl/Clock.h>   // for TIRTOS
    // or rely on Clock_getTicks() already exposed through the BLE stack includes

Alternatively, since the project already uses POSIX headers (mqueue.h is included),
we can use clock_gettime(CLOCK_MONOTONIC, ...) which is available in the
SimpleLink FreeRTOS POSIX layer. This is the cleanest cross-RTOS approach.

Choose clock_gettime(CLOCK_MONOTONIC, ...) for portability. Add:

    #include <time.h>

near the top of the file, alongside the existing <mqueue.h> include.

### Step 3: Define the stall threshold constant.

In the CONSTANTS section (around line 79):

    #define OAD_WATCHDOG_STALL_TIMEOUT_S    30U   /* seconds */

### Step 4: Modify OAD_processQueue to detect a prolonged stall and reset.

The logic should be inserted just after the mq_receive loop exits (i.e., after the
while loop at line 796), before the global status is stored. Here is the
precise location and the logic:

At line 796, just after the closing brace of the while (msgSize > 0) loop and
before "// Store the global status":

    // ---- Watchdog stall detection ----------------------------------------
    // If the state machine is stuck in a non-IDLE state but there are no
    // messages to process, track how long this condition has persisted.
    // If it exceeds OAD_WATCHDOG_STALL_TIMEOUT_S seconds, force a reset.
    if (state != OAD_IDLE && msgSize < 0)
    {
        struct timespec now;
        clock_gettime(CLOCK_MONOTONIC, &now);

        if (oadStallTimestamp == 0)
        {
            // First time we observed a stall: record the current time.
            oadStallTimestamp = (uint32_t)now.tv_sec;
        }
        else
        {
            uint32_t elapsed = (uint32_t)now.tv_sec - oadStallTimestamp;
            if (elapsed >= OAD_WATCHDOG_STALL_TIMEOUT_S)
            {
                // Queue has been stalled for 30+ seconds while in an active
                // OAD state. Perform a system reset to recover.
                SystemReset();
            }
        }
    }
    else
    {
        // Either we processed a message or we are in IDLE state;
        // either way the stall condition does not apply, reset the tracker.
        oadStallTimestamp = 0;
    }
    // ---- End watchdog stall detection ------------------------------------

### Step 5: Reset oadStallTimestamp inside oadResetState().

oadResetState() already resets all OAD state variables. Add:

    oadStallTimestamp = 0;

near the top of oadResetState() (around line 1058), after the state = OAD_IDLE
assignments, so that if the state machine is reset by any other mechanism (timeout,
disconnect, cancel), the stall timer is also cleared.

### Step 6: Ensure SystemReset() is accessible.

SystemReset() is the CMSIS function that issues a NVIC_SystemReset on Cortex-M.
It is already used in the same OAD directory (mark_switch_factory_img.c line 94).
It comes from <ti/devices/DeviceFamily.h> -> DeviceFamily_constructPath(driverlib/sys_ctrl.h)
or directly from the CMSIS core_cm*.h header pulled in by the driverlib includes
already present at the top of oad.c (<driverlib/chipinfo.h>, <driverlib/flash.h>).
No additional include should be needed, but if the linker complains, add:

    #include <ti/devices/DeviceFamily.h>
    #include DeviceFamily_constructPath(driverlib/sys_ctrl.h)

and call SysCtrlSystemReset() instead, which is the TI driverlib wrapper.

1. Add #include <time.h> near the top of oad.c (alongside <mqueue.h>).
2. Add #define OAD_WATCHDOG_STALL_TIMEOUT_S 30U in the CONSTANTS section.
3. Add static uint32_t oadStallTimestamp = 0; with the other static state variables.
4. After the while loop in OAD_processQueue (line ~796), insert the stall-detection
   block that calls clock_gettime(CLOCK_MONOTONIC) and invokes SystemReset() if
   30 seconds have elapsed while state != OAD_IDLE and the queue was empty.
5. Inside oadResetState(), add oadStallTimestamp = 0 to clear the tracker on any
   deliberate reset.

Important Caveats and Design Considerations
-------------------------------------------

- The existing oadActivityClk inactivity timer already handles normal disconnection
  or peer-side stalls by firing OAD_TIMEOUT after oadStateTimeout ms (default 10 s).
  The watchdog reset added here is a last-resort defense for cases where that timer
  fails to fire or the callback is never invoked. Setting OAD_WATCHDOG_STALL_TIMEOUT_S
  to 30 seconds gives the normal path a wide margin to succeed first.

- SystemReset() is abrupt — it does not gracefully close flash, free memory, or
  notify the peer. This is intentional for a watchdog scenario where normal
  recovery has already failed. The flash_close() call inside oadResetState() is
  not reachable in this path, but a 30-second stall suggests something has already
  gone wrong that warrants a hard reset.

- The stall detection only triggers when OAD_processQueue is actually called. If
  the application task is itself blocked and never calls OAD_processQueue, this
  watchdog will not fire. That would be a separate issue requiring a hardware
  watchdog timer at the system level.

- clock_gettime with CLOCK_MONOTONIC returns wall-clock seconds and is safe from
  wrap-around issues over 30-second windows. No tick-to-millisecond conversion is
  needed, avoiding potential overflow arithmetic.
