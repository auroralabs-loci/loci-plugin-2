# How to Add a Watchdog Reset Inside OAD_processQueue for a 30-Second Stall

## Understanding OAD_processQueue

OAD_processQueue (oad.c, line 629) is the OAD state-machine driver. It is called
from the application task loop whenever the OAD queue event fires
(SP_OAD_QUEUE_EVT). It is NOT an ISR — it runs in normal task context under
FreeRTOS. The function:

1. Calls mq_receive(hOadQ, ...) in a while loop to drain all pending messages.
2. Dispatches each message through a state machine (IDLE → CONFIG → DOWNLOAD →
   COMPLETE) using the module-static variables `state` and `nextState`.
3. Frees pData with ICall_free() after each iteration.
4. Returns a uint8_t status code.

The existing inactivity mechanism (oadActivityClk / oadInactivityTimeout) fires
when no BLE writes arrive for oadStateTimeout ms. That enqueues OAD_TIMEOUT and
asks the application task to call OAD_processQueue again. The timeout handles
peer silence — it does NOT handle the case where OAD_processQueue itself stops
being called (i.e., the application task is stuck or the queue is stalled at the
OS level).

The "queue stalled for more than 30 seconds" scenario means:
- Messages are present in hOadQ but OAD_processQueue is never called again, OR
- No new block arrives and the activity clock also fails to fire (OS-level stall).

The natural watchdog placement is therefore: record a wall-clock timestamp the
first time OAD enters a non-IDLE state, and on every call to OAD_processQueue
compare the current time. If >30 s have elapsed in a non-IDLE state without
completion, trigger a system reset.


## Preflight Analysis

### Call graph (CFI)

The new logic needs:

- A way to read wall time. The FreeRTOS path already uses clock_gettime with
  CLOCK_REALTIME (confirmed in icall_POSIX.c line 2223) and <time.h> is
  reachable. clock_gettime(CLOCK_MONOTONIC, ...) is the correct choice here
  (monotonic avoids wall-clock jumps).

  CLOCK_MONOTONIC is declared in <time.h>. That header is not currently included
  directly in oad.c, but is pulled in transitively through the FreeRTOS/POSIX
  chain (<mqueue.h> is already included at line 54). To be safe, a direct
  #include <time.h> should be added.

- A reset call. mark_switch_factory_img.c (same profile directory) uses
  SystemReset() (CMSIS core function, declared in core_cm0plus.h which is
  included via driverlib/chipinfo.h, already included at line 51). So
  SystemReset() is callable from oad.c without any new include.

- Two new static variables in oad.c:
    static bool         oadWdgArmed = false;
    static struct timespec oadWdgStart;

  struct timespec is from <time.h>.

No new forward declarations are needed. No recursion is introduced. The new code
runs in task context (same as today), so no ISR constraints apply.

Call graph: OK

### Arithmetic ranges

The elapsed-time calculation:

    elapsed_s = (now.tv_sec - oadWdgStart.tv_sec)
              + (now.tv_nsec - oadWdgStart.tv_nsec) / 1000000000LL;

Key risks:

1. tv_sec is time_t, which is a signed 64-bit integer on LP_F3 POSIX targets.
   The subtraction (now.tv_sec - oadWdgStart.tv_sec) cannot overflow in any
   plausible embedded lifetime. OK.

2. tv_nsec subtraction: tv_nsec is always in [0, 999999999]. The result of
   (now.tv_nsec - oadWdgStart.tv_nsec) is in [-999999999, +999999999]. If we
   only care about seconds granularity (which is fine for a 30 s watchdog), we
   can simply compare tv_sec difference:

       time_t elapsed_s = now.tv_sec - oadWdgStart.tv_sec;

   This is safe: since CLOCK_MONOTONIC cannot go backwards, elapsed_s >= 0
   always. No unsigned-wraparound risk, no shift hazard, no signed/unsigned mix
   (both sides are time_t, which is signed).

3. The threshold is a compile-time constant (30). No range issue.

Arithmetic: OK

### Freed-resource access

The two new static variables (oadWdgArmed, oadWdgStart) have static storage
duration and are never freed. No heap allocation is involved. The only resource
paths to check:

- oadWdgArmed is set to true when state first leaves IDLE. It is cleared in
  oadResetState(). oadResetState() already clears all state variables; adding
  a reset of oadWdgArmed there is the correct and complete cleanup.

- If OAD_processQueue triggers SystemReset(), the MCU reboots — no resource
  cleanup path is needed because the reset is intentional and total.

- pData (oadWriteEvt.pData) is freed in the existing loop at line 787.
  The new watchdog check fires at the *top* of OAD_processQueue before
  entering the while loop, so it does not interact with pData lifetimes at all.

- The mq_receive call at line 637 runs before the watchdog check, so no message
  is consumed and then abandoned if a reset fires — the reset happens before any
  message is read in the new invocation.

  Wait: if we put the check AFTER the first mq_receive (i.e., inside the loop),
  a message could be partially processed. Correct placement is BEFORE the first
  mq_receive — at the very top of the function, before line 637.

Resources: OK

## Adjusted Design Plan

### Step 0 — Add include (if not transitively guaranteed)

At the top of oad.c includes section, add:

    #include <time.h>

This ensures struct timespec and clock_gettime are declared without relying on
transitive inclusion order.

### Step 1 — New static variables (LOCAL VARIABLES section, ~line 285)

    /* Watchdog: detect queue stall > 30 s in any non-IDLE OAD state */
    #define OAD_STALL_WATCHDOG_SEC  30
    static bool            oadWdgArmed  = false;
    static struct timespec oadWdgStart;

### Step 2 — Arm the watchdog when state leaves IDLE

OAD_processQueue updates `state = nextState` at line 792 inside the while loop.
After that assignment, add:

    /* Arm stall watchdog when OAD becomes active */
    if (!oadWdgArmed && state != OAD_IDLE)
    {
        clock_gettime(CLOCK_MONOTONIC, &oadWdgStart);
        oadWdgArmed = true;
    }

### Step 3 — Check elapsed time at function entry (before first mq_receive)

At the top of OAD_processQueue, between the variable declarations and the first
mq_receive call (after line 636, before line 637):

    /* Stall watchdog: if non-IDLE OAD has made no progress for
     * OAD_STALL_WATCHDOG_SEC seconds, force a system reset. */
    if (oadWdgArmed)
    {
        struct timespec now;
        clock_gettime(CLOCK_MONOTONIC, &now);
        if ((now.tv_sec - oadWdgStart.tv_sec) >= OAD_STALL_WATCHDOG_SEC)
        {
            /* Queue stalled — reset to recover. */
            SystemReset();
        }
    }

Note: SystemReset() does not return, so no cleanup is needed after it. The
ICall_free paths in the while loop are never reached, which is correct — the
MCU resets immediately.

### Step 4 — Reset the watchdog in oadResetState()

In oadResetState() (line 1055), add near the top (after state = OAD_IDLE):

    oadWdgArmed = false;

This disarms the watchdog whenever the OAD state machine is reset normally (on
timeout, error, cancel, or close), preventing spurious resets on the next OAD
session.

### Step 5 — Reset the timestamp on each successful block write

Optionally, refresh oadWdgStart each time a block is successfully received so
the watchdog measures "time since last progress" rather than "total OAD time":

In OAD_processQueue, inside the OAD_DOWNLOAD / OAD_WRITE_BLOCK_REQ branch,
after a successful oadImgBlockWrite, add:

    if (oadWdgArmed)
    {
        clock_gettime(CLOCK_MONOTONIC, &oadWdgStart);
    }

This is the "sliding window" variant. If the intent is instead "total OAD
duration must not exceed 30 s", omit Step 5. For a stall detector (no progress),
include it.


## Risk Summary

No blocking issues were found. The design is clean with three small, localized
changes: a new include, two statics, one check at function entry, and one clear
in oadResetState. The most important ordering constraint is that the watchdog
check must precede the first mq_receive so no partially-consumed message is
abandoned on reset. SystemReset() is already established in this codebase
(mark_switch_factory_img.c) and reachable without new includes.

Static analysis only — LOCI MCP call graph was not queried for this function
(the available MCP tool is for assembly execution time, not for call graph
traversal).
