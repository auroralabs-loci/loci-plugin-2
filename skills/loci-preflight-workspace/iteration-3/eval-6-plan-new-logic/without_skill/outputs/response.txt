## Plan: Add Exponential Backoff to Connection Parameter Update Retry Logic

**Note:** I do not have direct access to `$LOCI_TEST_BLE_ROOT/examples/rtos/LP_EM_CC2340R53/ble5stack/basic_ble_profiles/app/app_connection.c` in this environment. The plan below describes exactly what to do once the file is opened.

### Step 1 — Read the existing retry logic

Open `app_connection.c` and locate the function responsible for connection parameter update retries. Identify:
- The function name (likely `AppConnection_connParamUpdateRetry`, `app_conn_param_update`, or similar).
- The current wait mechanism between retries (fixed constant, `os_delay_ms`, RTOS timer, etc.).
- The type of the retry counter and any maximum retry limit.

### Step 2 — Define constants

Add the following constants near the top of the file (or in the associated header):

```c
#define CONN_PARAM_RETRY_INITIAL_MS  500U
#define CONN_PARAM_RETRY_CAP_MS      10000U
```

### Step 3 — Rewrite the retry loop with exponential backoff

Replace the fixed-delay call with a doubling interval, clamped at 10 seconds:

```c
uint32_t retryInterval_ms = CONN_PARAM_RETRY_INITIAL_MS;

for (int attempt = 0; attempt < MAX_CONN_PARAM_RETRIES; attempt++)
{
    if (app_conn_param_update(connHandle, &params) == SUCCESS)
    {
        return SUCCESS;
    }
    os_delay_ms(retryInterval_ms);
    retryInterval_ms *= 2;
    if (retryInterval_ms > CONN_PARAM_RETRY_CAP_MS)
    {
        retryInterval_ms = CONN_PARAM_RETRY_CAP_MS;
    }
}
```

### Step 4 — Verify no other callers are affected

Search `app_connection.c` and related files for any other code that depends on the old fixed-delay behavior and confirm those callers are unaffected by the change.

### Step 5 — Clean up old constants

Remove or replace the old fixed-delay constant (e.g., `CONN_PARAM_UPDATE_DELAY_MS`) with the new initial/cap constants to keep the codebase consistent.
