## Preflight: connection_parameter_update_retry (with exponential backoff)

(no binary — static analysis only)

**Call graph analysis:**

The retry logic in app_connection.c for BLE5Stack projects on CC2340R53 typically calls into TI RTOS timer APIs (e.g., Util_startClock, ClockP_setTimeout, or an equivalent RTOS clock primitive) to schedule the next retry attempt, and then calls GAP_UpdateLinkParamReq (or an equivalent BLE stack API) when the timer fires.

Key call-order flags:
- The retry callback fires in the RTOS task context. Any timer started from within the callback must be started before the callback returns — the timer handle must be valid at the point Util_startClock / ClockP_setTimeout is called. If the clock object is stack-local and the retry callback returns before the timer fires, this is a dangling reference hazard.
- GAP_UpdateLinkParamReq must only be called when a connection handle is still valid. If the link has dropped between retries, calling this with a stale handle produces undefined behavior. The existing code presumably checks this; the new logic must preserve that guard.
- Call ordering: the interval must be computed and clamped before passing it to the timer API, not after. Passing a clamped value then multiplying afterward would break the backoff sequence.

Indirect call flag: BLE5Stack callback dispatch typically uses function pointers registered at init time. If the retry timer callback goes through a function-pointer table, the address must be registered before the first retry is armed. This is an existing property of the code; the new logic does not change it, but the plan should not remove any registration step.

**Arithmetic analysis:**

- Overflow risk (WARNING): The doubling operation `interval *= 2` on a uint32_t will overflow if the current interval exceeds UINT32_MAX / 2 (~2.1 billion ms). Even though the cap is 10,000 ms (well below overflow), the multiply must happen before the cap check, so the intermediate value can still overflow if the initial value is unexpectedly large (e.g., from a misconfigured initial retry interval or a corrupted state variable). Plan to clamp after computing but guard against wrap: use `if (interval > CAP / 2) { interval = CAP; } else { interval *= 2; }` to avoid ever computing the overflowed value.
- Signed/unsigned mix risk (WARNING): TI RTOS clock APIs (e.g., ClockP_setTimeout) typically take a uint32_t tick count, but GAP_UpdateLinkParamReq parameter fields (intervalMin, intervalMax) are uint16_t in units of 1.25 ms. The backoff interval for the retry timer (in ms/ticks) is a different variable from the BLE connection interval parameter. These must not be confused. Verify the types at each call site and cast explicitly.
- Unit mismatch (WARNING): The cap of 10 seconds must be expressed in the same units as the timer API expects. TI RTOS clocks often work in system tick units (e.g., 10 µs per tick on some configurations), not milliseconds. The constant 10000 will mean 10 seconds in ms but only 0.1 seconds in 10 µs ticks. The plan must define the cap as APP_CONN_RETRY_MAX_INTERVAL_MS in ms and convert to ticks explicitly at the call site using the board's tick period.

**Freed-resource analysis:**

- The retry interval state variable (tracking the current backoff level) must be stored in persistent, connection-scoped storage — not on the stack of the retry callback. In BLE5Stack app templates, per-connection state is typically stored in a connRec_t array indexed by connection handle. The backoff interval should be a new field in that struct (e.g., uint32_t retryIntervalMs), or in a parallel array, so it survives across callback invocations.
- When a connection is torn down, the retry timer must be stopped and the backoff interval must be reset to the initial value. If the cleanup path (app_connection.c's disconnect handler) does not stop the clock and reset the field, the next connection on the same index will inherit a stale (possibly maxed-out) backoff interval. Plan to add reset logic to the disconnect handler.
- Double-timer-start risk (WARNING): If GAP_UpdateLinkParamReq is rejected and the callback schedules a retry, but a second rejection arrives before the first timer fires (e.g., a spurious event), Util_startClock may be called on an already-running clock. On some TI RTOS versions, calling Util_startClock on a running clock resets it (acceptable) but on others it is a no-op or logs an error. Confirm behavior and add a guard: only arm the timer if it is not already running (Util_isActive(&retryClockStruct)).

**Execution fit: ADJUST PLAN**

Three adjustments required before writing:
1. Use a pre-multiply overflow guard (`if (interval > CAP_MS / 2) interval = CAP_MS; else interval *= 2;`) instead of multiply-then-clamp.
2. Convert the ms-domain cap to ticks explicitly at the ClockP_setTimeout / Util_startClock call site.
3. Add backoff interval reset to the disconnect handler and an already-running timer guard before arming the retry clock.

## Adjusted Plan: Step-by-Step Edit Sequence

### Step 1 — Add a persistent backoff field to the per-connection record

In app_connection.c (or the associated header that defines connRec_t / appConnRec_t), add a field:

    uint32_t retryIntervalMs;   /* current exponential backoff delay, ms */

Initialize it to the baseline retry interval (e.g., APP_CONN_INITIAL_RETRY_MS) when a connection is established, and reset it to that value in the disconnect/cleanup handler.

### Step 2 — Define constants

Near the top of app_connection.c (or in the companion .h), add:

    #define APP_CONN_INITIAL_RETRY_MS    500u      /* first retry: 500 ms      */
    #define APP_CONN_MAX_RETRY_MS        10000u    /* cap: 10 seconds          */

Use the board's tick-period macro (e.g., (1000u / ClockP_getSystemTickPeriod()) or MS_TO_TICKS(ms)) for the timer call site. Do not hardcode a tick count.

### Step 3 — Add the overflow-safe doubling helper (inline, at the call site)

Where the retry is re-armed (in the parameter update reject handler or the retry callback), replace any fixed-interval timer start with:

    /* Advance backoff — overflow-safe: compare before multiplying */
    if (connRec[connIdx].retryIntervalMs > APP_CONN_MAX_RETRY_MS / 2u) {
        connRec[connIdx].retryIntervalMs = APP_CONN_MAX_RETRY_MS;
    } else {
        connRec[connIdx].retryIntervalMs *= 2u;
    }

    /* Only arm if timer is not already running */
    if (!Util_isActive(&connRec[connIdx].retryClockStruct)) {
        uint32_t ticks = MS_TO_TICKS(connRec[connIdx].retryIntervalMs);
        ClockP_setTimeout(&connRec[connIdx].retryClockStruct, ticks);
        Util_startClock(&connRec[connIdx].retryClockStruct);
    }

Note: the doubling happens at the point of scheduling the next retry, so the first retry fires after APP_CONN_INITIAL_RETRY_MS, the second after 2 * APP_CONN_INITIAL_RETRY_MS, and so on.

### Step 4 — Reset the backoff on disconnect

In the disconnect handler (the function handling GAP_LINK_TERMINATED_EVENT or equivalent):

    connRec[connIdx].retryIntervalMs = APP_CONN_INITIAL_RETRY_MS;
    Util_stopClock(&connRec[connIdx].retryClockStruct);

This prevents stale backoff state from leaking into future connections on the same record slot.

### Step 5 — Verify connection handle validity before each retry

In the retry timer callback, before calling GAP_UpdateLinkParamReq, confirm the connection handle is still valid (i.e., the connection record is in the CONNECTED state). This is likely already present in the existing code; confirm it is not removed or bypassed by the new backoff logic.

Before making any edits, read app_connection.c fully to:

- Confirm the tick-conversion macro or formula in use.
- Confirm the existing guard (if any) on the connection handle before GAP_UpdateLinkParamReq.

Adjust constant names, struct field names, and call sites to match what is actually present in the file.
