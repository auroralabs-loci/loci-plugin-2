- If `Connection_getConnIndex` currently returns an `unsigned` type (e.g., `uint8_t`, `uint16_t`), returning `-1` will wrap to the maximum value of that type (e.g., `255` for `uint8_t`, `65535` for `uint16_t`). This silently becomes a valid-looking but out-of-bounds index.
- If the return type is a signed type (e.g., `int8_t`, `int16_t`, `int`), returning `-1` is safe in terms of representation, but any caller that stores the result in an unsigned variable will silently reinterpret it as a large positive value.
- Callers that compare the result to a maximum connection count (e.g., `if (idx < MAX_NUM_BLE_CONNS)`) may behave correctly after the change — but only if the comparison uses the right signedness.

**Arithmetic: ⚠ RISK — if the current return type is unsigned, returning -1 will wrap. The return type must be changed to a signed integer type (e.g., int8_t or int) as part of this modification.**
