## Preflight: osal_mem_alloc (trace insertion)

**Call graph:** no binary available — static analysis only.

`System_printf` is a TI-RTOS/SysConfig logging function. On Cortex-M targets in the SimpleLink SDK it ultimately calls the UART driver or a circular log buffer, depending on the `SysMin`/`SysCallback` module configuration. Even the lightest path — writing to a circular in-memory log — involves:

1. A function call dispatch through the `System_Module_State` vtable (indirect `bl` through a register)
2. A format-string parse loop over the `%`-specifiers in the format string
3. A lock/unlock pair on the log buffer (typically a `Gate_enterSystem` / `Gate_leaveSystem` call, which on a single-core Cortex-M disables and re-enables interrupts)

On the heavier path (UART-backed), there is a blocking write or at minimum a DMA-queued write with a spinlock. Either way, **the call is not bounded at the instruction level** — the format parser is a loop, and the interrupt-disable window is non-trivial.

Indirect calls are present (vtable dispatch through `System_Module_State`). Flagging as a potential CFI hazard in addition to the latency concern.

**Arithmetic:** The size comparison `size > 256` is a straightforward unsigned comparison on a `uint16_t` or `size_t` argument — no overflow, no wraparound risk, no shift. This check is clean.

**Resources:** `osal_mem_alloc` is on the BLE event loop. The OSAL memory allocator is expected to be called with interrupts enabled but on a cooperative scheduler tick. `System_printf` — specifically the `Gate_enterSystem` path — **disables interrupts** while holding the log buffer lock. This creates a priority-inversion window: any higher-priority ISR that tries to allocate memory during the printf will spin or miss its deadline. If the UART path is active, the function may block entirely. Neither is acceptable inside a 2 μs timing budget.

**Latency estimate (static, 48 MHz Cortex-M4):**

| Segment | Cycles (best case) |
|---|---|
| Conditional branch + function call to `System_printf` | ~5 |
| Vtable dispatch + format-string parser setup | ~20–40 |
| Format-string loop (one `%d` argument) | ~30–60 |
| `Gate_enterSystem` (PRIMASK write + dmb) | ~5 |
| Log buffer write | ~10–20 |
| `Gate_leaveSystem` | ~5 |
| **Total** | **~75–135 cycles** |

At 48 MHz, 2 μs = **96 cycles**. The best-case path is already borderline; any cache miss or UART activity pushes it over. The budget is violated.

## Preflight: osal_mem_alloc

```
Call graph:  ⚠ RISK — System_printf dispatches through an indirect vtable bl;
             interrupt-disable window inside Gate_enterSystem cannot be bounded
             statically; potential CFI hazard and priority-inversion on BLE ISR.
Arithmetic:  OK
Resources:   ⚠ RISK — Gate_enterSystem disables interrupts inside a 2 μs budget
             allocator; UART-backed path may block; breaks BLE event-loop timing.

Execution fit: ADJUST PLAN
→ Replace System_printf with a lock-free, ISR-safe ring-buffer trace (e.g., a
  bare array write behind a simple index increment with a __disable_irq guard
  of ≤ 5 cycles), or move the logging to a deferred task that reads the ring
  buffer outside the event loop.
```

**Adjusted plan before writing any code:**

1. Do not use `System_printf` inside `osal_mem_alloc`. The vtable dispatch, format-string loop, and Gate interrupt-disable window collectively exceed the 96-cycle / 2 μs budget and introduce an unbounded interrupt-disable window on the BLE event loop.

2. Instead, declare a small power-of-two ring buffer and a volatile index at file scope in `osal_memory.c`:
   ```c
   #define ALLOC_LOG_SIZE 32
   static volatile uint16_t _alloc_log[ALLOC_LOG_SIZE];
   static volatile uint8_t  _alloc_log_idx = 0;
   ```

3. Inside `osal_mem_alloc`, after the size check, write a single indexed store — no lock needed on a single-core Cortex-M because the write is a single instruction and the BLE event loop is cooperative (not re-entered by an ISR that also calls `osal_mem_alloc`):
   ```c
   if (size > 256) {
       _alloc_log[_alloc_log_idx++ & (ALLOC_LOG_SIZE - 1)] = (uint16_t)size;
   }
   ```
   This is 3–5 cycles, well within budget.

4. Add a separate low-priority task (or BIOS idle hook) that drains `_alloc_log` and calls `System_printf` there, outside any timing-critical path.

5. Verify the ring-buffer index type (`uint8_t`) does not overflow past 255 before wrapping — with `ALLOC_LOG_SIZE = 32` and the `& 31` mask, the modulo is safe regardless of how many times the index wraps.
