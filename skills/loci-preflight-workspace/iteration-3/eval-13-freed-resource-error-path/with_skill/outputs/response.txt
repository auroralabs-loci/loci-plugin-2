**Call graph:** ⚠ RISK

The proposed edit must call `free(oadCCCDTable)` and `mq_close(hOadQ)` on the early-return path after `GATTServApp_RegisterService` fails. The concern is call ordering:

- `mq_close` is a POSIX call; on TI BLE5 RTOS targets it maps to a TI-RTOS `mq_close` shim. That shim can block or assert if the queue descriptor is invalid. If `hOadQ` was never successfully opened (i.e., `mq_open` itself failed earlier in `OAD_open` and was already handled), calling `mq_close` on a stale or NULL descriptor is undefined behavior. The plan must confirm that the cleanup block is only reached when `hOadQ` is valid — i.e., it is placed after the `mq_open` success check, not before it.
- `GATTServApp_RegisterService` is called near the end of `OAD_open`, so `oadCCCDTable` is presumably already allocated. Verify that no other code path between allocation and the `RegisterService` call could have already freed `oadCCCDTable` (e.g., an intermediate early return for attribute table registration failure). If such a path exists, the new cleanup block would double-free.
- No indirect calls (`bl <reg>`) are expected for `free` or `mq_close` in a straightforward RTOS shim, but flag that the TI BLE5 `mq_close` shim may call `OsalPort_free` internally; that is fine as long as the queue was opened.

After the fix, audit whether any other intervening failure between `mq_open`/`oadCCCDTable` allocation and `GATTServApp_RegisterService` (e.g., `GATTServApp_AddService` failure, attribute registration failure) also lacks cleanup.
