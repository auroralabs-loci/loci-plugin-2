## Preflight: OAD_processQueue (watchdog reset on stall)

### Call graph (no binary — static analysis only)

The file lives at `$LOCI_TEST_BLE_ROOT/simplelink-lowpower-f3-sdk/source/ti/ble5stack_flash/profiles/oad/cc26xx/oad.c`. No compiled object is available locally, so analysis is based on source-level reasoning.

Callees the new stall-detection code would invoke:

- **Clock_getTicks() / Timestamp_get32()** — TI-RTOS system tick read. Typically an inline or a very short leaf; no CFG concern. However, the tick unit must be confirmed: TI-RTOS `Clock_getTicks()` returns ticks in units of `Clock_tickPeriod` microseconds (commonly 10 µs or 1 ms depending on the kernel config). Using the wrong unit would make the 30-second threshold wildly wrong.
- **SysCtrlSystemReset() / HAL_SYSTEM_RESET()** — the TI-RTOS / DRIVERLIB watchdog reset call. This is a noreturn peripheral write. No return value to check, but it must not be called while holding a mutex or inside a critical section — any held lock will never be released and may corrupt state on the next boot if state is preserved in retained RAM.
- **Queue_get() / OAD queue access** — if the stall timer is checked every time `OAD_processQueue` polls and the queue is empty, "stall" and "idle" must be distinguished. Calling reset when the queue is simply empty (i.e., no OAD session active) would incorrectly reset a healthy device.

CFG concerns:
- **Call-order assumption (RISK)**: the stall timer must only arm after an OAD session has begun. If it arms at module init, it fires 30 s after boot regardless of OAD activity. Confirm that the timer is started only on the first enqueue (or OAD_open), not in `OAD_processQueue` itself on every call.
- **Dead path (RISK)**: if `OAD_processQueue` is called from a Task with a `pending` / `pend` on a semaphore, the 30-second wall-clock stall may never be observed by this function — the task would simply be blocked and the function never entered. The watchdog logic must live at or above the pend site (e.g., a separate watchdog task or a hardware watchdog timer), not inside the body that only runs when work arrives.

### Arithmetic ranges

Two expressions need checking:

1. **Elapsed time: `currentTick - stallStartTick`**
   - Both are `uint32_t` (TI-RTOS tick counters). Unsigned subtraction wraps correctly modulo 2^32 even across a single rollover — no guard needed for the subtraction itself.
   - Wraparound budget: at a 10 µs tick period, `UINT32_MAX * 10 µs ≈ 42 949 s ≈ 11.9 hours`. A single rollover is safe for a 30-second window. However, if the tick period is 1 ms, the budget is `~49.7 days` — still safe. No overflow risk for the elapsed calculation.
   - **RISK**: converting ticks to seconds — `elapsed / TICKS_PER_SECOND` — requires knowing `Clock_tickPeriod`. If this constant is hardcoded rather than read from `ClockP_getSystemTickPeriod()` (or the equivalent), the threshold drifts when the kernel is reconfigured. Plan to derive `TICKS_PER_30S` from the runtime period, or add a `static_assert` that the assumed period matches the build-time constant.

2. **Threshold constant: `30 * TICKS_PER_SECOND`**
   - If `TICKS_PER_SECOND = 1 000 000 / tick_period_us = 100 000` (at 10 µs), then `30 * 100 000 = 3 000 000`, well within `uint32_t`. No overflow.
   - If tick period is 1 µs (some high-res configs), `30 * 1 000 000 = 30 000 000` — still within `uint32_t`. Safe.

3. **Comparison sign**: if either operand is cast to `int32_t` for the comparison (a common copy-paste pattern), a 30-second elapsed time of `3 000 000` ticks is positive and safe, but a rollover producing a large unsigned value could appear negative. Keep both operands `uint32_t`.

### Freed-resource access

- **Queue entry ownership**: `OAD_processQueue` typically dequeues a message pointer, processes it, and frees it. If the watchdog reset fires while a message pointer has been dequeued but not yet freed, that allocation is leaked — harmless on reset but worth noting for retained-RAM designs where the heap survives reset.
- **Stall-timer state variable**: `stallStartTick` must be a persistent variable (static local or module-level). If it is stack-local, it is re-initialized every call, making elapsed time always zero — the reset never fires.
- **Critical section gap**: if `OAD_processQueue` disables interrupts or holds a POSIX mutex around queue access, the stall timer check must happen outside that critical section. Calling `SysCtrlSystemReset()` inside a critical section that was protecting shared OAD state is safe for the reset itself (the chip resets) but is architecturally sloppy and complicates static analysis tools.
- **No double-free risk** from the watchdog path itself — `SysCtrlSystemReset()` is noreturn, so no subsequent free of a pointer occurs.

### Adjusted plan (before writing any code)

1. **Decide the watchdog site.** If the intent is purely "reset if OAD stalls for 30 s", the right location is a TI-RTOS watchdog timer callback (or a dedicated monitor Task with a 30 s timeout on `Semaphore_pend`), not inside `OAD_processQueue` itself. Confirm with the owner which approach is required.

2. **Add a session-active guard.** Arm the stall timer only when `OAD_open` / first OAD block is received. Clear it on `OAD_close` / completion. This prevents spurious resets during normal (non-OAD) operation.

3. **Derive tick threshold at init.** Call `ClockP_getSystemTickPeriod()` once at OAD init and compute `stallTicks = 30 000 000UL / tickPeriodUs`. Store as a `static uint32_t`.

4. **Use `uint32_t` throughout.** `stallStartTick`, `currentTick`, `elapsed`, and `stallTicks` all declared `uint32_t`. No signed casts.

5. **Check outside critical sections.** Place the elapsed check before any `Hwi_disable()` / mutex acquire call in `OAD_processQueue`.

6. **Add a comment marking the noreturn path.** Before `SysCtrlSystemReset()`, add `/* noreturn */` and release any resources that can be cheaply released (e.g., clear OAD state flags) so retained-RAM state is clean on the next boot.
