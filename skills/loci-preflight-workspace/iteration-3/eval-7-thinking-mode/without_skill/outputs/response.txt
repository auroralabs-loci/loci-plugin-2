I'll think through how to add a watchdog reset inside `OAD_processQueue` if the queue has been stalled for more than 30 seconds.

The file is referenced via $LOCI_TEST_BLE_ROOT, which is an external environment variable pointing outside the current workspace. The analysis below is based on the OAD (Over-the-Air Download) stack architecture for the TI SimpleLink F3 SDK.

---

## Problem Analysis

`OAD_processQueue` is the BLE OAD profile's event/message processing loop. It dequeues and handles OAD control messages (block requests, image notifications, etc.). A "stall" means the queue has stopped being serviced — no new messages are dequeued and processed — for an extended period (30 seconds in this case). This can happen due to:

- A lost BLE connection that was not properly cleaned up
- A peer device that stopped sending OAD block requests mid-transfer
- A task deadlock or missed event notification
- Buffer exhaustion causing the RTOS task to never be unblocked

The watchdog reset is a last-resort recovery mechanism: if the queue has been stuck for 30 seconds, the device hard-resets to recover to a known good state.

---

## Design Approach

### 1. Track the "last activity" timestamp

The stall is measured by comparing the current time to the last time a message was successfully dequeued and processed. Two approaches are viable:

**Option A — Wall-clock timestamp inside `OAD_processQueue`:**
Declare a static variable to hold the timestamp of the last successful dequeue. Each time a message is pulled from the queue, update it. On each call (or on each "empty queue" poll), compare the current time to the stored timestamp.

**Option B — A periodic RTOS timer callback:**
Set up a one-shot or repeating TI-RTOS `Clock` object. Reset the clock every time a message is processed. If the clock fires, that means 30 seconds elapsed without activity — trigger the reset from the clock callback.

Option B is cleaner because `OAD_processQueue` is only called when events are posted to the OAD task; if the task itself is never woken, the check inside `OAD_processQueue` would never run. A separate timer/clock fires independently, which correctly detects a stalled *task*, not just a stalled *function body*.

Option A works only if `OAD_processQueue` is called on a polling basis or if a timeout is applied to the semaphore/event pend that triggers it.

Given the TI BLE5 stack RTOS architecture, **Option B (a TI-RTOS `Clock` object)** is the correct choice.

### 2. Implementation Plan

#### A. Add necessary includes (if not already present)

```c
#include <ti/sysbios/knl/Clock.h>
#include <ti/drivers/Watchdog.h>
/* or, for a soft reset without a hardware WDT: */
#include DeviceFamily_constructPath(driverlib/sys_ctrl.h)  // SysCtrlSystemReset()
```

The simplest reset mechanism on CC26xx without a pre-initialized hardware watchdog is `SysCtrlSystemReset()`. If the application already opens a `Watchdog` driver handle, the WDT can be starved (stop calling `Watchdog_clear`) so it naturally fires, or `Watchdog_close` can be called to trigger a reset — but `SysCtrlSystemReset()` is more direct.

#### B. Add module-level statics

```c
static Clock_Struct  oadWatchdogClk;
static bool          oadTransferActive = false;
```

#### C. Write the clock callback

```c
static void OAD_watchdogClkCB(UArg arg)
{
    // 30 seconds elapsed with no queue activity — force reset
    SysCtrlSystemReset();
    // Execution never reaches here
}
```

#### D. Initialize the clock when an OAD transfer begins

In the function that handles the OAD Image Identify (start-of-transfer) event, construct and start the clock:

```c
// 30 seconds = 30,000 ms; Clock tick period is typically 10 µs on CC26xx,
// so 30 s = 3,000,000 ticks. Verify Clock_tickPeriod at runtime or use
// the macro: (30000 * 1000) / Clock_tickPeriod
uint32_t timeoutTicks = (uint32_t)(30000000UL / Clock_tickPeriod);

Clock_construct(&oadWatchdogClk, OAD_watchdogClkCB,
                timeoutTicks,
                NULL);          // one-shot; params default = one-shot
Clock_start(Clock_handle(&oadWatchdogClk));
oadTransferActive = true;
```

#### E. Reset (kick) the watchdog inside `OAD_processQueue` on each successful dequeue

```c
void OAD_processQueue(void)
{
    oadEvt_t *pMsg;

    while ((pMsg = (oadEvt_t *)Util_dequeueMsg(&oadQ)) != NULL)
    {
        // --- Kick the stall watchdog ---
        if (oadTransferActive && Clock_isActive(Clock_handle(&oadWatchdogClk)))
        {
            Clock_stop(Clock_handle(&oadWatchdogClk));
            Clock_start(Clock_handle(&oadWatchdogClk));  // restart from zero
        }

        // Process the message
        OAD_processOADEvt(pMsg);

        // Free the message
        ICall_free(pMsg);
    }
}
```

#### F. Disarm and destruct the clock when the transfer ends (success or abort)

```c
static void OAD_endTransfer(void)
{
    if (oadTransferActive)
    {
        Clock_stop(Clock_handle(&oadWatchdogClk));
        Clock_destruct(&oadWatchdogClk);
        oadTransferActive = false;
    }
}
```

Call `OAD_endTransfer()` from the OAD completion handler and from any error/abort path (connection drop, image validation failure, etc.).

---

### 3. Key Correctness Considerations

**Clock tick period:** On CC26xx with TI-RTOS, `Clock_tickPeriod` is configured in the application's `.cfg` file (commonly 10 µs). Do not hard-code a tick count; divide by `Clock_tickPeriod` (in microseconds) at runtime or at compile time using a `#define`.

**One-shot vs. repeating:** Use a one-shot clock (default `Clock_Params`). A repeating clock would reset the device every 30 seconds unconditionally even if the transfer is progressing, which is wrong. The one-shot is re-armed on each dequeue, so as long as blocks arrive within 30 seconds of each other, the timer never fires.

**Transfer scope:** The watchdog should only be active during an OAD transfer. Arming it at device boot would cause spurious resets whenever no OAD is in progress. Gate the arm/disarm on the transfer lifecycle (Image Identify -> start; OAD complete / abort / connection drop -> stop).

**RTOS context safety:** `Clock_stop` and `Clock_start` are safe to call from task context (which `OAD_processQueue` runs in). The callback fires in Swi context; `SysCtrlSystemReset()` is safe there.

**No double-destruct:** The `oadTransferActive` flag guards against destructing a clock that was never constructed.

**Hardware WDT alternative:** If the application already uses `Watchdog_open`, the preferred approach is to call `Watchdog_clear(handle)` each time a block is processed, and *stop calling it* if the watchdog detects a stall — but since the WDT runs autonomously, you simply don't kick it and let it reset the device. This avoids adding a second timer.

---

### 4. Summary of Changes

| Location | Change |
|---|---|
| Module-level | Add `Clock_Struct oadWatchdogClk` and `bool oadTransferActive` statics |
| OAD Image Identify handler | Construct + start the 30-second one-shot clock |
| `OAD_processQueue` | Stop + restart the clock on each successful dequeue |
| OAD complete / abort / disconnect handler | Stop + destruct the clock, clear `oadTransferActive` |
| New static function | `OAD_watchdogClkCB` calls `SysCtrlSystemReset()` |

This design correctly handles the case where `OAD_processQueue` is never called (because the RTOS task is never posted an event), because the TI-RTOS `Clock` fires from Swi context independently of the OAD task's scheduling.
