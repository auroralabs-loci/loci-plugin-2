## Plan: Fix unsigned subtraction wraparound in `osal_mem_free`

### Context

The bug is in `osal_mem_free` in `osal_memory.c`. When computing the size of the block being freed, the code performs an unsigned subtraction of the header size from the reported block size. If the header size exceeds the reported size (corrupted heap metadata, or misaligned pointer), the result wraps to a very large unsigned value, causing the allocator to treat garbage memory as a huge free region.

**Arithmetic:** The defect is exactly an unsigned wraparound:

```c
// Existing (buggy) pattern — pseudocode
uint16_t freeSize = block->size - OSAL_MEM_HDR_SIZE;
```

- `block->size` and `OSAL_MEM_HDR_SIZE` are both unsigned (commonly `uint16_t` or `size_t`).
- If `block->size < OSAL_MEM_HDR_SIZE` (corrupted or misaligned header), the subtraction wraps to a large positive value (e.g., `0 - 4` → `0xFFFC` for `uint16_t`).
- The wrapped value is then used to update the free-list, allowing the allocator to claim ownership of memory it does not own.
- Fix: guard the subtraction with an explicit size check before performing it. If the invariant is violated, treat the block as corrupted and return without modifying the free list (or assert/panic in debug builds).

Arithmetic: ⚠ RISK — unsigned subtraction with no underflow guard; the fix must add the guard before the subtraction.

---

**Resources:** `osal_mem_free` inserts the block back into the free list. If the function returns early (via the new guard) without inserting, the block is effectively leaked for the current session — but that is the correct safe behaviour when the metadata is corrupt. No double-free risk is introduced by the guard itself. No pointers to locals are returned. Critical-section macros must still be exited on every return path, including the new early-return path introduced by the guard.

Resources: ⚠ RISK — any early return added by the guard must ensure `HAL_EXIT_CRITICAL_SECTION` (or equivalent) is called before returning. A missing exit from the critical section would deadlock the system.

### Adjusted plan (ordered steps)

1. **Read the file** — open `osal_memory.c` and locate `osal_mem_free`. Identify:
   - The exact unsigned subtraction expression.
   - Where `HAL_ENTER_CRITICAL_SECTION` is called (note the variable name used, e.g., `intState`).
   - All existing return points to understand the critical-section exit pattern.

2. **Add the underflow guard before the subtraction** — immediately before the line that computes free size, insert:

   ```c
   if ( block->size < OSAL_MEM_HDR_SIZE )
   {
     /* Corrupted block header: size field is smaller than the header itself.
        Performing the subtraction would wrap to a huge unsigned value and
        corrupt the free list.  Exit the critical section and return safely. */
     HAL_EXIT_CRITICAL_SECTION( intState );
     return;
   }
   ```

   The guard must be placed *after* `HAL_ENTER_CRITICAL_SECTION` so the exit call is paired correctly.

3. **Verify all exit paths** — after adding the guard, trace every `return` in the function and confirm each one has a preceding `HAL_EXIT_CRITICAL_SECTION` call (or is before the `ENTER` call).

4. **Add a debug assertion** (optional but recommended) — in debug builds, trigger a stronger diagnostic:

   ```c
   OSAL_ASSERT( block->size >= OSAL_MEM_HDR_SIZE );
   ```

   Place this before the guard so debug builds surface the corruption loudly while release builds fail safely.

5. **Review the caller** — check whether the caller (`osal_mem_free`'s call sites) validates the pointer before calling. If not, note that the guard only prevents heap corruption; the root cause (whoever produced the bad block) still needs to be found.

6. **Test** — verify with a unit test that passes a block with `block->size < OSAL_MEM_HDR_SIZE` and confirms the function returns without modifying the free list.
