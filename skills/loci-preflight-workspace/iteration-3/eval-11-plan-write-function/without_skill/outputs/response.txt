**Preflight Report**

Before writing any code, the following facts were confirmed by reading the actual source file at
`$LOCI_TEST_BLE_ROOT/simplelink-lowpower-f3-sdk/source/ti/ble5stack_flash/hci/cc26xx/hci.c`
and related headers.

Key findings:

1. `HCI_SendDataPkt` signature (hci.c line 1323):
   ```c
   hciStatus_t HCI_SendDataPkt( uint16  connHandle,
                                uint8   pbFlag,
                                uint16  pktLen,
                                uint8  *pData )
   ```
   The function already rejects handles where `(connHandle & 0xFF00) != 0` (lines 1336-1342),
   enforcing an 8-bit limit with `HCI_ERROR_CODE_INVALID_HCI_CMD_PARAMS`. This is consistent
   with the comment "The TI LE only supports an eight bit connection handle."

2. Valid handle range: The TI BLE5 stack uses connection handles as zero-based indices into a
   connection table of size `maxNumConns` (declared `extern uint8 maxNumConns` in ll_config.h).
   A handle is in-range only when `connHandle < maxNumConns`. Special sentinel values exist:
   - `LL_CONNHANDLE_ALL     0xFFFD`
   - `LL_CONNHANDLE_LOOPBACK 0xFFFE`
   - `LL_CONNHANDLE_INVALID  0xFFFF`
   These must not be passed to `HCI_SendDataPkt`.

3. Return type: `hciStatus_t` (a `uint8`). The correct error code for a bad handle is
   `HCI_ERROR_CODE_INVALID_HCI_CMD_PARAMS` (0x12), consistent with how the existing function
   handles other parameter errors.

4. The guard is wrapped in `#if defined(CTRL_CONFIG) && (CTRL_CONFIG & (ADV_CONN_CFG | INIT_CFG))`
   — the new validator must live inside or respect the same preprocessor guard.

**Arithmetic check:** A valid handle `h` satisfies `h < maxNumConns`. Equivalently, since
`maxNumConns` is at most 8 on this platform, a handle that passes the existing 8-bit check
(`h <= 0x00FF`) but is still >= `maxNumConns` is out-of-range and must be rejected.

**Plan: Add `HCI_ValidateAndSendDataPkt` to hci.c**

**Step 1 — Read before touching anything**

Read `hci.c` around the `HCI_SendDataPkt` definition (lines ~1313-1355) to confirm the exact
preprocessor guard, the existing parameter checks, and the `maxNumConns` extern is reachable

**Step 2 — Add the validation wrapper**

Insert the following function immediately before `HCI_SendDataPkt` (inside the same
`#if defined(CTRL_CONFIG) && (CTRL_CONFIG & (ADV_CONN_CFG | INIT_CFG))` block):

```c
/*******************************************************************************
 * Validates a BLE ACL connection handle, then forwards to HCI_SendDataPkt.
 *
 * Returns HCI_ERROR_CODE_INVALID_HCI_CMD_PARAMS if connHandle is out of the
 * range [0, maxNumConns), otherwise returns the status from HCI_SendDataPkt.
 */
hciStatus_t HCI_ValidateAndSendDataPkt( uint16  connHandle,
                                        uint8   pbFlag,
                                        uint16  pktLen,
                                        uint8  *pData )
{
  /* Arithmetic range check:
   *   Valid handles are zero-based indices: 0 <= connHandle < maxNumConns.
   *   maxNumConns is a uint8, so the comparison is unsigned-safe.
   *   Sentinel values (0xFFFD, 0xFFFE, 0xFFFF) and any handle >= maxNumConns
   *   are rejected here before reaching HCI_SendDataPkt.
   */
  if ( connHandle >= (uint16)maxNumConns )
  {
    return HCI_ERROR_CODE_INVALID_HCI_CMD_PARAMS;
  }

  return HCI_SendDataPkt( connHandle, pbFlag, pktLen, pData );
}
```

**Why not modify `HCI_SendDataPkt` directly?**

`HCI_SendDataPkt` is a standard HCI interface whose behavior is documented and depended on by
L2CAP (the file comment explicitly warns: "L2CAP is affected by this routine's status values").
Adding the validation in a separate wrapper avoids breaking that contract and keeps the change
reviewable in isolation.
