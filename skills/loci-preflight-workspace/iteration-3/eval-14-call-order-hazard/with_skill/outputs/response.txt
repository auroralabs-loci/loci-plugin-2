## Preflight: OAD_open (call-order hazard analysis)

### Task

Insert `GAPBondMgr_FindAddr` as the first check in `OAD_open` in `oad.c`, so that OAD rejects image transfers from unbonded devices before any setup proceeds.

### 1. Call graph (CFG analysis)

No binary is available locally (the BLE project files do not exist). Proceeding with static analysis only.

**Callee: `GAPBondMgr_FindAddr`**

`GAPBondMgr_FindAddr` is part of the TI BLE5-Stack GAP Bond Manager. Its signature is typically:

```c
bStatus_t GAPBondMgr_FindAddr(uint8_t *pDevAddr,
                               GAP_Peer_Addr_Types_t addrType,
                               uint8_t *pIdx,
                               GAP_Bonded_Status_t *pBondedStatus,
                               uint16_t *pConnectionHandle);
```

It queries the SNV (Simple Non-Volatile storage) bond table. There are several call-ordering hazards here:

- **GAP Bond Manager initialization dependency**: `GAPBondMgr_FindAddr` reads from the bond table which is populated only after `GAPBondMgr_Register` has been called and the GAP Bond Manager has been initialized. In a typical BLE5 stack initialization sequence, `GAPBondMgr_Register` is called during application startup. If `OAD_open` is called before the BLE stack is fully initialized (e.g., during boot or prior to `GAP_DeviceInit` completing), the bond table may be uninitialized or stale, causing `GAPBondMgr_FindAddr` to return false negatives (incorrectly reporting a bonded device as unbonded).

  **Flag**: Call-order assumption — the plan must verify that `OAD_open` is only ever invoked after `GAPBondMgr_Register` and GAP initialization have completed.

- **Connection handle / address availability**: `OAD_open` is called in the context of an OAD service write (GATT). At the moment `OAD_open` is invoked, the `connHandle` should be valid and the peer address resolvable via `linkDB_GetInfo` or similar. However, `GAPBondMgr_FindAddr` requires the **resolved** (identity) address, not the current RPA (Resolvable Private Address). If the peer is using privacy (RPA), passing the current peer address directly into `GAPBondMgr_FindAddr` without first resolving it through `GAP_ResolvePrivateAddr` or relying on the stack's internal resolution will yield a miss on the bond table, causing bonded-but-private-address devices to be rejected incorrectly.

  **Flag**: ⚠ RISK — address type/resolution mismatch. The plan must account for RPA resolution before calling `GAPBondMgr_FindAddr`.

- **`GAPBondMgr_FindAddr` return value semantics**: The function returns `SUCCESS` (0) if the address is found in the bond table, `bleGAPNotFound` otherwise. The new guard must treat anything other than `SUCCESS` as a rejection. There is no known indirect call or recursion hazard in this callee.

- **No binary — static analysis only.**

### 2. Arithmetic ranges

No arithmetic is introduced by inserting this guard. The return value of `GAPBondMgr_FindAddr` is a `bStatus_t` (typedef of `uint8_t`). Comparing it to `SUCCESS` (defined as 0x00) is a straightforward unsigned comparison — no overflow or wraparound risk.

The only index involved is the bond index (`pIdx`, `uint8_t`), which is an output parameter written by the callee; the new code does not perform arithmetic on it.

**No arithmetic risks.**

### 3. Freed-resource access

`OAD_open` typically allocates OAD control-block state (e.g., sets up an `oadControlBlock_t`, opens an image header slot). By placing the `GAPBondMgr_FindAddr` check **before any allocation or setup**, the early-return path exits before any resource is acquired — so there is no resource that needs to be freed on the rejection path.

**No freed-resource risk**, provided the check is genuinely the very first substantive statement in `OAD_open` (before any `malloc`, OSAL memory alloc, or NV open call).

### Adjusted plan

1. **Verify initialization order** — confirm in the application's `SimplePeripheral_init` (or equivalent) that `GAPBondMgr_Register` is called before any GATT service that could trigger `OAD_open`. If not, add a `BLE_ASSERT` or runtime guard (`oadCB != NULL` check) to make the ordering explicit.

2. **Resolve the peer address** — before calling `GAPBondMgr_FindAddr`, retrieve the identity address for the connection. The TI stack typically stores this in `linkDB`; use `linkDB_GetInfo(connHandle, &linkInfo)` and prefer `linkInfo.addrType` / `linkInfo.addr` (identity address fields) over the raw advertisement address.

3. **Insert the guard** — as the first check in `OAD_open`:

   ```c
   bStatus_t OAD_open(uint16_t connHandle)
   {
       /* Preflight: reject unbonded devices before any OAD setup. */
       linkDBInfo_t linkInfo;
       if (linkDB_GetInfo(connHandle, &linkInfo) != SUCCESS)
       {
           return OAD_NO_RESOURCES;   /* or an appropriate rejection status */
       }

       uint8_t bondIdx;
       bStatus_t bondStatus = GAPBondMgr_FindAddr(linkInfo.addr,
                                                   linkInfo.addrType,
                                                   &bondIdx,
                                                   NULL,
                                                   NULL);
       if (bondStatus != SUCCESS)
       {
           return OAD_BONDING_REQUIRED;  /* device is not bonded — reject */
       }

       /* ... rest of existing OAD_open setup ... */
   }
   ```

   Note: `OAD_BONDING_REQUIRED` may need to be defined if it does not already exist in `oad.h`.

4. **Declare `OAD_BONDING_REQUIRED`** — add a status code to `oad.h` if one does not already exist for this rejection case, so callers can distinguish "not bonded" from other OAD errors.

5. **Test the rejection path** — exercise with an unbonded connection to confirm OAD is rejected before `oadControlBlock` is touched, and with a bonded RPA device to confirm it is not incorrectly rejected.
