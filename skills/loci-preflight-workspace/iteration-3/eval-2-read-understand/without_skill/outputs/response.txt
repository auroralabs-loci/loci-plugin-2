## GAPBondMgr_FindAddr

### What it does

`GAPBondMgr_FindAddr` searches the BLE bond manager's bonding table for a peer device by address. It has two distinct search modes depending on the address type passed in:

1. **Direct address match**: It iterates over every entry in the `bonds[]` array (up to `gapBond_maxBonds` entries) and checks whether the stored address and address type match `pDevAddr` and `addrType` exactly. If a match is found, the bond table index is written to `*pIdx` and `SUCCESS` is returned.

2. **Resolvable Private Address (RPA) resolution**: If `addrType` is `PEER_ADDRTYPE_RANDOM_OR_RANDOM_ID` and the address bytes indicate a Random Private Resolvable (RPR) address, the function additionally attempts to resolve the RPA against the Identity Resolving Key (IRK) stored in non-volatile memory for each bond entry. It reads the IRK from NV (`DEV_IRK_NV_ID(idx)`), reverses its byte order, and calls `LL_PRIV_ResolveRPA` to test whether the RPA was generated from that IRK. If resolution succeeds, it writes the bond index, the identity address type, and the identity address (i.e., the real underlying public/static address stored in the bond record) to the respective output parameters and returns `SUCCESS`.

If the address is not found and does not resolve against any stored IRK, the function returns `bleGAPNotFound`.

The function is widely used internally throughout `gapbondmgr.c` to look up whether a peer already has a bond record before pairing, erasing a bond, updating characteristic configuration, handling signature updates, and other bond management operations.

### Return values

- `SUCCESS` — address was found directly or resolved via IRK.
- `INVALIDPARAMETER` — `pDevAddr` is NULL.
- `bleInvalidRange` — `addrType` is outside the valid range of `GAP_Peer_Addr_Types_t`.
- `bleGAPNotFound` — address was not found and did not resolve to any entry in the bonding table.

### Parameters

**`uint8_t *pDevAddr`**
A pointer to the 6-byte peer device address to search for. This is the address the caller wants to look up in the bonding table. It must not be NULL (the function returns `INVALIDPARAMETER` immediately if it is). If `addrType` indicates the address may be a Resolvable Private Address, the function will also attempt IRK-based resolution against this value.

**`GAP_Peer_Addr_Types_t addrType`**
The type of the address passed in `pDevAddr`. This is used both for the direct comparison against stored bond records (the stored `bonds[idx].addrType` must match) and to determine whether RPA resolution should be attempted. If the value is `PEER_ADDRTYPE_RANDOM_OR_RANDOM_ID` and the address bytes look like an RPR address, the RPA resolution path is taken. The function validates that this value falls within the legal range of the `GAP_Peer_Addr_Types_t` enum and returns `bleInvalidRange` if not.

**`uint8_t *pIdx`**
An optional output parameter. If non-NULL and a matching bond entry is found (either by direct match or by RPA resolution), the index of that entry within the `bonds[]` array is written here. This index is subsequently used by callers to read or update NV-backed bond records (e.g., `MAIN_RECORD_NV_ID(idx)`, `LOCAL_LTK_NV_ID(idx)`). Passing NULL is safe and simply suppresses this output.

**`GAP_Peer_Addr_Types_t *pIdentityAddrType`**
An optional output parameter. Only populated when an RPA is successfully resolved (i.e., the direct-match path does not set this). If non-NULL and resolution succeeds, the address type of the identity address stored in the matched bond record (`bonds[idx].addrType`) is written here. This tells the caller the type of the true underlying identity address (public or static random) behind the RPA. Passing NULL is safe.

**`uint8_t *pIdentityAddr`**
An optional output parameter. Only populated when an RPA is successfully resolved. If non-NULL and resolution succeeds, the 6-byte identity address stored in the matched bond record (`bonds[idx].addr`) is copied here. This is the real, stable address of the peer device, as opposed to the rotating RPA that was passed in. Passing NULL is safe.
