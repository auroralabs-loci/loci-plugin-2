## Preflight: GAPBondMgr_FindAddr (extended variant)

*(no binary — static analysis only)*

**Understanding the existing function:**

`GAPBondMgr_FindAddr` in `gapbondmgr.c` is a lookup function that searches the bond table for an entry matching a given address. It returns a status or boolean indicating whether the address was found. The new variant needs to also surface the bond index via an output parameter (e.g., `uint8_t *pBondIdx`).

**Call graph analysis:**

The function internally iterates over bond records, calling storage or NV access helpers such as `gapBondMgrReadBondRec()` or equivalent NV read wrappers to load each bond entry and compare addresses. Key callees to consider:

- Bond table iteration: typically a `for` loop from `0` to `GAP_BONDINGS_MAX` calling an NV read helper per iteration.
- Address comparison: likely `MAP_osal_memcmp()` or similar on the stored address vs. the input address.
- No heap allocation is expected in the hot path — bond data is read from NV into a stack-local struct.

Hazards found (static reasoning):
- **Call-order assumption**: The bond manager must be initialized (`GAPBondMgr_Register` / NV init) before any find operation. There is no visible guard enforced at the call site in source — this assumption is implicit. The new function inherits this assumption. Flag: if called before initialization, NV reads may return stale or garbage data, and the output index would be written with an invalid value.
- **Indirect calls**: NV access wrappers on TI BLE5 stacks route through a function pointer table (`MAP_` prefix macros). These are indirect calls. Not a CFI hazard in normal operation but worth noting for tracing.
- No recursion or cycles expected in the callee graph.

**Arithmetic ranges:**

- Bond index ranges from `0` to `GAP_BONDINGS_MAX - 1` (typically `0..7` or `0..15` depending on configuration). The loop counter fits in `uint8_t`.
- Risk: if `GAP_BONDINGS_MAX` is `0` (misconfigured build), the loop body never executes and the output parameter is never written. The caller would read an uninitialized value.
- The output parameter `pBondIdx` is a pointer — if the caller passes `NULL`, a blind write `*pBondIdx = idx` will fault.
- No signed/unsigned mixing expected in the loop, but the return value (status) and the index (uint8_t) must remain clearly separated — mixing them into a single return would silently drop range information.

**Freed-resource access:**

- Bond record is typically read into a stack-local struct. No heap allocation → no free path → no use-after-free or double-free risk in the callee.
- The output parameter `pBondIdx` must not be written after an early-return error path where the address is not found — the value at `*pBondIdx` should be defined only when the return value signals success. Leaving it undefined on failure is acceptable but must be documented; writing a sentinel (e.g., `GAP_BONDINGS_MAX` or `0xFF`) on failure is safer.
- No RAII gap — no resources acquired mid-function in the expected implementation.
