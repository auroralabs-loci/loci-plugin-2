## Preflight: HCI_SendDataPkt (retry loop)

**Understanding the task:** Implement a retry loop inside `HCI_SendDataPkt` in `hci.c` (TI BLE5 stack, CC26xx) that retries up to 3 times if the connection handle is invalid before returning an error.

---

### CFG Analysis

No binary is available locally (files don't exist in this environment), so this is static analysis only — **(no binary — static analysis only)**.

Reasoning from function name and context:

`HCI_SendDataPkt` is a standard HCI data-packet transmission function in the TI BLE5 stack. It typically:
1. Validates the connection handle via a lookup — something like `LL_ConnHandleToIdx(connHandle)` or `hciConnLookup(connHandle)` — which returns `HCI_ERROR_CODE_UNKNOWN_CONN_ID` or similar if the handle is invalid.
2. Builds and enqueues an HCI ACL data packet to the Link Layer.
3. Returns a status code (e.g., `HCI_SUCCESS` or an error code).

The retry design as specified — retry if the connection handle is invalid — has a structural problem worth flagging:

**Call-order/dead-path hazard:** A connection handle is either valid or invalid at the moment of the call. It is assigned by the Link Layer and does not become valid spontaneously. Retrying the same handle lookup in a tight loop **without any delay, state change, or signaling mechanism** means the retry will deterministically produce the same result on each iteration. The loop collapses into a fixed number of redundant lookups and always falls through to the error return. This is a dead path for the "retry succeeds" branch — the recovery branch is structurally unreachable unless something outside the function changes the connection table between iterations (which cannot happen in a polling loop on a single-threaded embedded RTOS task without yielding).

**Indirect calls:** The connection-handle lookup function may resolve through a function pointer table (common in BLE stack layering). Any `bl <register>` in that callee would be a CFI hazard; without the binary it cannot be ruled out.

**Call-order assumption:** `HCI_SendDataPkt` presumably requires the BLE stack to be initialized and a connection to already exist. There is no guard visible from context; the retry loop does not address this.

---

### Arithmetic Ranges

- **Loop counter:** `uint8_t retries = 0; retries < 3; retries++` — range is 0–2, no overflow risk.
- **Connection handle:** Typically `uint16_t`, range 0x0000–0x0EFF (valid BLE connection handles). No arithmetic is performed on the handle itself in the retry loop — no range issue introduced by the loop.
- No new arithmetic expressions are introduced by a plain retry wrapper, so this check is clean if no index arithmetic is added inside the loop.

---

### Freed-Resource Access

- **RAII gap / resource lifetime:** If the existing `HCI_SendDataPkt` allocates a packet buffer (e.g., via `MAP_osal_bm_alloc` or `HCI_DataBufferAlloc`) before validating the connection handle, and the retry loop re-enters after a failed handle check *without freeing that buffer first*, there is a **buffer leak on each failed retry iteration**.

  This is the most concrete risk: if the function's existing flow is:
  ```c
  buf = allocate_buffer();
  if (handle_invalid) return ERROR;   // leaks buf
  ```
  then wrapping this in a retry loop multiplies the leak by the retry count.

- **Double-free:** If the buffer is freed on the error path and the loop retries, the next iteration allocates a fresh buffer — no double-free. But if the error path does NOT free (the leak case above), that must be fixed before the loop is added.

- **Post-error state:** Some BLE stacks decrement an available-buffer counter on allocation. Leaking buffers in a 3-iteration retry loop would exhaust the buffer pool for subsequent legitimate calls.

### Adjusted plan (before writing any code)

1. **Read `hci.c` — `HCI_SendDataPkt` body** to determine the exact order of: buffer allocation, connection-handle lookup, and error return.
2. **If allocation precedes validation:** Move the connection-handle check to the top of the function, before any allocation, so failed retries have nothing to free.
3. **Reconsider the retry semantics:** On an embedded BLE RTOS (TI-RTOS / FreeRTOS), if the goal is to tolerate a race where the connection is being established, the correct pattern is to post a semaphore or queue the packet, not busy-retry. A tight 3-iteration loop with no `Task_sleep` or `osal_sleep` will complete in microseconds and cannot observe a state change. Add a `osal_sleep(1)` or `Task_sleep(1)` between iterations if a retry is truly required.
4. **If the retry loop is still desired after steps 1–3:** Write it as:
   ```c
   for (uint8_t attempt = 0; attempt < 3; attempt++) {
       // validate handle BEFORE any allocation
       if (handleIsValid(connHandle)) {
           return HCI_SendDataPkt_internal(...);
       }
       osal_sleep(1); // yield to allow state change
   }
   return HCI_ERROR_CODE_UNKNOWN_CONN_ID;
   ```
5. **Write the code** only after steps 1–4 are confirmed.
